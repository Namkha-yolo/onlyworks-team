OnlyWorks v2 — Intern Starter Guide

You're an intern on the OnlyWorks team. This document tells you what to do in your first two weeks before touching production code, what the conventions are, and how decisions get made.

If you're reading this, you've already accepted the role. Welcome.

Week 0 — Before you touch OnlyWorks

Goal: prove to yourself you can ship in our stack before you touch our codebase. ~5 working days.

Day 1–2 · Build a template app

Build a tiny Electron + Vite + React + TypeScript + tRPC todo app from scratch. Yes, a todo app. It exists to teach you the stack, not to be impressive.

Requirements:

  • New repo on your GitHub account (private is fine)
  • electron-vite create my-todo --template=react-ts
  • Tailwind v4 + shadcn/ui set up
  • One tRPC procedure: todos.add(text), todos.list(), todos.toggle(id)
  • Renderer calls the procedures through a typed tRPC client
  • One Zustand store for UI state (e.g., filter state)
  • TanStack Query wrapping the tRPC calls
  • Magic-link auth? Skip. Just hardcode a user.
  • One animation on add (use the shadcn motion patterns)
  • macOS or Windows build that runs end-to-end with npm run dev
  • One unit test using Vitest

Done when:

  • You can run npm run dev, the window opens, you can add/check/delete todos, and your terminal shows tRPC calls happening.
  • You can explain to Namkha what tRPC over IPC means and why we use it.
  • You can explain why we pinned every dependency.

Day 3 · Read the planning docs

In this order. Don't skim. Take notes.

  1. 08-plain-english-plan.md — the whole product story. ~15 min.
  2. 02-intent-and-history.md — where OnlyWorks came from, what's been tried, what's been learned. ~25 min.
  3. 04-pmf-and-v2-direction.md — who v2 is for, three revenue streams. ~30 min.
  4. 05-v2.0-scope.md — feature spec for what you're building. ~25 min.
  5. 07-build-order.md — week-by-week task list. Find your tasks. ~15 min.
  6. visuals/wireframes.html + visuals/web-wireframes.html — what the product looks like. ~20 min.
  7. visuals/00-design-system.html — the design system you'll implement against. ~30 min.

At the end, write a 1-page response to Namkha covering:

  • The one part of the plan that you're most excited about
  • The one part you're skeptical about
  • One question you don't know the answer to

This is so we know you actually read it. It also tells us what to clarify on day 1.

Day 4 · Read v1 code

Skim the OnlyWorks v1 codebase at /Users/xerneas/onlyworks_desktop. Focus on:

  • src/main.js — what does the OnlyWorksApp class do at startup?
  • src/services/capture-service.js — the capture pipeline. Read it end to end (it's 1,417 lines, sorry).
  • src/services/ocr-service.js — the OCR pipeline with platform-native paths.
  • src/services/auto-updater.js — what we're porting verbatim.
  • src/services/backend-auth.js — encrypted token storage.
  • src/preload.js — the IPC bridge that's about to be replaced with tRPC.
  • One page module of your choice, e.g., src/app/modules/page-dashboard.js — see how the v1 renderer is built. You're throwing this pattern away. Understand why.
  • package.json — build block (electron-builder config). We're keeping most of this.

You will not be tested on this. But when you hit a tricky platform bug in v2, the answer is usually in v1.

Day 5 · Set up your environment

  • macOS interns: get an Apple Developer account, install Xcode CLT, set up signing certs (Namkha will provide).
  • Windows interns: get a Windows VM (Parallels or local), install Node.js 25+, Git.
  • Cross-platform interns: do both.
  • Install pnpm (faster than npm). Install GitHub CLI.
  • Get added to:
    • GitHub: Namkha-yolo/onlyworks_desktop (write access to the v2 directory we'll create)
    • GitHub: Namkha-yolo/onlyworks-backend-server (read access)
    • Linear or whatever issue tracker we use
    • Slack/Discord team channel
  • Add SSH key. Set up commit signing with GPG (optional but encouraged).

Done with Week 0 when:

  • Your template app builds and runs
  • You've read all the planning docs
  • You can sign in to all our shared tools
  • You've written your 1-page response

Week 1 — Your first OnlyWorks task

You have one task in week 1. It's listed in 07-build-order.md. You will finish it before Friday.

For everyone in week 1, the deliverable is small enough to be done in 3–4 working days. The fifth day is for code review iteration. Don't try to over-deliver. Don't add features. Just finish the listed task.

What a "good" week-1 PR looks like

  • One feature branch off main
  • One commit (squash before requesting review)
  • A clear PR description: what changed, why, what testing you did
  • All CI checks pass (lint, types, tests)
  • Build runs locally on macOS or Windows (whichever you own)
  • No new dependencies added without asking
  • No console.log debugging left in
  • No TODO comments left in the code (open an issue instead)

How code review works

  • Tag Namkha. She reviews every PR for weeks 1–4.
  • Expect 1–3 review rounds. This is normal. Don't take notes personally.
  • Address every comment before re-requesting review. If you disagree, explain why in a reply — don't just close the comment.
  • "LGTM" or "✅ approve" from Namkha = ship it (squash merge to main).

After week 4, intern PR review pairs:

  • A reviews B
  • B reviews C
  • C reviews A
  • Namkha reviews anything tagged needs-decision or architecture

Conventions you will follow

Code style

  • TypeScript: strict mode on. No any unless you write a comment explaining why. Prefer unknown + narrowing.
  • Imports: absolute paths from src/ root (configure tsconfig paths).
  • Files: one component per file. Component name = filename (PascalCase for components, kebab-case for utilities).
  • Naming: prefer descriptive over short. isCapturing not cap. onSessionEnd not onEnd.
  • No defensive code without a reason. Don't add try/catch "just in case." Let errors bubble; we have global error handlers.
  • No comments that restate the code. Only comment the WHY, not the WHAT.
  • No emojis in code. Yes really. The CSS has border-radius: 0 and we have a brand to maintain.

Component conventions

  • Use the design system tokens from visuals/00-design-system.html. No new design tokens without explicit ask.
  • All components composed of shadcn/ui primitives + Tailwind utility classes. No custom CSS unless absolutely necessary.
  • Sharp corners (rounded-none) — except marketing CTAs (rounded-full) and avatars/dots (rounded-full).
  • Buttons: only use the Button component with variant="primary|secondary|ghost|danger". No custom button styles.
  • Modals: only use the Modal component (which wraps shadcn's Dialog). No raw <div> modals.

Commits

We use conventional commits (loosely):

  • feat(capture): add SHA-256 dedup
  • fix(auth): refresh token after sleep/wake
  • refactor(portfolio): extract SkillCard component
  • chore(deps): pin electron to 33.2.1
  • docs(planning): update v2.0 scope

Squash commits before merging. The PR title becomes the commit message on main.

Branching

  • Branch from main. Name: intern-X/short-task-name or feat/short-task-name
  • Push frequently. Small commits are fine in your local branch — squash on merge.
  • Don't merge to main directly. PR + review only.
  • If your branch gets stale (>1 week), rebase on main. Don't merge main into your branch.

Dependency policy

  • We use pnpm. pnpm-lock.yaml is committed.
  • Don't add a dependency without asking. Many problems have a stdlib answer.
  • All deps pinned to exact versions ("react": "18.3.1", not "^18.3.1"). No surprise upgrades.
  • If you must add a dep, the PR description must answer: why this dep, why this version, what alternatives were considered, what's the bundle-size cost.

Decisions and process

Daily

  • 15-min standup in Slack at the start of your day. Format:
    • Yesterday: what shipped, what's blocked
    • Today: one task
    • Blockers: anything in your way
  • If async standup feels insufficient, propose a 10-min video call.

Weekly

  • Friday demo (30 min) — each person shows what shipped this week. Including bugs found.
  • Friday retro (30 min) — what worked, what didn't, what to adjust next week.

Pairing

  • For weeks 1–4: Namkha pair-programs with each intern twice/week for 1 hour. These are not optional. They surface bugs early.
  • After week 4: optional, intern-initiated.

No-meeting Wednesdays

  • No standups, no demos, no calls on Wednesdays.
  • Deep work only. If you need to interrupt someone, ask first.

When you're blocked

In this order:

  1. Search the codebase. git grep is your friend. v1 likely has the answer.
  2. Search the planning docs. Most gotchas are in 01-component-inventory.md and 03-architecture-review.md.
  3. Ask the team in Slack. Don't sit silent for >30 minutes if blocked.
  4. Ping Namkha for decisions or architecture questions.
  5. Open a GitHub Issue with label needs-decision if it'll block multiple people. Don't silently pivot the design.

When you disagree with a decision

Say so. We don't punish disagreement; we punish silence followed by surprise.

  • Slack DM the relevant person, not the whole channel.
  • Bring an alternative + tradeoff analysis, not just "I don't like X."
  • Once a decision is made, commit to it. Don't relitigate it in PR review.

What you should ship by end of Week 4

By end of week 4 (the foundation phase), the team collectively should have:

  • A green-building v2 desktop app that opens a window and shows "Hello"
  • Auth + auto-updater ported and working
  • CaptureEngine skeleton firing frame events
  • OcrConsumer working on at least one platform
  • TrainingDataConsumer scaffolded with consent flow
  • Sensitive content filter ported from v1
  • Backend Stripe Checkout + webhook + portal working

Individually you should have:

  • Shipped 8–12 PRs
  • Resolved every CI failure on your branches
  • Reviewed at least 4 PRs (from week 4 onwards)
  • Hit every Friday demo with something to show

If you're behind, raise it on day 1 of the week — not week 3.

Things we don't want you to do

These will get pushback in review. Save yourself the cycles.

  • Don't add a new design system class. Use the existing tokens.
  • Don't introduce a new state management library. Zustand + TanStack Query is the whole stack.
  • Don't write a custom modal. Use the Modal component.
  • Don't add // TODO, // FIXME, // HACK comments. Open an issue with the same description.
  • Don't push to main directly. Even tiny commits.
  • Don't merge your own PR. Wait for approval.
  • Don't skip CI by --no-verify. If a hook fails, fix the cause, not the symptom.
  • Don't add console.log to shipped code. Use the logger.
  • Don't refactor while porting from v1. Get the v1 code working in TS first. Refactor in a separate PR if needed.
  • Don't change the design without checking. If a screen needs to differ from the wireframe, propose in Slack first.
  • Don't optimize prematurely. First make it work, then make it right, then maybe make it fast.

Things we DO want you to do

  • Ask "why" questions. Many decisions have reasoning that isn't documented. Make us document it.
  • Find bugs while building. Open issues as you notice them. Don't fix random bugs in unrelated PRs.
  • Suggest improvements. After week 4, propose changes via GitHub issues. Even big ones.
  • Help each other. PR review is collaboration, not gatekeeping.
  • Take breaks. Tired code is buggy code.
  • Tell us if something feels off. Process, communication, scope, code — say something on day 1, not week 8.

The shape of a week

Here's roughly what your week looks like after Week 0:

Day Block 1 (AM) Block 2 (PM)
Mon Standup → one task Implementation
Tue Implementation Implementation
Wed Deep work · no meetings PR opens by EOD
Thu Review + iterate Pairing w/ Namkha (1 hr)
Fri Final fixes Demo + retro

Adapt to your rhythm. Just hit the demo on Friday with something real.

Final word

The most valuable thing you can do in your first month is make small mistakes loud and early. We will not punish you for breaking something. We will punish silence — finding out in week 6 that you've been blocked since week 3 is the failure mode we're trying to avoid.

If you don't know how to do something, that's fine. If you pretend you do and ship something broken, that's not.

Welcome aboard. Let's go.

— Namkha
