OnlyWorks v2.0 — Scope and Build Plan

Detailed feature spec for the v2.0 rebuild. 12-week scope. 4-person team (founder + 3 interns).

Goals

Workers: install OnlyWorks, run capture in the background, build a verified portfolio of their real work, get matched with hiring managers who pay to reach them.

Hiring managers: subscribe, search candidates whose skills are verified by actual captured work, express interest, get replies.

v2.0 ships:

  • New desktop app with unified capture pipeline (reports + training data)
  • HM paid tier (Stripe Checkout + webhook + billing portal)
  • Polished web public profile + hiring search UI
  • Talent community nurture (real-user signups don't go cold)

Success criteria

By end of week 12:

  • 100 weekly active workers (target: 60% from outside founder network)
  • 5 paid hiring managers (500−1,500 MRR)
  • 80% of v2.0 free users opt-in to training data sharing (signal that consent UX is clear)
  • Zero capture-pipeline bug reports for the last 4 weeks
  • 1M+ training frames collected with proper labels + consent

Out of scope (explicit non-goals)

These don't ship in v2.0 even if asked:

  • Team / internal reporting UI — deferred to v2.2 with new "save people time on status reports" positioning. Backend infra (team tables, role hierarchy, member-report generation) stays intact in v2.0; only the worker-facing team UI is cut. Do not delete team handlers, services, or migrations.
  • Goals UI (no PMF signal)
  • Activity / Sessions pages as separate nav (rolled into Portfolio)
  • Resume upload + AI parsing (defer to v2.2 or skip — verified-by-capture is the differentiator)
  • Promo packet generator (v2.1)
  • Weekly summary email (v2.1)
  • Billing / invoice automation, "Path A" (v3 or kill)
  • Assignment-based skill verification (v2.2 or skip)
  • LinkedIn / Google OAuth (v2.1 if magic link conversion suffers)
  • Team / Goals / Tutorial / Day Reports / Manager-Worker toggle
  • Local downloads of reports (link-based sharing only)
  • Dark mode / theming polish beyond default (v2.1)

Worker daily/weekly use

Per the founder gut check ("compounding portfolio that builds itself"), the daily UX is minimal:

  • Tray icon shows capture status (running / paused / not granted)
  • Optional: open the app weekly to check Portfolio growth
  • That's it. No required check-ins. No engagement gymnastics.

Capture Pipeline (v2 architecture)

Single trigger system feeding multiple consumers. Detailed in 03-architecture-review.md. Recap:

CaptureEngine
  ├─ Triggers: 10s interval (configurable), click, Enter key, clipboard copy, manual
  ├─ Frame: { jpeg, sha256, timestamp, sessionId, triggerType, activeApp, windowTitle }
  ├─ Guards: dedup, debounce, idle-stop, sensitive-content filter
  └─ Event: 'frame'

Consumers (all subscribe to 'frame' events)
  ├─ OcrConsumer
  │   - macOS Vision / Windows OCR (no Tesseract)
  │   - Extract text + bounding boxes
  │   - POST /api/pipeline/activity/stream (rolling buffer, flush every 30s)
  │
  ├─ TrainingDataConsumer (consent-gated)
  │   - Subsample (1-in-N or significant-change-only)
  │   - Privacy-filter applied
  │   - POST /api/pipeline/training/frames
  │   - Records training_consent_version + labeled categories
  │
  └─ MetricsConsumer
      - Active app tracking
      - Per-app time buckets
      - Idle period tracking
      - Reported via session lifecycle

Frames the user sees in "Recent captures" (Privacy screen) come from this same pipeline — keeps the "what we captured" feature honest.

Why unified matters:

  • One trigger, one dedup, one consent boundary
  • Training data quality benefits from same dedup / filter pipeline as production reports
  • Replaces v1's capture-service.js (52KB) + activity-buffer-service.js + training-data-service.js + duplicated trigger logic with ~500-800 lines total

API Contracts (key endpoints)

These are the desktop ↔︎ backend contracts that must be stable in v2.0. Most existing endpoints survive; new ones are marked.

Auth (existing, port verbatim)

  • POST /api/auth/email — request magic link
  • GET /api/auth/oauth/poll?state=… — long-poll callback
  • POST /api/auth/refresh — refresh access token

Session lifecycle (port)

  • POST /api/sessions — create
  • POST /api/sessions/:id/end — end (triggers report)
  • GET /api/sessions/:id — fetch
  • Backend stale-session finalizer continues to run every 10 min

Capture pipeline (NEW shape, simplified from v1)

  • POST /api/pipeline/activity/stream — batched OCR text + metadata (every 30s)
  • POST /api/pipeline/training/frames — full frame + label + consent token (NEW: consent_token required)

Reports + portfolio (port)

  • GET /api/profiles/:owId — public profile
  • GET /api/me/portfolio — verified skills + evidence
  • GET /api/reports/today — current day report
  • PUT /api/me/visibility — public profile visibility settings

Hiring (mostly built, gate by paid status — NEW Stripe integration)

  • POST /api/hiring/checkout — NEW: create Stripe Checkout session
  • POST /api/hiring/webhook — NEW: Stripe webhook listener (customer.subscription.created/updated/deleted)
  • POST /api/hiring/portal — NEW: create Stripe billing portal link
  • GET /api/hiring/search — search candidates (existing)
  • POST /api/hiring/jobs — create job post (existing, was paywalled but unreachable)
  • GET /api/hiring/jobs/:id/matches — view matches (existing)
  • POST /api/hiring/jobs/:id/express-interest/:owId — outreach (existing)

Talent community (port + NEW worker)

  • POST /api/talent-community/join — public magic-link signup (existing)
  • NEW: Background worker, setInterval in src/app.js, day-3/14/30 nurture emails using existing Resend

Sprint Plan (12 weeks, 4 people)

Founder + 3 interns. ~48 person-weeks of capacity. Founder splits time between code, hiring side, and sales/feedback.

Weeks 1-2: Foundation

  • Intern A: Scaffold onlyworks_desktop_v2/ — Electron 33+, electron-vite, React + TS + Tailwind + shadcn/ui + TanStack Router + TanStack Query, tRPC over IPC, Vitest

  • Intern C: Skeleton CaptureEngine — port trigger logic, dedup, idle/inactivity from v1's capture-service.js. No consumers yet.
  • Founder: Backend Stripe integration: Checkout session creation, webhook handler, billing portal link, gate /api/hiring/jobs properly

  • Intern B: TrainingDataConsumer + consent flow — consent token included in /api/pipeline/training/frames payload; backend validates against current consent version
  • Intern C: Sensitive content filter — port sensitive-filter.js pattern; apply once at CaptureEngine level (both consumers benefit)
  • Founder: Backend per-user AI budget across text + vision paths; delete AIAnalysisService mocks and /api/backtest

Weeks 5-6: Worker desktop UI

  • Intern A: Status screen + First-run onboarding flow (6 sub-screens)
  • Intern B: Portfolio screen + Privacy screen

Weeks 7-8: Web side (parallel)

Weeks 9-10: Dogfood + bug fix

Weeks 11-12: Soft launch

  • Roll out to existing v1 user list (~14 confirmed users) via opt-in beta channel of electron-updater
  • Recruit 20 new workers (founder network warm intros + reach to the 5-10 strangers from v1)
  • Recruit 30+ student users via 1-2 campus or bootcamp partnerships — students are the highest-intent supply for the marketplace
    • Each intern reaches out to their own school's CS club / design club / career center
    • Sponsor or partner with one hackathon in the launch window — workshops + free Pro for attendees
    • Optional: post in r/cscareerquestions about a small pilot
  • Recruit 5 HMs (paid pilot — offer $99/mo Starter free for 60 days, then convert; lean on HMs who hire early-career / interns where verified-by-capture is the strongest signal)
  • Daily feedback channel (Slack / Discord / email)
  • Weekly retro; ship hotfixes

Decisions Needed Before Week 1

These should be locked before any code lands. Each is a question for the founder.

  1. Repo strategy: new onlyworks_desktop_v2/ directory in a sibling repo, or a new branch on the existing repo? (Recommend: new directory in this repo to share package.json lessons; merge to main once v2.0 ships.)

  2. Naming: ship v2.0 as "OnlyWorks 4.0" (continues v1's version 3.x line) or relaunch under a new product name? (Recommend: continue version line; new product name is marketing complexity for a 4-person team.)

  3. Auto-update channel: force v1 users onto v2.0 immediately, or opt-in beta channel until parity? (Recommend: opt-in beta until 4 weeks bug-free, then flip default.)

  4. Stripe pricing finalize: $99 / $299 / $599 from 04-pmf-and-v2-direction.md — confirm or adjust?

  5. Training data terms: what do users actually consent to? Need a 1-page T&C addendum users see during onboarding. Founder + lawyer review.

  6. Intern starting point: each intern's first task is "build a tiny Electron + Vite + React + tRPC app from a template" — confirm 1 week of ramp time before they touch OnlyWorks code.

  7. Student verification: unlock free Pro for students via .edu email verification, or skip the student-specific tier and keep Free + Pro only? (Recommend: ship with .edu verification — high-intent supply for the marketplace, low cost to give away Pro to a price-insensitive segment. Use the same magic-link infrastructure; just check email domain.)

Risks and Mitigations

  1. Training data consent UX is hard. Users either don't read it (silent consent = legal risk) or read and decline (training set goes nowhere). Mitigation: make the default ON but the language plain and the toggle prominent. Track decline rate; if >30%, redesign.

  2. HM-side has chicken-and-egg. No HMs means no Express Interest pings; no pings means workers don't see the hiring payoff. Mitigation: Founder personally recruits the first 5 HMs from her network as paid pilots. Until then, simulate the experience: send manual "Hey, an HM searched for someone like you" emails to first workers to build the feedback loop signal.

  3. Stripe webhook reliability. A missed customer.subscription.created event = HM pays but stays gated. Mitigation: add idempotent retry, log all events, set up a daily reconciliation script.

  4. Capture pipeline regressions vs v1. Three months of bug fixes (macOS TCC relaunch, Windows uIOhook, active-win fallbacks) must port cleanly. Mitigation: port intact (don't refactor) the platform-specific paths; keep v1 capture as a fallback during v2.0 beta.

  5. Intern ramp slower than expected. 1 week of stack ramp might be 2-3 weeks of actually understanding the codebase. Mitigation: founder reviews every PR week 1-4; pair-program twice a week; pin every dependency version.

  6. Backend single-instance scaling. In-memory rate limits, magic-link store, vision-cost stats all fragment under multi-instance. Mitigation: v2.0 stays on single Render instance; v2.1 migrates to Redis-backed.

Open Questions for Validation Sprint (run in parallel with weeks 1-4)

These should be answered before week 5 (when worker UI lands):

  1. Do the 5-10 strangers actually share their public profile with anyone? If so, who?
  2. Has any of them gotten a job offer through OnlyWorks? (Even tangentially.)
  3. What HM do they imagine "should" find them? (Quotes inform Express Interest copy.)
  4. Would they pay $9/mo for Pro? At what price would they hesitate?
  5. (HM side) Talk to 10 hiring managers, recruiters, or staffing-firm partners: would they pay $99 / $299 / $599 for verified-by-capture candidates?

If the answers to 1-3 are weak (no one shares, no one gets hired, no one imagines an HM), the v2.0 plan needs adjustment in week 4 before the worker UI is built. The Portfolio screen is built around the assumption that the portfolio is the artifact users want — validate that assumption is real.
