Weeks 7-8 — Web Marketplace & Stripe

This is the phase that makes OnlyWorks a business. The worker desktop app is the supply side; the demand side — and the only revenue in v2.0 — lives on the web: hiring managers searching verified candidates, posting jobs, and paying for it through Stripe. By the end of Week 8, Stripe runs end to end, the paywall actually gates paid features, and the hiring-manager dashboard and public profile are live.

Most of this work is in the website repo (onlyworks_verified, Next.js 14) and the backend (Express), not the desktop app. The web surface uses the light "warm paper" palette — a different surface from the desktop dark theme. Implement web against the web wireframes (visuals/web-wireframes.html).

0 · How to use this document

Same conventions as docs 15, 19, and 20. Note the repo split: desktop is onlyworks_desktop_v2; web is onlyworks_verified; the backend is onlyworks-backend-server. Be explicit in PRs about which repo you are in.

1 · Where we are right now

  • Stripe does not exist yet. The backend has stripe_customer_id and subscription_status columns, but no checkout, webhook, or portal — paid HMs are gated only by a manual database update today. This phase builds it.
  • The website (onlyworks_verified) is already in production and already uses shadcn + Tailwind; it is extended, not rewritten.
  • The desktop app (from W3-6) has working capture, OCR, auth, and all five screens — the supply side is ready to feed the marketplace.

2 · The phase at a glance

  • Namkha (backend). Stripe checkout / webhook / portal, the paywall, and the per-user AI budget cap. This is the revenue-critical path and Namkha owns it.
  • Group A — Jacqueline (web). The public profile redesign, the Job Post creation UI, and a mobile-friendly profile.
  • Groups B & C. The web auth token swap and dashboard support.

3 · Cross-group dependency & hand-off map

  • Namkha → A: the checkout session endpoint and the subscription_status the dashboard reads to show plan state and gate features.
  • C/B → A: the web auth token swap (supabase session → backend JWT) that authenticated web pages depend on.
  • Desktop → web: the verified-skill data and public profile (/ow/{shareToken}) generated from the desktop capture pipeline is what HMs search and what the public profile renders.

4 · Namkha — Stripe & AI budget (backend)

Mission. Make money move. Stripe end to end, idempotent, and gating the right things — plus a hard cap on AI spend before launch.

  • POST /api/hiring/checkout — create a Stripe Checkout session for the $99 / $299 / $599 hiring-manager tiers. [CONFIRM with Namkha: tier prices and per-tier limits are final.]
  • POST /api/hiring/webhook — handle customer.subscription.created/updated/deleted, idempotent (a replayed event must not double-apply).
  • POST /api/hiring/portal — the Stripe billing portal session.
  • Gate POST /api/hiring/jobs and Express-Interest by real subscription_status, not a manual flag.
  • Per-user monthly AI budget cap (new ai_usage table). Today vision is capped at $0.50/user/day but text reports are uncapped — close that before launch. Add the split models (gemini-2.0-flash for the pipeline, gemini-2.0-pro for final reports) and skip AI for sessions under five minutes.

Definition of done. A test HM can check out, get a real subscription_status, post a job because they paid, open the billing portal, and a replayed webhook does not double-charge or double-grant. The AI budget cap rejects spend past the monthly limit.

5 · Group A — Frontend / Web (Jacqueline)

Mission. The web surfaces HMs and workers actually look at: a public profile worth sharing and a job-post flow worth paying for.

Required reading. visuals/web-wireframes.html (HM Search, HM Dashboard, Job Post, Public Profile), docs/v2-stack/nextjs.html, the existing onlyworks_verified components.

  • Redesign the public profile (app/p/[owId] / /ow/{shareToken}): the verified-skills view a candidate shares and an HM lands on. SEO-indexed, sharable, mobile-friendly.
  • Build the Job Post creation UI (the backend exists; the frontend is missing). Gate the publish action behind paid subscription_status.
  • Make the public profile responsive — it is the most-shared surface.

Definition of done. A public profile renders real verified-skill data and looks good on mobile; an HM can create and publish a job post only when on a paid plan; the dashboard reflects the live plan state.

Pitfalls. Mixing the desktop dark palette into the light web surface; forgetting the paywall gate on publish; a public profile that leaks data it should not (respect the worker's visibility settings).

6 · Groups B & C — auth bridge & dashboard support

Mission. Connect web auth to the backend and support the HM dashboard.

  • Web auth token swap: supabase access_token → POST /api/auth/website/website-token → backend JWT, so authenticated web pages (dashboard, job post) can call the backend. (C owns auth; B helps on the IPC/transport mindset.)
  • HM dashboard support: the data endpoints and the notification center the dashboard renders.
  • Keep the desktop app stable — this phase should not regress capture/auth on desktop while attention is on web.

Definition of done. An authenticated HM web session can call the backend with a valid JWT; the dashboard shows the HM's plan, posts, and matches.

7 · End-of-phase demo (Thursday, Week 8, 19:00)

  • Namkha: a full paid-HM loop — checkout, webhook updates subscription_status, post a job, open the billing portal; replay a webhook and show it is idempotent.
  • Group A: a shared public profile on desktop browser and mobile, and the job-post flow gated by plan.
  • Groups B/C: an authenticated web session calling the backend.

8 · Risks watch list

  • Stripe is the revenue gate and it is new. Webhook idempotency and the paywall are where bugs become billing incidents — test replays and edge states.
  • AI cost exposure. Text reports are uncapped today; the $0.50/user/day vision cap is already ~$15/user/month against a $9/mo plan. The monthly cap must land this phase, not later.
  • Founder bandwidth. Stripe, the AI cap, and landing the first paid HMs all sit on Namkha. Interns do not pair on sales, but flag if the backend work is at risk so it does not slip into W9-10.
  • Two surfaces, two palettes. Keep the web light theme and desktop dark theme strictly separate.

9 · When something is wrong

Be explicit about which repo a PR touches. Surface backend/Stripe blockers early — they gate the launch. Write Stripe decisions (which events, how idempotency is keyed) in the PR. Never test against live Stripe keys; use test mode.
