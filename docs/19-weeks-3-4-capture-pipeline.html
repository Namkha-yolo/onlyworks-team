Weeks 3-4 — Capture Pipeline

This is the phase where OnlyWorks becomes a product instead of a scaffold. By the end of Week 4, the unified CaptureEngine emits frame events, an OcrConsumer turns those frames into activity text and ships it to the backend, a consent-gated TrainingDataConsumer stores frames for the model bet, auth is fully working end to end, and a first-run onboarding skeleton with the macOS permission flow exists.

One structural thing to internalize before you read your section: the CaptureEngine core is Namkha's. You do not build the engine. You build the things that hang off it — consumers and screens — against a typed FrameEvent contract and a capture-events subscription that Namkha publishes early in Week 3. Until that handoff lands, your consumer work is blocked, so read §3 carefully and plan around it.

0 · How to use this document

This document contains instructions, not code. It uses the same conventions as the foundation plan (doc 15): each group has a Mission, Required reading, a week-by-week task list, Hand-offs in/out, a Definition of done checklist, the Pitfalls that cost previous teams days, and a Self-check.

Turn an instruction like "subscribe the OcrConsumer to the capture-events stream" into working code by reading docs/v2-stack/, the v1 source at /Users/xerneas/onlyworks_desktop, and the upstream docs. If an instruction is ambiguous, ask in standup or write a one-line proposal in the PR description. Do not invent something complicated to feel safe.

All the shared operating norms from the foundation plan still apply: one PR per task, standup Tue/Thu 14:30, demo Thursday 19:00, code-review SLA 4 hours, pin every dependency to an exact version, branch names like w3/groupB/ocr-consumer, never merge red CI.

1 · Where we are right now

End of Week 2, on main of OnlyWorksCodeHub/onlyworks_desktop_v2:

  • The hand-rolled tRPC-over-IPC bridge is merged. There is one ipcMain.handle('trpc') channel; the renderer talks to it through a custom ipcLink built on createTRPCClient<AppRouter>. This is not electron-trpc — do not reintroduce it.
  • src/main/trpc/capture.ts is an in-memory stub: capture.start flips a boolean and capture.status reports it. Its own comment says "the real capture engine … lands in W3" and "streaming capture events over a subscription" arrives the same week. That is this phase.
  • Shared contracts are merged: src/shared/contracts/, procedure-names.ts (which already whitelists the auth.* paths as forward declarations), and the ApiResult envelope (used for backend HTTP only — tRPC procedures return bare shapes and throw).
  • The renderer has three code-based routes (/ Status, /portfolio, /settings); Status is wired with useMutation/useQuery and an error-only Dialog.
  • Auth services are ported but not on the bridge: backend-auth.ts, backend-api.ts, auto-updater.ts, config-manager.ts exist in src/main, but there is no auth.router.ts, no electron-store, and no email-code views yet. Group C auth is the thing most behind and is the first job this phase.
  • There is no CaptureEngine, OCR, consumer, sensitive-filter, Privacy screen, or onboarding screen yet. All of that is W3-4.

2 · The phase at a glance

Four parallel workstreams converge by the Thursday-of-Week-4 demo:

  • Namkha — CaptureEngine. Builds the engine, publishes the FrameEvent contract and the capture-events subscription early in W3. Everything below depends on that handoff.
  • Group A — Jacqueline (Frontend). Live capture state on Status; the First-run onboarding skeleton + macOS permission request screen; a recent-captures reel scaffold for the future Privacy screen.
  • Group B — Lihan (Desktop/IPC). The OcrConsumer — port v1 OCR (macOS Vision binary + Windows OCR with a circuit breaker), subscribe to the engine, batch text to the activity stream.
  • Group C — Yingzhu (Auth & Secure Storage). Finish email-code auth on the bridge with encrypted token storage; then the consent-gated TrainingDataConsumer.

3 · Cross-group dependency & hand-off map

The single most important dependency this phase:

  • Namkha → B and C (early W3): the capture contract. Namkha publishes the typed FrameEvent shape in src/shared/contracts/capture.contract.ts plus the capture.events subscription on the bridge. B's OcrConsumer and C's TrainingDataConsumer cannot finish until this lands. Build behind the agreed contract shape so you are not idle — but expect the real wiring to land mid-W3.
  • C → A (mid W3): working auth. Onboarding's last step signs the user in; A needs C's email-code flow to complete the first-run funnel.
  • Namkha + B: macOS Accessibility / TCC coordination. The engine's input hooks and B's native OCR both touch macOS permissions; the relaunch trick (see §7 and the v1 reference) must be owned in one place — coordinate so it is not implemented twice.
  • B → Namkha: OcrConsumer registration. B's consumer registers against the engine's consumer interface; agree the registration API with Namkha before writing it.

4 · Group A — Frontend (Jacqueline)

Mission. Make capture visible and consentful in the UI: live status driven by real engine events, and the front door (onboarding + permissions) that gets a brand-new user to their first capture in under four minutes.

Required reading.

  • visuals/wireframes.html — the First-run and Status screens.
  • docs/v2-stack/tanstack-query.html — you will consume the capture subscription through Query / a subscription hook.
  • The merged src/renderer/src/routes/status.tsx — your existing pattern (useMutation + useQuery + error-only Dialog) is the template to extend.
  • The FrameEvent contract Namkha publishes — your live state renders from it.

Week 3.

  • Once Namkha publishes capture.events, subscribe to it from the Status route and render live capture state (running / idle, last-frame time, frames this session). Keep server-owned state in TanStack Query; UI-only flags in the existing Zustand store.
  • Stand up the First-run onboarding route as a 6-step flow shell (welcome → permissions → sign-in → consent → first capture → done). Wire navigation between steps; real content fills in over W3-4 and polishes in W5-6.
  • Build the permissions request screen: it asks for macOS Screen-Recording and Accessibility, explains why, and reflects current grant state. The actual permission APIs and the relaunch behaviour belong to the native side — coordinate with Namkha/B; your job is the screen and the polling that detects when a permission flips to granted.

Week 4.

  • Scaffold the recent-captures reel (the strip of recent frames) as a component you will mount on the Privacy screen in W5-6. For now it can read from a thumbnail/metrics consumer or a placeholder; the point is the data path and the layout, not final styling.
  • Wire the consent step UI: four toggles, each defaulted off (capture is required to proceed; AI reports optional; public profile off; training data opt-in only). The toggle state is owned by Group C's consent logic — you render and bind it.
  • Make sure a fresh user can go welcome → granted permissions → signed in → consented → first capture without a dead end. Time it.

Hand-offs. In: the FrameEvent contract + capture.events from Namkha; working email-code sign-in from C. Out: the onboarding shell and consent UI that C binds its consent logic into.

Definition of done (Friday W4).

  • ☐ Status shows live capture state driven by real engine events, not the stub.
  • ☐ First-run flow navigates all 6 steps end to end with no dead ends.
  • ☐ Permissions screen reflects real macOS grant state and advances when granted.
  • ☐ Consent step renders four off-by-default toggles bound to C's logic.
  • ☐ Recent-captures reel component exists and renders sample data.

Pitfalls.

  • Do not store fetched capture state in Zustand — it belongs in Query. Zustand is for UI flags only.
  • Permission state can change while the app is running; poll for it and handle the macOS relaunch (the native side triggers it — your screen must survive the relaunch and resume at the right step).
  • Do not default the training-data toggle on. Opt-in is a legal requirement, not a funnel optimization (see §9).

Self-check. Can you explain how a frame captured in the main process becomes a number on the Status screen, every hop? If not, you do not yet understand the subscription path.

5 · Group B — Desktop/IPC (Lihan)

Mission. Turn raw frames into activity text. Own the OcrConsumer: on-device OCR on both platforms, fed by the engine, shipped to the backend — fast, free, and offline.

Required reading.

  • v1 src/services/ocr-service.js — the macOS Vision + Windows OCR paths and the circuit breaker you are porting.
  • v1 src/utils/windows-ocr.ps1 and src/services/windows-preflight.js.
  • docs/v2-stack/macos-vision-ocr.html and docs/v2-stack/windows-ocr.html.
  • The engine's consumer-registration interface (from Namkha).

Week 3.

  • Port the macOS path first: the compiled Swift Vision binary (src/native/ocr_helper) → Swift-interpreter fallback. Drop tesseract entirely — if Vision/Windows OCR is unavailable, skip OCR, do not fall back to tesseract.
  • Register an OcrConsumer against the engine: it receives FrameEvents, runs OCR on the image, and produces { text, app, windowTitle, timestamp }. Build behind the agreed contract until Namkha's subscription lands, then wire it for real.
  • Batch OCR output and POST /api/pipeline/activity/stream every 30 seconds (port the v1 activity-buffer batching behaviour).

Week 4.

  • Port the Windows path: Windows.Media.Ocr via the PowerShell wrapper, with the mandatory 3-consecutive-failure circuit breaker (after 3 WinRT failures, stop trying OCR for the rest of the session). Port windows-preflight. asarUnpack the .ps1 and native modules (uiohook-napi, sharp, active-win, *.ps1).
  • Coordinate the macOS Accessibility / TCC handling with Namkha's engine: uIOhook requires a systemPreferences.isTrustedAccessibilityClient(false) precheck before .start() (a try/catch cannot catch the C-level abort), and the TCC relaunch trick must live in exactly one place. Decide with Namkha where.

Hand-offs. In: the FrameEvent contract + subscription + consumer interface from Namkha. Out: a working activity stream that the backend (and, later, reports) consume; the OCR text path the rest of the pipeline assumes exists.

Definition of done (Friday W4).

  • ☐ OcrConsumer registered on the engine; receives real frames.
  • ☐ macOS Vision OCR returns text from a live capture; no tesseract anywhere.
  • ☐ Windows OCR works with the 3-failure circuit breaker; preflight ported.
  • ☐ Activity batches POST to /api/pipeline/activity/stream every 30s.
  • ☐ Native modules + .ps1 are asarUnpacked; the app still builds.

Pitfalls.

  • The macOS Vision binary must be universal (x86_64 + arm64) and bundled to resourcesPath; a missing/non-universal binary fails silently on one architecture.
  • The Windows circuit breaker is not optional — without it, a broken WinRT install spins forever and burns CPU.
  • OCR quality directly drives report quality. Match v1's recognition settings rather than guessing.

Self-check. What happens, exactly, when Windows OCR fails three times in a row? If you can't answer, the circuit breaker is not done.

6 · Group C — Auth & Secure Storage (Yingzhu)

Mission. Two missions, in order: (1) finish email-code auth on the bridge with encrypted token storage — it is behind and blocks A's onboarding — then (2) the consent-gated TrainingDataConsumer.

Required reading.

  • v1 src/services/backend-auth.js and src/core/token-refresh.js (the ported TS versions are already in src/main).
  • Security & compliance (doc 14) — the consent model and the redaction pipeline.
  • The backend auth endpoints (see facts/§ below) and their token shapes.

Week 3 — finish auth first.

  • Add src/main/trpc/auth.router.ts with auth.requestEmailCode, auth.verifyEmailCode, auth.status, auth.signOut, and mount it under appRouter (coordinate the mount with Lihan). The renderer never sees raw tokens.
  • Wire the backend calls: POST {OW_BACKEND_URL}/api/auth/email and POST .../api/auth/email/verify. The staging base URL comes from Namkha via OW_BACKEND_URL — do not hardcode it.
  • Encrypt tokens at rest with electron-store + safeStorage (key in the OS keychain). Hard stop: never write a raw token or key to disk. Verify the on-disk config shows gibberish base64.
  • Read token expiry from the JWT exp claim (decode in main), not from expires_in. Refresh proactively on powerMonitor 'resume' (the wall clock jumps on sleep).
  • Build the 6-digit email-code views with react-hook-form + zod. auth.status in the renderer is a TanStack Query poll (refetchInterval ~10s) — subscriptions come later.

Week 4 — consent + training consumer.

  • Register a TrainingDataConsumer on the engine, gated on the training consent flag. It uploads consented frames to POST /api/pipeline/training/frames with a required consent_token (a hash of consent-version + userId + timestamp).
  • Implement the four-toggle consent model server-logged with timestamps; binding into A's consent UI. Capture is required; reports optional; public profile off; training opt-in only. Revocation must be supported from day one.

Hand-offs. In: the bridge (B), the engine + contract (Namkha), the consent UI shell (A). Out: working sign-in for A's onboarding; the consent gate the TrainingDataConsumer and the whole training-data bet depend on.

Definition of done (Friday W4).

  • ☐ A real email-code sign-in works end to end against staging.
  • ☐ Tokens are encrypted at rest; on-disk config is unreadable; sign-out wipes them.
  • ☐ Wrong/expired code saves nothing; a refresh network failure clears tokens.
  • ☐ TrainingDataConsumer uploads only consented frames, with a valid consent token.
  • ☐ Consent toggles persist and are server-logged; revocation path exists.

Pitfalls.

  • The token shapes differ: /email/verify nests tokens under data.tokens, but /refresh nests them flat under data. A 429 returns error as a string, not an object — handle it in your post() helper.
  • The verify user object only guarantees id, email, full_name; ow_id and profile_complete are optional. Do not assume they are present.
  • Do not default training consent on to chase the 80% opt-in metric — it is opt-in by law and by the security plan.

Self-check. If a user signs in, sleeps the laptop for two hours, and wakes it, what happens to their tokens and why? Walk the exp-claim + resume-refresh path.

7 · Namkha's track — the CaptureEngine

The CaptureEngine is owned by Namkha and its spec is kept private (it is the product core and the training-data bet). You do not need its internals. What you need is the published contract:

  • A typed FrameEvent shape in src/shared/contracts/capture.contract.ts.
  • A capture.events subscription on the hand-rolled tRPC bridge that delivers those events to the renderer and to in-main consumers.
  • A consumer-registration interface that B's OcrConsumer and C's TrainingDataConsumer register against.

Namkha publishes this early in Week 3. Build behind the agreed shape until then; do not reach into engine internals or re-implement triggers, dedup, or the permission/relaunch logic yourself.

8 · End-of-phase demo (Thursday, Week 4, 19:00)

  • Namkha: a live capture session — frames firing on interval + click, dedup and idle-pause visibly working, events arriving over the subscription.
  • Group A: a fresh-install run through onboarding to first capture, with live status updating; under four minutes.
  • Group B: OCR text extracted from a live frame on macOS, and the Windows path + circuit breaker demonstrated (VM is fine).
  • Group C: email-code sign-in end to end; the encrypted on-disk config shown as gibberish; a consented frame uploaded with its consent token.

9 · Risks watch list

  • The capture contract is the critical path. If Namkha's handoff slips, B and C slip with it. Flag any slip in the Tuesday standup, not on Thursday.
  • macOS TCC / uIOhook abort. Getting Accessibility wrong crashes the app at the C level. Implement the precheck + relaunch once, test the cold-grant path.
  • Sleep/wake. The v1 capture timer kept firing through sleep and spammed screenshots on wake. The engine must pause/resume on powerMonitor; verify it.
  • Consent default. The security plan says training consent is opt-in; some older wireframes show it on by default. [CONFIRM with Namkha: the consent defaults are off — reconcile any wireframe that shows otherwise.]
  • Auth being behind. If C's auth is not done by mid-W3, A's onboarding funnel and every authenticated screen downstream slip. Auth is the first job.

10 · When something is wrong

Blocked on a hand-off? Say so in the Tuesday/Thursday standup and post it in #swe-team the moment it happens — do not wait for the demo. A red CI is fixed before anything else. A non-trivial decision (a token shape, a consumer API, a permission flow) goes in the PR description as a 4-7 line "Decision" note. If you and another group disagree on a contract, it goes to a 10-minute call and Namkha decides. The wrong move is to silently build around a blocker; the right move is to surface it early.
