OnlyWorks Desktop — Architecture Review (for v2 rebuild)

Source: code audit at 2026-05-28. This document tells the v2 team what to throw away, what to keep, and what to port verbatim.

TL;DR

The codebase is a 3-year-old Electron app that grew organically into a fully imperative innerHTML + window.* globals SPA glued to a ~169-channel IPC surface. Throw away the entire renderer layer (vanilla JS modules, app.html script-tag loading, onclick handlers, 5,700-line CSS file) and the IPC bridge as written. Keep the native plumbing: OCR helpers, capture-service.js event/duplicate/inactivity logic, auto-updater flow, OAuth + token refresh, and the src/ipc/ domain split as a pattern (not the files themselves). The two-pipeline capture system is over-engineered for what is essentially "screenshot + extract text + send to backend."

Stack: What's Outdated

  • src/app/app.html (386 lines) — 37 <script> tags loaded in a strict order, with a hard-coded "load order matters" comment. The first time an intern adds a module out of order, things break silently. There is no module system, no tree shaking, no bundler.
  • src/app/modules/exports.js (206 lines) — manually re-exports ~100 functions onto window.* so that onclick="..." attributes in innerHTML strings can find them. Every new function requires a touch to this file or it's silently dead.
  • src/app/modules/page-dashboard.js (22 KB) / page-ow-profile.js (47 KB) — each page is one giant function that string-concatenates HTML with ${} interpolation, with sanitizeHTML() calls sprinkled defensively. 92 inline onclick= handlers across src/app/modules/. Any tab switch tears the entire DOM down and rebuilds it.
  • src/app/styles/styles.css (123 KB / ~5,700 lines) — single file, no preprocessor, no scoping. Class-name collisions managed by social convention (the CLAUDE.md rules about ow-* and the deleted .btn-*/.modal-* classes are evidence of firefighting).
  • src/preload.js (444 lines) — every IPC channel hand-mirrored. No types, no codegen, no validation. Renderer can't know what's available without grepping. ~169 channels exposed (counting nested auth.*, teams.*, etc. — there are 109 actual ipcMain.handle callsites in src/ipc/, the rest are event listeners and aliases).
  • vendor/anime.min.js + animate.min.css in app.html plus the 150-line inline <script> at the bottom — a different animation paradigm than the rest of the code, and the inline IIFE attaches IntersectionObserver + sparkle effects to the global DOM. Pure noise in 2026.
  • Electron 27 (current is 33+). At least two majors behind on security patches.
  • tesseract.js shipped as a fallback even though both macOS Vision and Windows.Media.Ocr exist on the platforms that matter.

Stack: What's Worth Keeping (conceptually)

  • src/ipc/ domain split (auth / teams / reports / sessions / etc.) is the right shape — it just needs a typed router instead of hand-rolled registerXHandlers(context) boilerplate.
  • components.js factories (owButton, owModal, owPanel, owStatCard, owEmptyState) — the API is clean and maps 1:1 to a React/Solid component model. The bodies are HTML strings, but the call sites are correct.
  • Utility-class CSS (ow-flex, ow-mb-lg, ow-text-muted, ow-gap-sm visible all over page-dashboard.js) — already Tailwind-shaped, just hand-rolled.
  • src/core/ lifecycle split (window-manager, tray-manager, menu-manager, lifecycle, auto-updater-flow, pipeline-init, token-refresh, oauth-flow, error-handlers) — clean separation, mostly portable.
  • OnlyWorksApp class with a context object passed to all IPC handlers (src/main.js:191) — a primitive DI container. Decent pattern, just needs typing.
  • macOS TCC snapshot at launch (_launchTrustState in main.js) — clever, hard-won, must survive the rewrite.
  • Domain-specific native handling: macOS active-win + AppleScript fallbacks, Windows active-win + PowerShell fallback, Mica backdrop, Windows timer throttling switches. Months of bug-fix learning.

Recommended v2 Stack

Optimize for: 3 interns who can ship features without breaking each other.

  • Shell: Electron, not Tauri. Tauri sounds appealing (Rust core, smaller binary), but the rough edges that matter to OnlyWorks — desktopCapturer, globalShortcut, system tray, deep links, auto-updater parity, native uIOhook bindings, sleep/wake events, screen recording permission flow, NSIS + DMG with signing — all are first-class in Electron and DIY in Tauri. Interns will learn faster on the bigger ecosystem. Save Tauri for a later optimization pass.
  • Renderer: React + TypeScript + Vite. Boring, default, hireable. Solid/Svelte are slightly better tech but interns will have written more React already, and the docs / Stack Overflow surface for React is unbeatable. TypeScript is non-negotiable for a team this size — it replaces a huge amount of the runtime defensive code (sanitizeHTML calls, if (result?.data?.user) chains, window.* global lookups).
  • Routing: TanStack Router (or react-router). The current "blow away #main-content and rebuild" pattern in router.js becomes obsolete.
  • Styling: Tailwind v4 + shadcn/ui. The team is already 60% there with the ow-* utility classes. shadcn gives copy-pasteable components that interns can read and modify — better than a black-box component library.
  • State: TanStack Query for server state + Zustand for UI state. No Redux. The current pattern is window.AppStore + manual cache invalidation (invalidateReportsCache in exports.js) — React Query replaces 100% of that.
  • Forms: React Hook Form + Zod. Replaces owFormField + manual validation.
  • Build: Vite + electron-vite or electron-forge. Hot reload in the renderer is a massive intern-velocity win.
  • Testing: Vitest (already in use — keep it). Add Playwright for end-to-end if the team grows.

IPC / Backend Communication

109 ipcMain.handle callsites + ~60 nested aliases on the preload bridge. Two structural problems:

  1. Many of them shouldn't exist at all. Channels like getDailyAnalytics, getWeeklyAnalytics, get-platform-usage, reports:getUserReports, teams:getMembers, goals:getGoals — every one of these is a thin pass-through to a backend HTTP call. Have the renderer talk to the backend directly via fetch + auth header. The desktop's only reason to proxy via IPC is to attach tokens stored in electron-store, which can be solved by exposing a single signed apiCall(method, path, body) channel — or by minting a short-lived JWT into the renderer at session start.
  2. What remains is genuinely native and should stay IPC: capture/screenshot, OCR, permissions, deep-link OAuth, auto-update, tray, window controls, system info, settings persistence, clipboard. Maybe ~20-30 channels.

Recommendation: Use tRPC over IPC or a thin typed electron-trpc-style bridge. A single electronAPI.invoke<'capture.start'>(sessionId) signature with full TS inference. Replaces preload.js (444 lines) with a generated client. ~3 interns × ~2 days to wire up the router.

Capture Pipeline: Unified Single-Trigger, Multi-Consumer

capture-service.js is 1,417 lines. It does (deep breath): screenshots on interval + on click + on Enter + on clipboard copy; SHA-256 dedup; per-app time tracking; idle period tracking; inactivity warnings + auto-stop; window-title change counting; two output pipelines (activity-buffer-service for OCR text, training-data-service for high-res ML frames); platform-specific active-window detection (3 fallback paths per platform); uIOhook input listeners; circuit breakers for everything.

The current two-pipeline split is the wrong abstraction. It's organized by what data each pipeline emits, which led to duplicated triggers, duplicated dedup logic, and a 1,400-line god class. The v2 model is one capture trigger system feeding multiple consumers.

This is also a strategic requirement: training data collection and report generation must share one capture pipeline. The same screenshot serves both jobs:

  • For reports → OCR locally → text + metadata → backend → AI interpretation → portfolio
  • For training → full frame (consent-gated) → backend storage → labeling → eventual fine-tune of an OnlyWorks proprietary model (replaces Gemini at scale)

What v2 architecture looks like:

CaptureEngine (1 file, ~400 lines)
  ├─ triggers: interval, input, clipboard, manual
  ├─ emits: { jpeg, meta, hash, triggerType, sessionId } via EventEmitter
  └─ guards: dedup (SHA-256), debounce, idle/inactivity, sensitive-content filter

Consumers (1 file each, ~100-150 lines, all subscribe to the same event stream)
  ├─ OcrConsumer: macOS Vision / Windows OCR → extracted text → POST /api/pipeline/activity/stream
  ├─ TrainingDataConsumer: full-res frame (consent-gated) → POST /api/pipeline/training/frames
  ├─ MetricsConsumer: app/window-title tracking, per-app time buckets, session metrics
  └─ (later) ThumbnailConsumer: small preview for "what we captured today" UI

Why this matters for the model-training bet:

  • One pipeline, two destinations: training data and report data come from the same trigger system; no duplicate captures, no duplicate uIOhook listeners, no duplicate dedup.
  • Consent layer: users opt into training data sharing separately from the core product. The TrainingDataConsumer honors the opt-in flag. Workers who decline still get a fully functional product; the data just doesn't flow to the training set.
  • Sample rate flexibility: OCR consumer might run on every frame; training consumer can sample 1-in-N or trigger only on significant-change frames. Independent of each other.
  • Long-term cost trajectory: Gemini calls are bounded ($0.50/user/day vision cap) but not cheap. Training your own multimodal "work activity interpreter" replaces Gemini at scale — needs the data flowing from day one of v2.0 to be ready in 12-18 months.
  • Privacy story: sensitive content filter (sensitive-filter.js pattern from v1) is the SAME filter for both consumers. One privacy boundary, two output paths.

OCR on-device stays. Reasons: (a) latency/cost — OCR locally is <100ms free vs ~1-2s + $ per multimodal call, (b) bandwidth — sending text to backend is ~1 KB vs ~150 KB per JPEG, (c) privacy story, (d) works offline. macOS Vision and Windows.Media.Ocr are already great. Drop tesseract.js entirely. If Vision/WinOcr isn't available, skip OCR.

process-detection-service.js (635 lines) and the integration-service.js (679 lines) likely still fold away — those are reading running processes for "what apps are open" detection. Just call active-win once per screenshot in the CaptureEngine.

UI Layer

Move to React/TSX without ceremony. The failure modes of the current pattern are real and ongoing:

  • sanitizeHTML(userName) must be remembered at every interpolation site — one miss is an XSS.
  • onclick="showSection('dashboard')" requires showSection to be on window. Refactoring renames break invisibly.
  • Tab switches blow away the DOM, so any open modal, input focus, scroll position, or transient state is lost — reports/reports-history.js re-fetches from network every time.
  • IntersectionObserver + anime.js in app.html fights with the innerHTML lifecycle: anything dynamically rendered after the initial mount never gets observed.

The owButton/owModal/owPanel/owStatCard/owEmptyState/owFilterBar/owFormField/owLoadingState API maps directly to React components — same prop names, same variants. Port the API, drop the implementation. A <Button variant="primary" size="lg" onClick={startCapture}>Start Session</Button> reads identically to owButton('Start Session', { variant: 'primary', size: 'lg', onclick: 'startCapture()' }). Interns who learn one learn the other.

Pages become composable: <DashboardPage /> mounting <SessionPanel />, <DailyBriefing />, <RecentSessions />, <SkillsSection /> — each a file under 200 lines instead of the current 1,500-line page-*.js monoliths.

Build / Dist

Mostly fine — keep the bones:

  • electron-builder config in package.json is solid: GitHub releases publishing, NSIS + DMG with custom layouts, Apple notarization via afterSign hook, Mica on Windows, asarUnpack list correct for native modules. Port verbatim.
  • electron-updater with latest-mac.yml/latest.yml is the industry standard. Tauri's updater is functional but you'd be rebuilding this. Stay.
  • GitHub Releases as dist channel is fine for current scale. Once you cross ~10k installs or want analytics/staged rollouts, look at Update Server (self-hosted), Sparkle (mac native), or a paid service like Tauri Update Server / Squirrel. Not urgent.
  • Code signing complexity is unavoidable. Apple notarization works via @electron/notarize (already wired). Windows EV cert (currently verifyUpdateCodeSignature: true with signAndEditExecutable: false is unusual — verify with an intern day-1 that this actually signs on CI).
  • Add CI signing (currently appears manual via .env.local) — GitHub Actions with secrets, automate the whole release. Great first intern task.

Code That Can Be Ported vs Must Be Rewritten

Port nearly verbatim (the hard-won, platform-specific stuff):

  • src/native/ocr_helper/ (Swift binary source + Windows PowerShell) — keep as-is, wrap in TS
  • src/services/ocr-service.js (629 lines) — drop the Tesseract fallback path (~200 lines saved); port the rest. Platform detection, binary discovery, circuit breaker for Windows are correct.
  • src/services/auto-updater.js (640 lines) + src/core/auto-updater-flow.js — port. Startup screen + skip/install flow is well-thought.
  • src/services/backend-auth.js (561 lines) + src/core/token-refresh.js + src/core/oauth-flow.js — port. Token refresh on sleep/wake (main.js:288) is a real-world fix you don't want to relearn.
  • src/services/windows-preflight.js + src/services/permissions.js — port
  • src/services/user-settings.js — port (electron-store-backed, simple)
  • The electron-builder config block — copy

Heavy rewrite but keep the algorithm:

  • capture-service.js — port the duplicate detection, debouncing, inactivity logic, time tracking, active-window detection with fallbacks. Restructure as CaptureEngine + consumers.
  • session-service.js (754 lines) — port the lifecycle, drop anything that's now a direct backend call.

Rewrite from spec, throw out the code:

  • Everything under src/app/ — all 32 module files, app.html, styles.css, exports.js, router.js, tutorial.js. ~7,000+ lines gone, ~2,500 lines of React replaces it.
  • preload.js (replaced by typed bridge)
  • src/ipc/*.handlers.js (109 handlers — most fold into direct backend calls; remaining ~25 get rewritten in the typed router)
  • src/services/integration-service.js, process-detection-service.js, todo-service.js, cleanup-service.js, supabase.js (1,863 lines!) — these are the strongest "is this needed at all?" candidates. The supabase.js size suggests the desktop is doing DB work it should delegate to the backend.

Keep operational, even if no v2.0 UI ships against it:

  • src/ipc/teams.handlers.js + backend teams services + team_* tables — the team feature is repositioned for v2.2 as "internal reporting" (status reports written for you). Do not delete the team infrastructure. The v2.0 worker desktop just doesn't expose any team UI; v2.2 rebuilds the UI with consent-first positioning.

Note on training-data-service.js: in v2 this becomes a consumer on the unified CaptureEngine (the TrainingDataConsumer described above), not a separate parallel pipeline. Keep the algorithm (sample rate, app-category labeling), restructure as a subscriber to CaptureEngine events.

Already gone: the Untitled-1 file in src/services/ is editor debris — delete it.

Risks of Rebuild

  • The capture pipeline has years of platform bug fixes embedded. macOS TCC mid-session permission grants (_launchTrustState), Windows background throttling switches, uIOhook crashing libuv when accessibility isn't granted, active-win vs PowerShell fallback ordering, OCR cold-start latency, Tesseract being unable to load through asar — every one is a 1-2 week debugging session waiting to happen. Mitigation: port capture-service.js and ocr-service.js with minimal restructuring; do not rewrite from scratch.
  • Auth + auto-update are silent killers. They look simple, then ship a broken build and users can't update to the fix. Mitigation: port backend-auth.js, auto-updater.js, and the OAuth flow code as a single unit; do not refactor them in the same PR as anything else.
  • Backend coupling. Reducing IPC by moving calls to the renderer requires every endpoint to accept the desktop's JWT and exposes CORS/origin questions. Mitigation: keep IPC for auth-bearing calls in v2.0; migrate to direct calls in v2.1 once the backend has a clean public API.
  • Scope creep into a rewrite-everything-at-once trap. Three interns + founder rewriting in parallel = merge hell. Mitigation: put the new app in a sibling directory (/Users/xerneas/onlyworks_desktop_v2), ship to internal users only for 6-8 weeks before flipping the auto-update channel. Run both codebases in parallel until v2 has feature parity on the top 3 user flows only. Cut everything else from v2 scope.
  • Intern onboarding is the actual bottleneck. A modern Vite + React + TS + Tailwind + tRPC stack is more moving pieces than vanilla JS + innerHTML, even if each piece is simpler. Mitigation: week 1 of intern time is "build a tiny Electron + Vite + React + tRPC todo app from a template" before touching OnlyWorks. Pin every dependency.
