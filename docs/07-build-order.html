OnlyWorks v2.0 — Build Order

Week-by-week task list. 4 people: Namkha (founder) + Intern A (Stack & UI) + Intern B (Auth & Capture) + Intern C (Capture Engine).

Each row in the per-week tables: who owns it + what gets shipped + dependencies. Tasks are sized to ~1 day unless noted; if a task is longer, it's split.

Definition of Done (per phase)

A phase is done when:

  • All listed tasks merged to main
  • CI builds green (signed + notarized on both platforms)
  • End-of-week checkpoint demo works for a fresh-install user
  • No P0/P1 bugs open

If a phase can't be marked done, stop and fix before starting the next phase. Don't pile work on a broken foundation.

Communication Cadence

  • Daily standup (async on Discord, 5 min): yesterday / today / blockers
  • Weekly demo Friday: 30 min, each person shows what shipped
  • Friday retro (30 min): what worked, what didn't, adjust week+1 plan
  • Pair programming sessions: Namkha + each intern, 2x/week for weeks 1-4
  • No-meeting Wednesdays: deep work only

Tools

  • Code: GitHub (existing repo, new directory)
  • CI: GitHub Actions
  • Issue tracking: GitHub Issues + labels (week-1, intern-a, priority-p0, etc.)
  • Communication: Discord — one channel for the team
  • Design: Figma for any mockups (free tier)
  • Docs: This CLAUDE/planning/ directory — keep current

When something goes wrong

Hierarchy of fallbacks:

  1. Ask the team in Discord — likely someone has hit it
  2. Read v1 code — most platform-specific issues have an answer there
  3. Check 01-component-inventory.md and 03-architecture-review.md — they document the gotchas
  4. Ping Namkha — for stuff that needs decisions or really doesn't have an answer
  5. Open a GitHub Issue with needs-decision — if it'll block multiple people

Do not silently pivot the design. If a screen needs a structure change, propose in Discord, get a thumbs-up, then change.
