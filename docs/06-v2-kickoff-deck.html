OnlyWorks v2

Kickoff for the rebuild

Team: Namkha + 3 interns Duration: 12 weeks to v2.0 launch Date: 2026-05-28

Why we're here

  • v1 has ~14 active users and a 5,700-line CSS file
  • 109 IPC channels, 47KB single-page components, two parallel session APIs
  • Subscription is stubbed. No Stripe. No revenue.
  • The team feature got built but was framed as bossware
  • The hiring marketplace got built but the paid tier is unreachable
  • We've built a lot. We've sold zero things. Time to rebuild around a clear story.

What v1 taught us

Hard-won Lesson
Verified-by-capture skills Real differentiator — strangers actually want this
Forensic auditor > supportive mentor prompt Don't let AI hallucinate praise
Privacy filtering Load-bearing — without it the product can't ethically exist
Caches lie Build durable storage for high-value derived data
Hardcoded enums kill features Schema/UI must accept what the AI produces
OAuth in embedded BrowserWindow doesn't work System browser + polling is the only durable pattern

The validation signal

When asked what they liked, users outside our network said:

"Building resume, verifying skills, and hopes of getting hired by the managers on our platform."

This is the product. Not "AI productivity tracker." Not "workplace use." Not "screenshot tool."

Verified work history → matched to hiring managers → get hired.

Who we build for

Three ICPs, one product:

Segment Who Job
A. Students & early-career CS undergrads, design students, bootcamp grads, interns Build projects with OnlyWorks running → portfolio that gets first job
B. Mid-career ICs 3-15 yr eng/PM/design at digital companies Passive career history, get found by recruiters, promo prep
C. Teams (B2B) Companies tired of writing status reports Auto-generated reports from real work — save 30 min/week/person

Plus: Hiring managers (the people who pay)

The unified capture pipeline

CaptureEngine (one trigger system)
  ├─ triggers: interval, click, Enter, clipboard
  ├─ frame: { jpeg, sha256, sessionId, app, windowTitle }
  └─ guards: dedup, debounce, idle, sensitive-content filter

Consumers (all subscribe to same frame stream)
  ├─ OcrConsumer       → text → backend → AI → portfolio
  ├─ TrainingConsumer  → full frame → backend → OUR future model
  └─ MetricsConsumer   → app/time/session stats

Why unified: one dedup, one consent boundary, one privacy filter. Training data and report data come from the same source.

The long bet: in 12-18 months we replace Gemini with our own fine-tuned multimodal model. Data collection starts day 1.

Backend:

The stack

Picked for 3 interns shipping in parallel without breaking each other:

Layer Choice Why
Shell Electron 33+ Battle-tested, Tauri rough edges hurt us
Renderer React + TypeScript + Vite Boring, hireable, intern-friendly
Styling Tailwind v4 + shadcn/ui Copy-pasteable, no black box
State TanStack Query (server) + Zustand (UI) No Redux ever
IPC tRPC over IPC Replaces 444-line preload.js with typed bridge
Forms React Hook Form + Zod Compile-time validation
Build electron-vite Hot reload in renderer
Tests Vitest (kept from v1) Add Playwright later

What gets ported verbatim from v1

The hard-won, platform-specific stuff:

  • src/native/ocr_helper/ (Swift binary + Windows PowerShell)
  • src/services/auto-updater.js + auto-updater-flow.js
  • src/services/backend-auth.js + oauth-flow.js + token-refresh.js
  • src/services/windows-preflight.js
  • The macOS TCC permission relaunch trick (_launchTrustState in main.js)
  • electron-builder config
  • Sensitive content filter logic

Rule: don't refactor while porting. Get it working as-is. Refactor later.

How we work

  • Daily 15-min standup (async OK if it's just on Discord)
  • One PR per task — small, reviewable, mergeable in a day
  • Pair programming twice/week for the first 4 weeks (Namkha + each intern)
  • Code review: Namkha reviews every PR weeks 1-4. After that, interns review each other.
  • Pin every dependency — no auto-upgrades during the 12 weeks
  • Branches: main is always shippable. Feature branches off main. Squash-merge on approval.
  • Failed CI = no merge. Hooks + tests must pass.

No surprises: if something will take longer than estimated, say so on day 1, not week 3.

Week 1 — what you actually do Monday

Before touching OnlyWorks code, each intern builds a tiny Electron + Vite + React + tRPC todo app from a template. Goals:

  • Understand the stack we picked
  • Wire up one tRPC procedure end-to-end
  • Build + sign locally (macOS or Windows)
  • 1-day exercise, max 2 days

Then Monday week 2:

  • Intern A → scaffold onlyworks_desktop_v2/
  • Intern B → start porting auto-updater.js
  • Intern C → start CaptureEngine skeleton
  • Namkha → Stripe Checkout endpoint

See 07-build-order.md for week-by-week task list.

Decisions we lock today

  1. Repo: new onlyworks_desktop_v2/ directory in this repo, merge to main on v2.0 ship
  2. Naming: ship as "OnlyWorks 4.0" — continue version line, no rebrand
  3. Update channel: opt-in beta until 4 weeks bug-free, then auto-update default
  4. HM pricing: $99 / $299 / $599 (confirm now or never)
  5. Student tier: free Pro for .edu email — yes/no
  6. Training data T&C: Namkha + light legal review by week 4
  7. Capture all v1 critical context now so we don't relearn it: TCC relaunch, uiohook, OCR fallback paths

What success looks like in 12 weeks

By end of week 12:

  • 100 weekly active workers (60% from outside our network)
  • 5 paid hiring managers (500−1,500 MRR)
  • 80% training data opt-in rate (clear consent UX signal)
  • 30+ student users via 1-2 campus partnerships
  • 1M+ training frames collected
  • Zero capture-pipeline bugs for the last 4 weeks

Beyond v2.0:

  • v2.1 (M4-6): Promo packet + weekly summary + LinkedIn OAuth
  • v2.2 (M7-9): B2B internal reporting feature — the second revenue stream
