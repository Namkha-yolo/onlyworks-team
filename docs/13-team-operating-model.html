OnlyWorks Team Operating Model

The operating system for a four-person engineering team. Read this once; disagree in writing; we merge the changes.

This doc replaces the "intern playbook" framing. OnlyWorks is built by colleagues — founder plus three engineers — with overlapping responsibilities and clear zones of ownership. Flat teams work only when the implicit becomes explicit. Without these patterns, every question routes through Namkha, Namkha becomes a bottleneck, and engineers can't grow into the senior shape they deserve to grow into.

1 · Ownership zones

OnlyWorks has four zones. Each zone has one owner. The owner is the buck-stops-here on architecture, code quality, tests, monitoring, on-call response, documentation, and hiring input as the team grows.

Ownership does not mean exclusive coding rights. Anyone can open a PR in any zone. But the owner approves architecturally significant changes, sets direction, and is who you ping at 2am if production is down.

Zone Owner Surface
Product · Design · Business Namkha Product strategy, design system custody, fundraising, GTM, customer conversations, founder-led sales, hiring
Desktop (Electron + capture) Engineer A Electron main + renderer, capture engine, OCR, on-device redaction, auto-updater, native bridges (Swift / PowerShell), platform packaging + signing
Backend + AI pipeline Engineer B Express API, Supabase schema + RLS, AI orchestration (Gemini), guardrails service, sessions + reports services, training-data pipeline, billing webhooks
Web (marketplace + public profile) Engineer C HM search, HM dashboard, job posting, public worker profile, marketing site, web auth, OG previews

What ownership includes

  • Writing RFCs for non-trivial changes in your zone
  • Approving PRs that materially touch your zone
  • Maintaining documentation in your zone (READMEs, runbooks, decision logs)
  • Setting up monitoring and alerts for your zone
  • Being primary on-call when alerts fire in your zone
  • Proposing your own roadmap for the next quarter

What ownership does not include

  • A monopoly on coding in your zone — others contribute freely
  • Veto power over cross-zone decisions — those go through RFC
  • Permission to change the schema, public API, or design tokens without RFC
  • Solo authority on anything in the "we decide together" row of § 4

Cross-zone protocols

Most real work touches more than one zone. Three patterns:

  • API contract changes (Desktop ↔︎ Backend, Web ↔︎ Backend) → RFC required. Producer owns the change; consumer must approve.
  • Shared types in shared/types.ts → PR with both producer and consumer owners as required reviewers.
  • Schema changes in Supabase → RFC always, no exceptions. Migrations land via PR with backend owner as required reviewer.

Zone refresh

Zones get re-balanced at the end of each quarter, or whenever someone wants to swap. Lock-in is a smell — we should not require an engineer to own the same zone for two years if they want to grow elsewhere.

2 · RFC process

The single most important document type in our codebase is not the README. It's the RFC — a one-to-three-page written proposal for a non-trivial change.

When to write one

Write an RFC if your change is any of these:

  • Touches more than one zone
  • Changes a public API, schema, or design token
  • Introduces a new dependency or vendor that sees user data
  • Changes architecture (where state lives, what process talks to what)
  • Costs >$50/month in infra at current scale
  • Removes or deprecates a user-facing feature
  • Changes security or privacy posture (captures, PII, auth, consent)

Skip the RFC for: bug fixes inside one zone, no-behavior-change refactors, doc updates, test additions, dependency bumps to patch versions, UI polish that doesn't change information architecture.

When in doubt, write one. The cost of a 1-page RFC is one hour. The cost of not writing one and finding out you misaligned with two other engineers is one sprint.

RFC template

# RFC-NNN: <Title>

**Author:** <name>
**Status:** Draft | Review | Accepted | Rejected | Superseded
**Date opened:** YYYY-MM-DD
**Decision deadline:** YYYY-MM-DD (default: 3 working days from posting)
**Required reviewers:** <names>

## Context
Today's situation. What changed that prompted this?

## Problem
The specific problem we're solving. Who feels it? How do we know?

## Proposal
What we'll do. Concrete. Include code sketches, schema, API shapes.

## Alternatives considered
At least two. For each: what was good, why we didn't pick it.

## Tradeoffs
What we give up by choosing this.

## Migration / rollout
How we get from today's state to the proposed state without breaking things.

## Open questions
Anything unresolved. Mark blockers vs nice-to-knows.

## Decision
(Filled in when accepted or rejected.)

Process

  • RFCs live in docs/rfcs/RFC-NNN-slug.md in the repo
  • Posting an RFC = opening a PR with that file. Reviewers comment inline.
  • Default review window is 3 working days. Author can extend if reviewers need more time; can shorten if everyone has signed off early.
  • An RFC is accepted when the author + all required reviewers + Namkha approve. Non-required reviewers can comment but don't block.
  • An RFC is rejected explicitly by the deadline — silence ≠ rejection.
  • After acceptance, the implementation PR links back to the RFC.
  • Superseded RFCs are not deleted — they stay, with status updated and a link to what replaced them. Decision history is institutional memory.

What makes an RFC good

  • Specific, not vague (numbers, code, schema — not "we'll improve performance")
  • Names alternatives honestly (not strawmen)
  • Names tradeoffs (every decision has a cost)
  • Estimates effort and risk
  • Says "open questions" out loud instead of hiding ambiguity

What makes an RFC bad

  • Pre-decided ("I already started building this; please approve")
  • Over-engineered ("here's a 12-month plan to refactor everything")
  • Underspecified ("we should use X" with no how)
  • Asks for permission instead of presenting an analysis

3 · Weekly 1:1s

Each engineer has a recurring 1:1 with Namkha. 30–45 minutes, same day, same time, weekly.

The principle

The 1:1 is yours, not Namkha's. You set the agenda. Namkha listens more than they talk. The 1:1 is not a status meeting — status happens async (see § 6).

The shared doc

Each engineer maintains a shared Google Doc with Namkha titled 1:1 — <name>. Append-only by convention. Three sections:

  1. This week's agenda — bullets you want to discuss
  2. Open items — running list of things either of you owes the other
  3. Past notes — history of previous 1:1s

You add agenda items as they come up during the week. By the time the 1:1 starts, the doc is ready.

1:1 template (per week)

## YYYY-MM-DD

### Agenda (from engineer)
- 

### Notes
- 

### Action items
- [ ] (owner) — by <date>

What 1:1s are for

  • Blockers that aren't visible in standups
  • Feedback for Namkha (running the company, the team, the docs)
  • Career: where are you growing, what do you want next
  • Decisions that need a conversation and aren't worth an RFC
  • The state of your zone — risks, fires, what's worrying you
  • Things you don't want to write in a public channel

What 1:1s are not for

  • Status updates ("here's what I shipped this week") — that's Friday demo + async update
  • Architectural debates that need other engineers — write an RFC
  • Things you should be telling the whole team — share in #engineering

Cancelling

Either party can cancel a 1:1 if there's nothing to discuss. Don't burn 30 minutes on "all good!" Async updates absorb the load.

4 · Decision rights matrix

The single biggest source of friction in a flat team is unclear decision rights. This table is the resolution.

Topic Who decides
Code style within a file Engineer alone
Choice of internal helper / library inside one zone Engineer alone
Adding a dependency ≤1 MB, well-maintained, MIT/Apache license Engineer alone (note in PR description)
Adding a dependency >1 MB or with a non-permissive license RFC required
Database schema change Backend owner + RFC
Public API contract change Producer owner + RFC; consumer must approve
Choice of cloud provider, hosting, CI vendor RFC + Namkha
Adding any vendor that sees user data RFC + Namkha (security review)
Hiring, compensation, equity Namkha (with engineer input)
Product roadmap and priorities Namkha (with engineer input via RFC)
Architecture decisions spanning zones RFC + all zone owners + Namkha
Security and privacy posture RFC + Namkha (no exceptions)
Branding, design tokens, marketing copy Namkha
Naming a new module / file / API endpoint Engineer alone — but be readable
Deleting v1 code in your zone Engineer alone if dead; RFC if anyone depends on it
Refactoring inside your zone (no behavior change) Engineer alone
Refactoring across zones RFC
When to cut a release Backend owner (for backend) + Desktop owner (for desktop) + Namkha
Public-facing announcement of unreleased features Namkha

The default

When in doubt, the default is "you decide, you tell the team afterward." We optimize for speed, not committee.

Reversibility

Bezos's framing applies: irreversible decisions get RFCs; reversible decisions get tried. Ask "could we undo this in a week?" If yes, do it. If no, write it down first.

What to do when you disagree with a decision

  • Say so. We don't punish disagreement; we punish silence followed by surprise.
  • DM the relevant person, not the whole channel.
  • Bring an alternative + tradeoff analysis, not just "I don't like X."
  • Once a decision is made, commit to it. Don't relitigate in PR review.

5 · The no-asks list

Things you do not need to ask permission for. Just do them.

  • Open a PR — any time, any size
  • Refactor inside your zone (no behavior change)
  • Add a dep ≤1 MB with a permissive license — note it in the PR
  • Pick library X over Y when both are reasonable and the choice is reversible
  • Take Friday afternoon off if you shipped what you said you'd ship
  • Block 4 hours on your calendar as deep work; skip non-essential meetings
  • Decline a meeting that doesn't have an agenda
  • Skip standup on a heads-down day (post your update async in #engineering)
  • Write a blog post about your work (run drafts past Namkha for spoilers / IP)
  • Speak at a meetup or conference about your work (let us know)
  • Contribute to an open source project, including ones we depend on
  • Try a new IDE, terminal setup, AI tool, prompt — your machine, your call
  • Spend up to $200/month on tools that make you better at your job (Cursor, Linear, Figma, etc.) — submit the receipt
  • Push a hotfix to production for a sev-1 bug, then write the postmortem (§ 9)
  • Pair with another engineer for an hour without scheduling

Things you do need to ask about

  • Adding a new vendor that sees user data (PII, captures, billing)
  • Changes to the schema, the consent UX, or the privacy policy
  • Major architectural changes (RFC)
  • Anything legal, HR, or financial
  • Spending company money over $200/month
  • Speaking publicly about unreleased plans or financials
  • Replying to press, analysts, or investors on the company's behalf

The principle: the no-asks list is generous because we hire people we trust. If you abuse it, we'll talk about it in your 1:1, not in public.

6 · Async updates and standup

We don't do daily synchronous standup. We do async updates.

Format

Every weekday by 11:00 local time, post in #engineering:

Yesterday: <what shipped, links to PRs, blockers cleared>
Today: <one or two things, links to RFCs or issues>
Blocked: <anything in your way; tag people>

That's it. No "please review" pings (open the PR; reviewers see it). No status theater.

Why async

  • Different focus windows don't collide
  • Writing forces clarity
  • Reading takes 60 seconds vs 30 minutes of standup
  • Builds a searchable history of who did what

Synchronous weekly kickoff

Once a week, a 30-minute video on Monday morning. Goal: align on what's shipping this week, surface cross-zone dependencies, flag risks. Not a status round-robin. If there's nothing to align on, we cut it short.

No-meeting Wednesdays

No standups, no demos, no calls on Wednesdays. Deep work only. If you need to interrupt someone, ask first.

7 · Friday demo + retro

End of the week, 60 minutes, video or in-person.

Demo (30 min)

Each engineer shows what shipped that week. Working software, not slides. Includes bugs found, hard moments, surprising results. No prep beyond opening the app. Tone: casual.

If you shipped nothing visible (you spent the week on infra, refactors, or research), demo the artifact — a graph, a diff, a doc, the new lock-free schema. The point is shared knowledge, not theater.

Retro (30 min)

Three columns on a whiteboard or Miro:

  • What worked — keep doing
  • What didn't — fix it
  • What to try — experiments for next week

Each person adds 1–3 items per column. We discuss the top 3 most-voted items and pick action items with owners. The retro doc gets committed to docs/retros/YYYY-MM-DD.md.

Why

Demo makes work feel real and gives the team a shared sense of progress. Retro prevents friction from compounding into resentment. Skipping either is a bad smell — if there's nothing to demo, why? If there's no retro fuel, the team isn't stretching.

8 · Code review

Defaults

  • Every PR gets at least one reviewer
  • Default reviewer is a peer in or near the affected zone — not Namkha by default
  • Namkha reviews: architectural changes, anything labeled needs-decision, anything in the design system, anything touching consent / capture / security
  • Reviewers respond within 24 working hours of being requested
  • If you can't review in 24 hours, comment "can't get to this today, will tomorrow" — silent backlog is worse than late review

Reviewer etiquette

  • Be specific. "This is unclear" → "I expected X here based on the function name; can you rename or add a one-line comment?"
  • Be kind. Code review is feedback on code, not on the engineer.
  • Focus on this change. Don't ask for unrelated refactors — open a follow-up issue.
  • Approve when you'd ship it, not when it's perfect.
  • Use nit: prefix for non-blocking comments.

Author etiquette

  • Address every comment before re-requesting review
  • If you disagree, reply with reasoning — don't silently close
  • Squash before merging
  • The PR title becomes the commit message; make it readable

What we don't do

  • We don't gate-keep PRs to make ourselves feel important
  • We don't ask for major restructuring in review — that should have been an RFC
  • We don't merge our own PRs without approval
  • We don't --no-verify past failing hooks

9 · On-call and incidents

We don't have a formal rotation yet — too small. Instead:

  • Zone owner is primary on-call for their zone. Alerts route to them.
  • Sev-1 (users can't use the product): all hands. Drop what you're doing.
  • Sev-2 (degraded but functional): zone owner fixes within 1 working day.
  • Sev-3 (specific edge case): add to backlog with incident label, fix this sprint or next.

Postmortems

Every sev-1 and sev-2 gets a written postmortem within 5 working days. Template in docs/postmortems/. Blameless — we focus on the system that allowed the bug, not the person who shipped it.

# Postmortem: <one-line summary>

**Date of incident:** YYYY-MM-DD
**Severity:** sev-1 | sev-2
**Duration:** <minutes>
**Author:** <name>

## What happened
Timeline of events. No blame.

## Impact
Who was affected, how many, for how long.

## Root cause
The system-level reason this was possible.

## What worked
Detection, communication, fix — what went well.

## What didn't
Where we lost time or made the problem worse.

## Action items
- [ ] (owner) — what we're changing so this can't happen the same way

10 · Quarterly review + career conversation

Twice a year (every 13 weeks), each engineer has a structured review with Namkha. ~90 minutes.

Format

  1. Engineer writes a self-eval the week before:
    • What I shipped (top 5)
    • What I learned
    • Where I want to grow
    • Feedback for Namkha and the team
  2. Peers write a paragraph each (anonymized): what this person is great at, what they should work on
  3. Namkha synthesizes into a written review (~1 page)
  4. Conversation: walk through, no surprises, agree on next-quarter goals
  5. Compensation discussion (if applicable) happens here, not in the moment

Career conversation (separate, ~45 min, mid-quarter)

  • Where do you want to be in two years?
  • What skills do you want to build in the next quarter?
  • What support do you need from me / the team?
  • Are you growing? Are you bored?

Distinct from performance review on purpose. Career talk gets dwarfed if you yoke it to compensation.

11 · Onboarding (first two weeks)

Week 1 — Orientation

  • Day 1: laptop setup, accounts, read 08-plain-english-plan and 04-pmf-and-v2-direction
  • Days 2–4: Week 0 todo app exercise (12-week0-todo-app-guide)
  • Day 5: code review with Namkha (20-min screen share)

Week 2 — First contact with real code

  • Day 1: pair with Namkha on a tour of your zone in v2
  • Day 2: write a 1-page RFC for your first significant piece of work
  • Days 3–5: ship a small first PR — something visible enough to feel real, contained enough to ship in two days

By end of Week 2 you should have: a green-building dev environment, an accepted RFC for your first real piece of work, one merged PR in the v2 repo, and a 1:1 doc with Namkha.

12 · What this doc is not

  • Not HR policy — PTO, benefits, leave: ask Namkha. We'll formalize when we hire person #5.
  • Not legal employment terms — those live in your offer letter and the IP assignment agreement.
  • Not a job description — your zone is a starting point, not a cage.
  • Not finished — this is a living doc. Open a PR against it whenever something breaks down or could be better.

13 · Sign-off and amendments

This doc is reviewed quarterly. Changes happen via PR with at least one engineer + Namkha approving. If a section is silently being ignored, that's a signal — either the doc is wrong or the team is. We figure out which.

Sections that get the most edits over time are usually the decision-rights matrix (§ 4) and the no-asks list (§ 5). That's expected; those are the lived boundaries.
