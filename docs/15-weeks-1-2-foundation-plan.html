Weeks 1-2 — Foundation Plan

This is the instruction manual for the Foundation phase of OnlyWorks v2.0. The phase has four independent workstreams that run in parallel. Each is owned by one of four groups. By the end of Week 2, the four streams converge: signin works end-to-end, the capture engine emits events, and the renderer can call a typed procedure that flips a real backend bit.

Read your group's section in full before Monday of Week 1. Read every other group's Mission and Hand-offs sections so you know who to talk to and when.

0 · How to use this document

This document does not contain code. It contains instructions. The expectation is that you, the intern, can take an instruction like "wire TanStack Query to the tRPC client" and turn it into working code by reading the relevant docs in docs/v2-stack/, the v1 source at /Users/xerneas/onlyworks_desktop, and the upstream project's documentation.

If an instruction is ambiguous, that is a feature, not a bug. We want you to think, ask questions in your daily standup, and write a one-line proposal in the relevant PR description. The wrong move is to invent something complicated to be safe. The right move is to ask Namkha or your group lead.

Conventions used below:

  • Mission — the one-sentence purpose of the group.
  • Skill profile — who should be assigned to this group.
  • Required reading — finish before you write code. Each item points to a file in this repo or in v1.
  • Day-by-day — what should land in main by EOD that day. Days are 5-6 productive hours, not 8. If a day's work spills into the next, that is acceptable once; twice in a row means escalate.
  • Hand-offs in / out — what you must receive from another group before you can finish your work, and what others are blocked on from you.
  • Definition of done — what Namkha will check on Friday. If you can't check every box, the week isn't done.
  • Pitfalls — the specific ways previous teams have lost days on this kind of work. Read them carefully.
  • Self-check — questions you should be able to answer without looking. If you can't, you don't understand what you built.

1 · Shared operating norms

These apply to every group across both weeks.

  1. One PR per task, opened the day you start. Even if the PR is empty for the first hour. This gives everyone visibility into what is in flight.
  2. Standup Tuesday and Thursday at 14:30, 15 minutes, hard stop. Yesterday/today/blockers. If you have no blockers, say "no blockers". Don't fill time. Mon / Wed / Fri are heads-down — use the end-of-day Discord note for status on non-standup days.
  3. End-of-day Discord note in #swe-team on Discord. One sentence per person, what moved forward, anything other groups should know. This is also where you flag a hand-off that landed.
  4. Code review SLA: 4 working hours. If you opened a PR before noon, it gets reviewed before EOD. Namkha reviews everything in W1-2. Group leads (assigned by Wednesday of Week 1) co-review.
  5. No bikeshedding in PRs. If a comment thread reaches five replies, it goes to Discord, then to a 10-minute call, then resolved. Do not let style debates block hand-offs.
  6. Decisions are written down or they did not happen. If you make a non-trivial choice (which library, which file layout, which protocol), write a 4-7 line note in the PR description titled "Decision". Namkha lifts those into an ADR file weekly.
  7. End-of-week demo: Thursday at 19:00, 30 minutes. Each group shows what works. No slides. Live demo, or it doesn't count. If your demo is broken on Thursday morning, fix it first; everything else waits. Friday becomes the post-demo buffer day — ship the fixes raised in demo, finish docs, prep next week's hand-offs.
  8. Lock dependency versions. Every package installed must be pinned to an exact version, not a caret range. We are not chasing the moving edge of any library during Foundation.
  9. Branch naming: w1/groupA/scaffold-electron-vite style. Week, group, short slug. Everyone can grep the repo for their group.
  10. Never merge red. CI must pass before merge. If CI is flaky, fix the test, do not retry until green. Flaky CI in Week 1 becomes broken CI by Week 4.

2 · The four groups at a glance

Group Mission Best for Critical path?
A — Frontend / UI Platform The renderer is alive: app shell, routes, base components, query layer. Strongest React + design sensibility. Yes — repo scaffold blocks everyone in W1.
B — Desktop & IPC Platform Main and renderer talk over typed tRPC; build & signing work. Likes tooling, debugging, system glue. Yes — tRPC bridge blocks A & C in W2.
C — Auth & Secure Storage Signin works end-to-end, tokens survive restart and sleep/wake. Careful, security-minded, OAuth-comfortable. Soft — only blocks itself; gated on B's bridge.
D — Capture Engine A native event-emitter that fires on time and on input, with guards. Likes low-level, hooks, native modules. Soft — runs solo W1, pairs with B briefly in W2.

Group A is the only group whose Week 1 work blocks everyone. If A is behind on Tuesday, B/C/D should pause non-portable work and help A land the scaffold.

4 · Group A — Frontend / UI Platform

Mission

Stand up the renderer side of the app: scaffold the repository, install the UI stack, define the route tree, set up state and data layers, and make sure everything builds, signs, and runs on macOS by end of Week 1. In Week 2, layer the app shell on top and connect to the tRPC bridge that Group B ships.

Skill profile

  • Has built at least one non-trivial React app and can explain component composition without notes.
  • Comfortable with TypeScript: knows what discriminated unions and generics are, doesn't fear tsconfig.json.
  • Has used Tailwind or a similar utility-CSS framework, or is willing to read the entire Tailwind v4 docs in one sitting before Monday.
  • Has design sensibility — can look at a wireframe and reproduce spacing, hierarchy, and rhythm without being told every value.
  • Enjoys building "the platform other people build on" more than building one specific feature.

Required reading (finish before Monday of W1)

  • docs/v2-stack/electron-vite.html — read all of it. You own this tool.
  • docs/v2-stack/react.html — skim if experienced; read in full if new to hooks.
  • docs/v2-stack/typescript.html — read the "Why TS" and project conventions sections.
  • docs/v2-stack/tailwind.html — read all of it including v4 changes from v3.
  • docs/v2-stack/shadcn-ui.html — read all of it.
  • docs/v2-stack/tanstack-router.html — read all of it. Pay attention to file-based vs code-based routing.
  • docs/v2-stack/tanstack-query.html — read all of it.

  • docs/v2-stack/vitest.html — read the "getting started" section.

Week 1 — day by day

Monday — Repo scaffold and first build.

  • Confirm the Apple Developer credential is provisioned and the CI signing work is owned by Group B. You only need it to work — Group B does it.
  • Initialize the repository named onlyworks_desktop_v2 using the official electron-vite React + TypeScript template. Do this in a fresh directory, not by copying the Week 0 todo app.
  • Immediately remove example assets, demo components, and any boilerplate that isn't structurally required. The first commit is a clean template; the second commit removes demo code.
  • Pin every dependency to an exact version. No carets, no tildes. Write the rationale in the PR.
  • Open the PR by lunch. Ask Namkha and Group B for a same-day review. This must land Tuesday EOD at the latest.
  • End of day: the empty scaffold builds and opens a blank window locally.

Tuesday — Tailwind v4 and the tsconfig that everyone will inherit.

  • Install Tailwind v4 in the renderer only. Do not configure for the main process — the main process never renders UI.
  • Set up the design tokens from visuals/00-design-system.html as CSS custom properties inside your Tailwind theme. Do not invent new tokens.
  • Fix path aliases for the renderer (@/components, @/lib, etc.). Group B's tRPC client will use the same aliases.
  • Make sure the TypeScript project references between main, preload, and renderer compile cleanly. This is fiddly with electron-vite — read the docs carefully.
  • End of day: blank window now renders a single styled element using a design-system token color.

Wednesday — Routing and state primitives.

  • Install TanStack Router. Choose code-based routing, not file-based — the renderer has too few routes for file-based to be worth its learning cost.
  • Define three routes: / (Status placeholder), /portfolio, /settings. Each route renders a single heading and nothing else; it's a navigation skeleton.
  • Install TanStack Query and wire its provider at the root. The Query client uses default config — no custom cache time, no retry tuning. We optimize once we have real data.
  • Install Zustand. Create exactly one store (useUiStore with a single boolean, e.g. isDevPanelOpen) as the canonical pattern. Document the pattern in a 5-line README inside src/renderer/state/.
  • End of day: three routes navigate, one Zustand store toggles, Query provider mounted but no queries yet.

Thursday — shadcn primitives, tests, demo (19:00).

  • Run the shadcn/ui CLI and generate exactly five components: Button, Input, Card, Dialog, Tabs. Nothing else. We add more on demand.
  • Do not customize the components yet. Bring them in as the shadcn CLI emits them. Customization happens in W5 when designers see the screens.
  • Create a single "component gallery" route (only in dev builds, gated behind your Zustand isDevPanelOpen) that renders all five components in their default states. This is your self-check that they styled correctly under your Tailwind setup.
  • Install Vitest. Write one passing component test for the Button.
  • Add a pnpm test script and wire it into CI. Group B's CI pipeline must run it on every PR by Monday W2.
  • Write a 1-page README for the repo: clone, install, run dev, build a signed dev binary.
  • Demo at 19:00: clone the repo on a fresh machine (or simulated fresh), pnpm i, pnpm dev, navigate three routes, open the component gallery, run pnpm test. If any step fails, the week is not done.

Friday — Buffer.

  • Fix anything raised in Thursday's demo.
  • Polish the README based on questions other groups asked while cloning.
  • Confirm with Group C that they can build their magic-link views on top of your shadcn primitives on Monday — share component locations and any layout helpers.
  • If everything is green, this is a half-day. Don't invent work.

Week 2 — day by day

Monday — App-shell layout.

  • Build the persistent app-shell layout: top bar with brand mark, left nav with three nav items, content outlet using TanStack Router's outlet primitive.
  • Match the spacing and typography from visuals/00-design-system.html. If something is missing from the design system, ask Namkha; do not guess.
  • Active route is visually highlighted in the nav.
  • End of day: navigating any of the three routes feels like one app, not three pages.

Tuesday — Prepare for the tRPC hand-off.

  • Group B lands the tRPC bridge today. Until it merges, prepare: re-read docs/v2-stack/trpc.html end-to-end, including the electron-trpc client section.
  • Stub a placeholder "capture.start" button on / that calls a no-op function. Once B's bridge merges, swap the no-op for the real client.
  • End of day: button exists, no-op logged, ready to swap.

Wednesday — Wire Query to the bridge.

  • Replace your placeholder client with the electron-trpc client. Use a single React Query mutation hook for capture.start.
  • The mutation should show a pending state (button disabled), a success state (toast or inline message), and an error state. Use the shadcn Dialog only for errors — successes can be inline.
  • Add one TanStack Query useQuery for a trivial read procedure (Group B will ship capture.status for this purpose; if they haven't, ask). The point is to exercise both a mutation and a query.
  • End of day: click button → real tRPC call → main process logs it → button shows success.

Thursday — Type-safety, tests, demo (19:00).

  • Confirm the tRPC client types flow into Query hooks. If you find yourself writing any or as unknown as, the import chain is wrong — talk to Group B.
  • Write one Vitest test that mounts the Status route with a mocked tRPC client and asserts the button toggles a status query result. This is the template for all renderer tests going forward.
  • Verify Tailwind purge / content paths cover all source files. Build a production bundle and inspect CSS size; if it's over a megabyte something is wrong.
  • Pair with Group C earlier in the day so their magic-link views render correctly inside your app shell (their signin views must reuse your shadcn Input and Button).
  • Demo at 19:00: launch, nav, click capture.start(), see status update, run tests, walk Group C's signin flow.

Friday — Buffer.

  • Fix anything raised in Thursday's demo.
  • Tighten the "how to ship a feature" section of the README based on what tripped you up this week.
  • Half-day if green.

Hand-offs

Receive from:

  • Group B — CI pipeline (Tue W1), tRPC bridge (Tue W2).
  • Group C — magic-link views to embed in app shell (Thu W2).
  • Namkha — design-system tokens and any clarifications on the visual language.

Ship to:

  • Everyone — the repo itself (Tue W1).
  • Group C — shadcn primitives and the app shell to embed signin views in (Mon W2).
  • Group B — a renderer surface that exercises tRPC end-to-end so they can validate their bridge (Wed W2).

Definition of done

  • Repo exists, builds, signs in CI on macOS, opens to a non-empty window.
  • Three routes navigate; app-shell layout persists across them.
  • Tailwind v4 installed, design tokens wired, five shadcn primitives present and matching defaults.
  • TanStack Router + TanStack Query + Zustand mounted at root.
  • One Vitest unit test and one Vitest integration test pass; CI runs both on every PR.
  • Renderer calls one tRPC procedure end-to-end and renders the result.
  • README in repo root explains clone-to-run in fewer than 10 commands.

Pitfalls

  • Picking file-based TanStack Router by default. File-based shines at 30+ routes. We have three. Code-based is faster to learn and easier to debug; pick it and move on.
  • Customizing shadcn components before screens exist. You will throw the customizations away. Resist the urge to make the Button "ours" this week.
  • Tailwind v4 content paths. v4 changed how content scanning works. Read the migration notes; do not trust v3 examples from blog posts.
  • Mixing main-process and renderer-process types in one folder. Keep them in separate tsconfig project references. Main code must never import renderer code, and vice versa.
  • Building before signing is ready. A binary that builds but does not sign is not a deliverable. Coordinate with Group B early.
  • Shipping the Week 0 todo app code into v2. The Week 0 exercise was practice. v2 starts clean from the official template.

Self-check

  1. Why are we using electron-vite instead of plain Vite?
  2. What is the difference between the main, preload, and renderer processes in our repo, and which one runs React?
  3. What does TanStack Query give us that useEffect + fetch does not?
  4. Why is the Zustand store kept tiny and why is data fetched via Query instead?
  5. If a designer changes the brand purple, how many files do you edit?
  6. What is the production CSS bundle size and why is it that size?

Monday — Sign-off and signing.

  • Confirm with Namkha that the Apple Developer account, certificates, and notarization credentials are provisioned and available as GitHub Actions secrets.

  • End of day: workflow file drafted, ready to layer onto the repo once it exists.

Tuesday — CI green on Group A's scaffold.

  • Within an hour of Group A's scaffold merging, open a PR adding the GitHub Actions workflow.
  • Drive the workflow to a green build on macOS. Expect to spend most of the day on environment quirks: Node version mismatch, pnpm cache location, missing build deps. Document each fix in the PR.
  • Do not attempt notarization on the first run. Get unsigned build → signed build → notarized build in three distinct commits.
  • End of day: signed (not yet notarized) macOS build artifact downloadable from a green CI run.

Wednesday — Notarization and electron-rebuild.

  • Add notarization. Use electron-builder's built-in notarization step rather than a separate tool — fewer moving parts.
  • Add an electron-rebuild step in the workflow. There are no native modules yet, but Group D needs the recipe to exist by end of Week 1.
  • Test by adding a no-op native dependency (e.g. a small native package from the registry) in a throwaway branch, rebuilding, then removing the dependency. The point is to prove the recipe.
  • End of day: notarized macOS artifact, electron-rebuild step verified.

Thursday — Local dev experience, tRPC spike, demo (19:00).

  • Document the local dev workflow in the repo README jointly with Group A: how to run, how to enable devtools, how to inspect the main process, how to test a signed build locally.
  • Make sure pnpm dev works on a fresh clone in under two minutes including install.
  • Add a pre-push git hook (using a tool the team likes; suggest simple-git-hooks) that runs pnpm typecheck and pnpm test. The hook is opt-in via local setup; do not force-install it.
  • Spike in parallel: choose between electron-trpc and a hand-rolled IPC-over-tRPC implementation. Read the electron-trpc source and recent issues. Look at maintenance cadence, bundle cost, and how it handles streaming subscriptions (we need subscriptions in W3 for capture events).
  • Write a 1-page decision doc in the PR description: trade-offs, recommendation, what we lose if we pick the wrong one. Present it during the demo so Namkha can decide live.
  • Demo at 19:00: green CI run producing a signed, notarized macOS .dmg; walk the bridge decision 1-pager.

Friday — Buffer and decision lock.

  • If Namkha didn't lock the bridge decision in the Thursday demo, get it locked Friday morning. Monday W2 starts implementation — no decision means no Monday.
  • Fix any CI flake raised in the demo.
  • Stub the chosen bridge's file layout in a draft PR so Monday starts at the keyboard, not in the editor menu.

Week 2 — day by day

Monday — Implement the bridge.

  • Implement the tRPC bridge based on Friday's decision. Start with exactly one procedure: capture.start(). It does nothing real yet — it logs to the main process console.
  • Wire it through the preload script with proper context isolation. The preload script exposes the minimum surface possible; everything else flows through the typed bridge.
  • Confirm types flow from main to renderer cleanly.
  • End of day: bridge merged behind a feature flag if not ready; on main if it is.

Tuesday — Ship the bridge end-to-end.

  • The bridge must merge by EOD. This is a critical-path day.
  • Add capture.status() as a query procedure (returns a boolean) so Group A has both a mutation and a query to wire on Wednesday.
  • Write a 5-minute walkthrough doc in the repo: how to add a new procedure, how to call it from the renderer, how to add a Query hook. This unblocks every other group adding procedures going forward.
  • End of day: Group A unblocked.

Wednesday — Windows build path.

  • Add Windows runner to the GitHub Actions workflow. Build only, no signing yet — Windows code-signing certs are a separate procurement.
  • Expect Windows-specific issues: path separators, line endings, native module rebuild differences. Document each.
  • End of day: unsigned Windows .exe artifact from CI.

Thursday — uiohook-napi pairing, profiling, demo (19:00).

  • Group D added uiohook-napi as a real native dependency earlier in the week. Pair with them in the morning to make sure the electron-rebuild step in CI handles it correctly on both macOS and Windows.
  • Profile the bridge: how fast does a procedure round-trip from renderer to main and back? Document the number. We'll watch it across the project.
  • Demo at 19:00: live CI run producing a signed macOS build with a working native module loaded and a working tRPC procedure call.

Friday — Buffer.

  • Fix any cross-platform CI failures raised in the demo.
  • Polish the "how to add a procedure" doc based on questions Group C and Group A asked while wiring auth procedures and Query hooks.
  • Half-day if green.

Hand-offs

Receive from:

  • Group A — the scaffold itself (Tue W1).
  • Group D — a real native module to test rebuild against (Thu W2).
  • Namkha — signing credentials, and decision on the tRPC bridge approach (by EOD Thu W1 demo, latest Fri W1 morning).

Ship to:

  • Group A — CI pipeline including test step (Tue W1), tRPC bridge (Tue W2).
  • Group C — tRPC bridge they can hang auth procedures on (Tue W2).
  • Group D — electron-rebuild recipe (Wed W1), Windows build path (Wed W2).

Definition of done

  • CI pipeline green on macOS and Windows; macOS produces a signed, notarized build artifact; Windows produces an unsigned build artifact.
  • electron-rebuild step works for at least one real native module.
  • tRPC-over-IPC bridge ships at least two procedures end-to-end (capture.start, capture.status) with full type-safety from main to renderer.
  • Preload script implements context isolation correctly; no direct ipcRenderer exposure.
  • Decision doc on the bridge approach exists and is merged.
  • How-to-add-a-procedure doc is in the repo.
  • Local dev round-trip (clone → run) under two minutes.

Pitfalls

  • Notarization on day one. Notarization needs Apple's servers and quirky credential handling. Get signing working first.
  • Hand-rolled IPC. If you pick hand-rolled, you take on ownership of the bridge forever. Pick it only if electron-trpc has a real blocker (streaming subscription bugs, abandonware, etc.).
  • Exposing ipcRenderer in preload. A common shortcut. Never. The preload exposes the typed tRPC client and nothing else.
  • Skipping context isolation. If the renderer can touch Node APIs directly, we have a security regression vs v1. Read Electron's security docs and the preload section twice.
  • Forgetting electron-rebuild for native modules. Without it, native modules built for Node will fail to load in Electron with an opaque error. Group D will lose half a day.
  • Letting CI go red and merging anyway. Never.
  • Caching native module builds incorrectly in CI. Cache hits with the wrong Electron version cause the most confusing build failures in this stack.

Self-check

  1. What is context isolation and what breaks if it is off?
  2. What is the difference between contextBridge.exposeInMainWorld and a normal global on window?
  3. Why do we need electron-rebuild instead of just running pnpm install?
  4. What is the latency of one tRPC procedure call on your machine, in milliseconds, and where does the time go?
  5. What happens to a signed build if the user's clock is wrong?
  6. If notarization fails with an opaque error, where do you look first?

6 · Group C — Auth & Secure Storage

Mission

Port the auth and auto-update code from v1 to TypeScript verbatim, preserving the sleep/wake handling and encrypted token storage. By end of Week 2, signin via magic-link works end-to-end: user types email, clicks link in mailbox, app is authed, token survives restart and a laptop sleep cycle.

Skill profile

  • Detail-oriented — the kind of person who reads error messages before asking for help.
  • Comfortable with OAuth flows: knows what an authorization code is, what a refresh token is, why we need both.
  • Security mindset — instinctively asks "what happens if this token leaks?" and "what does the encryption-at-rest story look like?"
  • Patient with edge cases: laptops sleep, tokens expire at inconvenient moments, mail clients mangle URLs, users open the magic link on a different device.
  • Doesn't need design chops, but should be able to embed views in Group A's app shell using their shadcn primitives without re-inventing styles.

Required reading (finish before Monday of W1)

  • docs/v2-stack/electron.html — focus on main-process, session, and protocol handler sections.
  • docs/v2-stack/electron-builder.html — focus on the auto-updater section.
  • docs/v2-stack/typescript.html — skim.
  • docs/v2-stack/react.html — skim if new; you'll write small renderer views in Week 2.
  • docs/v2-stack/react-hook-form-zod.html — read all of it. You will use both for the magic-link form.
  • v1 source — read these in detail, you are porting them:
    • src/services/auto-updater.js
    • src/core/auto-updater-flow.js
    • src/services/backend-auth.js
    • src/core/oauth-flow.js
    • src/core/token-refresh.js
  • docs/14-security-and-compliance.html — read the section on token storage and the threat model. Your work touches this.

Week 1 — day by day

Monday — Read v1 auto-updater code end-to-end.

  • Structured logging and basic metrics surfaced on the engine instance.

Pitfalls

  • Skipping the spec doc. If you skip Monday/Tuesday's write-up, you will rebuild the engine in Week 4 because the contract isn't clear. Write it.
  • Leaving the uiohook listener attached after stop. The most common bug in this kind of code. The OS keeps reporting events; your engine fires "stopped" events. Always pair attach/detach.
  • Dedup based on wall-clock alone. Use the trigger identity plus payload hash plus a time window. Wall-clock alone misses cases where two near-simultaneous distinct events should both fire.
  • Idle detection that races with the event loop. Use a monotonic clock and a single source of truth (one last-active timestamp), not multiple timers that can drift.
  • Treating clipboard polling as free. A 100ms poll loop is expensive over 8 hours. Use the OS event hook where available, poll with backoff otherwise, and document the decision.
  • Hand-rolling a typed EventEmitter wrapper. The overhead is negligible, but the API drift from Node's emitter causes confusion. Use a small, well-known typed-emitter package or keep it plain with documented event names.
  • Forgetting the Windows path until Week 7. You don't need to test on Windows this phase, but write your code so porting is mechanical, not architectural. Group B opens the Windows CI in Week 2 — use it.

10 · When something is wrong

The right thing to do, in order:

  1. Search. Search the repo and the v1 source for similar patterns. Most of the answer is already written down.
  2. Read the relevant docs/v2-stack/ page. The reference pages exist for a reason. Read the whole page, not just the section you think is relevant.
  3. Ask your group. Post in the group's Discord channel with the question and what you've tried. Wait 15 minutes for a reply while you continue trying.
  4. Ask Namkha. If you're still stuck after an hour, ping Namkha directly. Include: what you're trying to do, what you tried, what failed, what you think the cause is.
  5. Don't suffer silently past lunch. If you spent the morning stuck on the same thing, raise it at standup. The team would rather you ask early than miss a hand-off.

The wrong thing to do, in order:

  1. Reinventing a piece of infrastructure another group owns.
  2. Adding a dependency without writing a one-line rationale in the PR.
  3. Merging a PR with a failing CI step that you believe is "flaky".
  4. Deleting code you don't understand because it looks unused.
  5. Working past 19:00 on a Friday. Foundation is two weeks; pace it.
