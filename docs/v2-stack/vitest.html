Vitest

Unit + integration tests. Jest-compatible API, Vite-native, fast. Already in v1's package.json and the backend.

Why we use this

Vitest is what you reach for when you want a test runner that works with Vite without configuration. The API is Jest-compatible (describe, it, expect), it understands ESM, TS, and JSX out of the box, and it's noticeably faster than Jest on real projects.

For v2: unit tests on every service in the desktop main process, every Express controller, every Zod schema validator. Integration tests on the tRPC router. Component tests on React components that have non-trivial logic. We don't aim for 100% coverage; we aim for "the critical paths are tested and the test suite runs in <30s on CI."

What you need to know

  • describe, it, test, expect — the basics
  • Matchers — toBe, toEqual, toMatchObject, toThrow, resolves/rejects
  • Setup / teardown — beforeAll, beforeEach, afterEach, afterAll
  • Mocking — vi.fn(), vi.mock(), vi.spyOn(), automatic vs manual
  • Timers — vi.useFakeTimers(), vi.advanceTimersByTime() — useful for testing the activity buffer's timers
  • Async tests — just async functions, plus resolves/rejects
  • Snapshot tests — toMatchSnapshot(), inline snapshots
  • Component testing with Testing Library — render, screen, userEvent
  • Coverage — --coverage, V8 or istanbul
  • Watch mode — vitest (no args) → re-runs on file change
  • UI mode — vitest --ui opens a browser dashboard

Patterns you'll see in v2

A service unit test

// src/main/services/ocr/ocr-service.test.ts
import { describe, it, expect, vi } from 'vitest';
import { OcrService } from './ocr-service';

describe('OcrService', () => {
  it('caches results by image hash', async () => {
    const service = new OcrService();
    const fakeBuffer = Buffer.from('test');

    const extract = vi.spyOn(service as any, 'extractTextNative')
      .mockResolvedValue('hello world');

    const first = await service.extractText(fakeBuffer);
    const second = await service.extractText(fakeBuffer);

    expect(first).toBe('hello world');
    expect(second).toBe('hello world');
    expect(extract).toHaveBeenCalledTimes(1); // second was a cache hit
  });
});

Testing time-based logic with fake timers

it('flushes the activity buffer every 30 seconds', async () => {
  vi.useFakeTimers();
  const send = vi.fn();
  const buf = new ActivityBuffer({ sendCallback: send, sendIntervalMs: 30_000 });

  buf.start('session-1', 'user-1');
  buf.addFrame(makeFrame());
  buf.addFrame(makeFrame());

  await vi.advanceTimersByTimeAsync(30_000);
  expect(send).toHaveBeenCalledOnce();
  expect(send.mock.calls[0][0].frames).toHaveLength(2);

  vi.useRealTimers();
});

Component test with Testing Library

import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { Button } from './Button';

it('fires onClick when clicked', async () => {
  const onClick = vi.fn();
  render(<Button onClick={onClick}>Save</Button>);
  await userEvent.click(screen.getByRole('button', { name: 'Save' }));
  expect(onClick).toHaveBeenCalledOnce();
});

Integration test for an Express route

import request from 'supertest';
import { app } from './app';

it('rejects login with bad password', async () => {
  const res = await request(app)
    .post('/api/auth/login')
    .send({ email: 'a@b.com', password: 'wrong' });
  expect(res.status).toBe(401);
});

Hands-on: build this

Add Vitest to your Express auth lab:

  1. Install vitest + supertest + their type packages.
  2. Write 5 tests: register (happy path), register (duplicate email), login (success), login (wrong password, rate-limited), GET /me with valid + invalid token.
  3. Mock the in-memory user store using vi.mock.
  4. Add fake timers for the rate-limiter window test.
  5. Configure --coverage and aim for >80% on the auth slice.
  6. Wire it into a GitHub Action: on every PR, run pnpm test + pnpm tsc --noEmit.

How this fits OnlyWorks v2

v2 has Vitest in three places:

  • Backend — tests every service and controller path
  • Desktop main process — tests services in isolation (mock the Electron APIs, mock native modules)
  • Desktop renderer — component tests via Testing Library; not every component, but the ones with non-trivial logic

CI runs the whole suite on every PR. We don't merge with failing tests. End-to-end / UX tests live in Playwright — next chapter.
