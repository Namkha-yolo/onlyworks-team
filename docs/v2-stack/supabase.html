Supabase

Postgres + auth + storage + edge functions, as a hosted service. Our database, our user table, our row-level-security policies, our file storage.

Why we use this

Supabase is Postgres with batteries: an auth service that handles OAuth + email magic links + password reset, row-level-security policies enforced by the database (so even a misbehaving service role can't leak data), storage with signed URLs, edge functions for Deno-based serverless, and a TypeScript client that handles JWT signing/refresh.

For OnlyWorks, this means we don't host or operate a Postgres ourselves, don't write our own auth, don't write our own file-upload signed-URL logic. The trade-off is vendor lock-in — we use the auth schema, the storage API, the RLS model. So far the speed gain has been worth it.

Tables that matter: web_users, profiles, screenshot_sessions, activity_log, generated_reports, training_frames, teams, team_members, team_invitations, hiring_managers, job_posts, job_matches, verified_skills, ow_profile_cache. See CLAUDE/supabase_schema.sql in the desktop repo.

  • The two clients — anon (RLS enforced) vs service role (bypasses RLS, server-only)
  • Row Level Security (RLS) — policies are SQL predicates that run per-row per-query
  • auth.uid() in policies — the current user's UUID
  • Migrations — numbered SQL files in migrations/; how to apply via Supabase CLI or dashboard
  • The JS client — createClient, queries with .from().select().eq()
  • Auth helpers for Next.js — @supabase/auth-helpers-nextjs, server vs browser clients
  • Storage — buckets, public vs private, signed URLs
  • Edge functions — Deno-based, useful for webhooks and lightweight serverless
  • Realtime — Postgres change-feeds via websockets

Patterns you'll see in v2

Two clients (admin vs anon)

// On the backend (Node)
import { createClient } from '@supabase/supabase-js';

export const supabase = createClient(SUPABASE_URL, SUPABASE_ANON_KEY);

// Admin client — used by repositories that need to bypass RLS.
// NEVER expose the service role key to client code.
export const supabaseAdmin = createClient(SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY, {
  auth: { autoRefreshToken: false, persistSession: false },
});

An RLS policy (SQL)

-- Only let users read their own sessions.
create policy "Users see their own sessions"
on screenshot_sessions for select
using (auth.uid() = user_id);

-- Allow insert only if the user_id matches the authenticated user.
create policy "Users create their own sessions"
on screenshot_sessions for insert
with check (auth.uid() = user_id);

A repository (TypeScript)

export class ReportsRepository {
  constructor(private readonly db = supabaseAdmin) {}

  async listByUser(userId: string) {
    const { data, error } = await this.db
      .from('generated_reports')
      .select('id, title, created_at, report_type')
      .eq('user_id', userId)
      .order('created_at', { ascending: false })
      .limit(50);
    if (error) throw error;
    return data;
  }
}

Hands-on: build this

Convert your Express auth lab to use Supabase:

  1. Create a free Supabase project. Note the URL, anon key, service role key.
  2. Create a profiles table: id uuid PK, email text unique, full_name text, created_at timestamptz default now().
  3. Replace the in-memory Map from the Express lab with Supabase calls.
  4. Add RLS: only the row's owner can select/update their own profile.
  5. Write a small Node script that uses the anon client to fetch /profiles after signing in. Confirm RLS prevents it from seeing other users' rows.
  6. Write the same script using the service role client. Confirm it sees everything (and decide where you can/can't use this client).
  7. Add a Supabase Storage bucket for avatars. Write a route that returns a signed upload URL for the current user.

How this fits OnlyWorks v2

Supabase is unchanged from v1. The same project, the same tables, the same migration numbering. Read CLAUDE/supabase_schema.sql in the desktop repo for the full schema reference.

One v1 footnote that affects v2: the Teams repository uses supabaseAdmin to bypass an RLS infinite-recursion bug. Authorization is enforced in the Express TeamController::requireMember middleware, not in RLS. Every team endpoint must go through that middleware. Don't recreate the RLS bug.

The talent-community magic-link flow, the OW Profile cache, the durable verified_skills table (added after the April 14 postmortem) — these all stay. v2 just adds new columns / migrations as needed for the unified capture pipeline and the proprietary model dataset.
