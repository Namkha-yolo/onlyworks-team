Fine-tuning with LoRA

Adapt a 7B+ model on a single consumer GPU. LoRA + QLoRA + PEFT. How we actually customize a base VLM to OnlyWorks's task without renting a cluster.

Why LoRA

Full fine-tuning of a 7B model means: update 7 billion parameters. That requires ~140 GB of GPU memory for the weights + gradients + optimizer state, and the resulting checkpoint is 14 GB. You'd need 4–8 H100s per training run; checkpoint storage adds up; serving each fine-tune requires loading 14 GB into VRAM.

LoRA (Low-Rank Adaptation) freezes the base model and learns small low-rank matrices — typically <1% of the parameter count — that "patch" the model's behavior. You can fine-tune a 7B model on a single 24 GB GPU. The trained "LoRA" is ~50 MB; you load the base once and swap LoRAs at inference time.

QLoRA goes further: quantize the frozen base to 4-bit. You can fine-tune a 7B model on a single 12 GB GPU. Standard answer for "I have one GPU and want to fine-tune."

For OnlyWorks: LoRA is the right default. We're not training from scratch; we're adapting an existing strong VLM to be better at "interpret a screenshot of work activity." A few hundred MB of adapter weights is enough.

What you need to know

LoRA fundamentals

  • Rank (r) — the bottleneck of the low-rank decomposition. Lower r = fewer params = less capacity. Common: 8, 16, 32, 64.
  • Alpha — a scaling factor. Usually 2x the rank.
  • Target modules — which linear layers to attach LoRAs to. Usually q_proj, k_proj, v_proj, o_proj, sometimes the MLP layers too.
  • Dropout — LoRA-specific dropout to prevent overfitting.
  • Merging — after training, optionally merge the LoRA into the base for a standalone model.

QLoRA additions

  • 4-bit quantization via bitsandbytes (load_in_4bit=True)
  • NF4 — the quantization data type used in QLoRA
  • Double quantization — quantize the quantization constants too, for more memory savings
  • Paged optimizers — offload optimizer state to CPU memory

Hyperparameters that matter

  • Learning rate — LoRA tolerates higher LR than full fine-tuning. 1e-4 to 5e-4 is normal.
  • Batch size + gradient accumulation — effective batch = per-device batch × accum × num_devices
  • Epochs vs steps — small datasets: a few epochs. Big datasets: think in steps + eval-based stopping.
  • Warmup & LR schedule — linear warmup, cosine decay is the standard combination.
  • Sequence length — trade-off between context window and memory.

Specific to VLMs

  • Whether to train the vision encoder — usually freeze it.
  • Image resolution — higher = more visual tokens = more memory.
  • Visual instruction format — the chat template matters; mismatch silently hurts.

Patterns you'll see in v2

LoRA setup with PEFT

from peft import LoraConfig, get_peft_model, TaskType
from transformers import AutoModelForCausalLM

base = AutoModelForCausalLM.from_pretrained(
    'Qwen/Qwen2.5-7B',
    torch_dtype='auto', device_map='auto',
)

lora_config = LoraConfig(
    r=16,
    lora_alpha=32,
    target_modules=['q_proj', 'k_proj', 'v_proj', 'o_proj', 'gate_proj', 'up_proj', 'down_proj'],
    lora_dropout=0.05,
    bias='none',
    task_type=TaskType.CAUSAL_LM,
)

model = get_peft_model(base, lora_config)
model.print_trainable_parameters()
# trainable params: 41,943,040 || all params: 7,657,943,040 || trainable%: 0.55%

QLoRA on a 12GB GPU

from transformers import BitsAndBytesConfig

bnb = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type='nf4',
    bnb_4bit_compute_dtype='bfloat16',
    bnb_4bit_use_double_quant=True,
)

base = AutoModelForCausalLM.from_pretrained(
    'Qwen/Qwen2.5-7B',
    quantization_config=bnb,
    device_map='auto',
)

# Then attach LoRA as above.

Saving and loading just the adapter

# after trainer.train()
model.save_pretrained('./ow-activity-interpreter-v0.1-lora')

from peft import PeftModel
base = AutoModelForCausalLM.from_pretrained('Qwen/Qwen2.5-7B', torch_dtype='auto', device_map='auto')
model = PeftModel.from_pretrained(base, './ow-activity-interpreter-v0.1-lora')

Merging the adapter for deployment

merged = model.merge_and_unload()
merged.save_pretrained('./ow-activity-interpreter-v0.1-merged')
# now a standalone model, ready for vLLM / TGI deployment

Hands-on: build this

Run a small QLoRA fine-tune you can actually finish in a few hours:

  1. Rent a single L4 (24 GB) on RunPod (~$0.50/hr) — ask first. Or use Colab Pro.
  2. Pick a small base: Qwen/Qwen2.5-3B-Instruct or meta-llama/Llama-3.2-3B.
  3. Pick a public instruction dataset: tatsu-lab/alpaca (~52K instructions).
  4. Run a QLoRA fine-tune with r=16, 1 epoch. Use the Trainer.
  5. Confirm the adapter saves as ~50 MB. Push it to the Hub privately.
  6. Run inference: same base + your adapter on three custom instructions. Compare to base-only output.
  7. Merge the adapter. Confirm the merged model gives the same output.
  8. Log everything to Weights & Biases (next-but-one chapter).

How this fits OnlyWorks v2

The OnlyWorks fine-tuning loop:

  1. Build the dataset — see dataset prep. Pairs of (sequence of screenshots + activity metadata) and (target narrative summary).
  2. Pick a base — start with Qwen2.5-VL-7B. Drop down to 3B if 7B is too expensive to iterate on.
  3. QLoRA fine-tune — on rented GPUs (Modal / RunPod / Lambda). Log metrics to W&B.
  4. Eval — compare outputs to Gemini Flash on a held-out set. Use both automatic metrics (ROUGE, BLEU) and human review for the first few iterations.
  5. Iterate — tweak data, hyperparams, base. The first 3 rounds will be bad.
  6. Deploy — when eval looks plausibly good, merge the adapter and deploy via vLLM behind the existing /api/pipeline/activity/stream endpoint. A/B against Gemini on real traffic.

None of this is short. Plan for 6–12 months from "first fine-tune" to "shipped to a meaningful fraction of users." But every step is doable on the budget we have.
