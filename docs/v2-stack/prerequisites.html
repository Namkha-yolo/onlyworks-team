Prerequisites

What every track assumes you already know. If any of this is unfamiliar, spend a few days here before moving on. Each item is short — none of it should take more than a weekend.

Why this page exists

The rest of these docs jump straight into TypeScript, React, Electron, AI. They assume you can clone a repo, run a Node script, read MDN when something is unfamiliar, and not get lost when someone says "make a PR against main." If any of that is fuzzy, this page is the cheapest way to fix it.

What you need

  • A working dev machine — macOS, Windows, or Linux. macOS is easiest for the desktop track because of the Vision OCR work; Windows is required for the Windows port.
  • Git installed, a GitHub account, SSH key configured.
  • Node.js 20+ via nvm — don't install Node from the website, you'll regret it.
  • pnpm globally installed (npm install -g pnpm). v2 uses pnpm.
  • An editor — Cursor or VS Code. The team uses both.
  • A terminal you're comfortable with — iTerm2 on macOS, Windows Terminal on Windows.
Spend the first hour

Getting your dev environment right. A messy environment will leak into everything you do for the rest of the internship. Ask for help if you get stuck for more than 30 minutes — this is the one task where asking early is correct.

Topics to cover

Modern JavaScript (ES2020+)

Most of the desktop and web codebases are JavaScript / TypeScript. You need to be fluent in:

  • let / const, arrow functions, destructuring, spread/rest
  • Promises, async / await, error handling with try/catch
  • Modules: import / export (ESM), and how CommonJS (require) differs
  • Array methods: map, filter, reduce, find, some, every
  • Object shorthand, computed keys, optional chaining (?.), nullish coalescing (??)
  • Template literals, tagged templates

Git & GitHub flow

  • clone, branch, checkout, add, commit, push, pull, merge, rebase
  • How to resolve a merge conflict without panicking
  • How to write a good commit message (one-line summary + body explaining why)
  • Pull request flow: branch → push → open PR → request review → address comments → merge
  • git log, git blame, git diff — read history, never just rewrite it

The command line

  • Navigate: cd, ls, pwd, mkdir, rm, cp, mv
  • Pipes & redirects: |, >, >>, <
  • Background & foreground jobs (&, Ctrl+Z, jobs, fg)
  • Process management: ps, kill, lsof
  • Reading man pages

Node.js basics

  • package.json: dependencies, devDependencies, scripts
  • node_modules, lockfiles (pnpm-lock.yaml) and why you commit them
  • CommonJS vs ESM — you'll meet both
  • Running scripts via pnpm run <script>

HTTP & the browser

  • HTTP methods, status codes, headers, query strings, request body
  • fetch API, JSON serialization, CORS at a high level
  • Browser dev tools: Network, Console, Elements, Sources, Application (cookies/localStorage)
  • What a JWT looks like and why we never put one in localStorage when we can avoid it

Official docs to bookmark

  • Node.js — Getting Started
  • pnpm docs

Hands-on: build this

Before you move on to TypeScript, build and ship a tiny CLI tool. It should:

  1. Live in its own GitHub repo under your account.
  2. Be a Node.js script — no framework, no bundler, just node index.js.
  3. Take a GitHub username as a command-line argument.
  4. Use fetch to call https://api.github.com/users/<username>/repos.
  5. Print out the user's top 5 repos by stars, formatted nicely.
  6. Have a real README.md that explains what it does and how to run it.
  7. Be installed cleanly: package.json with a "start" script, node_modules in .gitignore.

Open a PR against your own repo from a feature branch and merge it. Then share the link. If you can do this without Googling more than 3 things, you're ready for TypeScript.

How this fits OnlyWorks

Everything in v2 builds on these basics. The desktop app is a Node process (main) + a browser process (renderer); the backend is a Node + Express server; the website is a Node + Next.js server. The model training stack is the only one in Python — and even there, the data pipeline that produces the training set runs in Node.

You will read git history daily. You will write at least one PR per day. You will live in the terminal. If you can't move comfortably between those three, the rest of the stack will feel twice as hard.
