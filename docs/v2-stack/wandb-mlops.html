Weights & Biases + MLOps

Experiment tracking, model registry, eval pipelines. The discipline layer that makes "we did 47 training runs last quarter" useful instead of a mess.

Why we use this

If you do 50 training runs and don't track them, you have 50 unlabeled checkpoints and no way to remember which one was best. Weights & Biases (W&B) records every metric, hyperparameter, system stat, dataset version, and code revision per run, then lets you compare them side-by-side in a web UI.

It's not the only choice (MLflow, Neptune, Aim, ClearML), but it's the most polished and has the best free tier for small teams. wandb.init() at the top of a training script and a wandb.log() in the loop is the minimum viable setup.

Beyond W&B, "MLOps" covers: model registry (where do checkpoints live), eval pipelines (how do we measure quality), deployment + monitoring (how do we know production is healthy). For OnlyWorks's scale, we keep this minimal but consistent.

What you need to know

W&B basics

  • wandb.init — project, run name, config, tags
  • wandb.log — metrics, step counter, images, tables, histograms
  • Config — wandb.config for hyperparams; sweeps for hp search
  • Artifacts — versioned datasets + models + reports
  • Reports — shareable write-ups with embedded run charts
  • Sweeps — grid / random / Bayesian hyperparam search
  • Integration with HF Trainer — report_to='wandb' in TrainingArguments

Beyond W&B (full MLOps loop)

  • Model registry — Hugging Face Hub private repo is enough for us
  • Eval suite — a script that loads a model + a held-out dataset and returns a fixed set of metrics
  • Eval comparison harness — "is v0.2 better than v0.1 on the same data?"
  • A/B testing in production — flag a fraction of users to call the new model. Compare outcomes.
  • Drift detection — does production data look different from training data over time?
  • Inference monitoring — latency, error rates, output length, refusal rates
  • Reproducibility — pin model + dataset + code commit per run, so you can rebuild any historical experiment

Minimal W&B in a training script

import wandb

wandb.init(
    project='onlyworks-vlm',
    name='qwen2_5-vl-7b-lora-v0_1',
    config={
        'base_model': 'Qwen/Qwen2.5-VL-7B-Instruct',
        'dataset_version': 'ow-activity-v0.1',
        'lora_rank': 16,
        'learning_rate': 2e-4,
        'batch_size': 4,
        'gradient_accumulation_steps': 8,
    },
    tags=['vlm', 'lora', 'qwen'],
)

# ... training loop ...
for step, batch in enumerate(loader):
    loss = train_step(batch)
    wandb.log({'train/loss': loss.item(), 'train/lr': optimizer.param_groups[0]['lr']}, step=step)

    eval_steps=200,

Hands-on: build this

Track your MNIST experiment from the PyTorch chapter on W&B:

  1. Sign up for a free W&B account. pip install wandb; wandb login.
  2. Add wandb.init(project='ow-warmup', name='mnist-cnn-baseline') at the top of your MNIST script.
  3. Log train loss + test accuracy per epoch with wandb.log.
  4. Open the run in the web UI. Confirm the metrics show up.
  5. Run the script 5 times with different learning rates (1e-3, 1e-4, 1e-5, 3e-4, 5e-4). Tag the runs lr-sweep.
  6. In the W&B UI, group by lr, compare the runs side-by-side. Determine which LR is best.
  7. Bonus: define a Sweep (random search across LR + dropout) and run 20 trials. Look at the parallel-coordinates plot.
  8. Write a W&B Report with the findings. Share the URL.

Goal: feel why "track every run" pays back. After this, you'll never train without it.

House rules for experiments

  • Every training run gets a W&B entry. If you forgot to wandb.init, the run effectively didn't happen.
  • Run names follow a convention — <base>-<variant>-<date>. e.g. qwen2_5vl-7b-lora-2026-06-08.
  • Tag liberally — experiment type, dataset version, infrastructure provider. Lets you slice runs later.
  • Log the full config — including dataset hash + git commit. Reproducibility starts here.
  • Push the resulting checkpoint to the HF Hub — private repo, linked in the W&B run's notes.
  • One eval suite, run on every model — the same script, same data. Cross-run comparison only works if eval is fixed.
  • If a run is a flop, write down why — one sentence in the run's notes. You'll thank past-you in 6 months.

How this fits OnlyWorks v2

W&B is the journal of the model-training program. Every experiment we run, every eval, every dataset version — all in one project. By the time we have to choose "should we ship v0.3 to production?", the answer is one chart away: v0.3 vs v0.2 on the eval set, plotted across run dates.

Long-term, the MLOps loop closes with:

  • Eval pipeline — a script run after every training that scores the model on a fixed test set. The numbers go to W&B.
  • Deployment — the chosen checkpoint is pushed to a vLLM / TGI endpoint behind the existing /api/pipeline/activity/stream route.
  • Production monitoring — per-call latency, output length, refusal rate, and (eventually) a small sample of human-reviewed outputs. Drift in any of those triggers a re-eval.
  • Continuous learning — quarterly retrain with the latest 3 months of consented data, full re-eval, A/B test, ship if better.

This is the full loop the v2-roadmap deferred ("only after approaches 1–3 are exhausted"). When the team is ready to make the proprietary model real — this is the infrastructure layer that makes it manageable instead of chaos.
