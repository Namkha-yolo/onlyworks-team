React 18

The UI framework for the desktop renderer and the website. Boring, default, hireable — and a complete replacement for the innerHTML + window.* + tab-rebuilding pattern in v1.

Why we use this

The current desktop renderer rebuilds #main-content from a string of HTML every time you switch tabs. Any modal that was open, any text the user was typing, any scroll position — all gone. Every onclick="…" handler has to be re-attached to window.*. Every interpolated value has to be sanitized by hand or it's an XSS.

React replaces all of this. Components are composable. State persists across re-renders. Event handlers are real function references, not strings. onClick={fn} doesn't need window.fn to exist. JSX escapes by default, so you can't accidentally inject HTML.

We pick React over Solid / Svelte because interns will have written more React. The hireability and Stack Overflow surface area are unmatched.

What you need to know

Core concepts

  • JSX — what it compiles to, why expressions go in {}, why className not class
  • Components — function components only; class components are legacy
  • Props — passing data down, typed with TS interfaces
  • Children — children: React.ReactNode as the standard typed pattern
  • Conditional rendering — {cond && <X />}, ternaries, early returns
  • Lists & keys — and why "use the index as key" is usually wrong
  • Controlled vs uncontrolled inputs

Hooks

  • useState — local state, the lazy initializer form
  • useEffect — when to use it (almost never for data fetching — use TanStack Query), cleanup functions, dependency arrays, the StrictMode double-call
  • useRef — DOM refs, mutable values that don't trigger renders
  • useMemo / useCallback — when they help, when they're noise
  • useContext — and why context is for theme/auth/locale, not for state management
  • useReducer — when state transitions get complex
  • Custom hooks — extracting reusable logic, naming convention

Patterns & mental model

  • Lifting state up vs. pushing it down
  • Composition over inheritance; render props vs. children-as-function
  • How React knows what to re-render (reconciliation, keys, memo)
  • Error boundaries
  • The new Concurrent / Suspense model: <Suspense>, useTransition, useDeferredValue
  • What server components are (mainly for Next.js; the desktop is all client components)

Patterns you'll see in v2

Typed props

interface ButtonProps {
  variant?: 'primary' | 'secondary' | 'ghost' | 'danger';
  size?: 'sm' | 'md' | 'lg';
  onClick?: () => void;
  children: React.ReactNode;
  disabled?: boolean;
}

export function Button({
  variant = 'primary', size = 'md', onClick, children, disabled,
}: ButtonProps) {
  return (
    <button
      className={cn('ow-btn', `ow-btn-${variant}`, `ow-btn-${size}`)}
      onClick={onClick}
      disabled={disabled}
    >
      {children}
    </button>
  );
}

Custom hook for the session

function useSession() {
  return useQuery({
    queryKey: ['session', 'current'],
    queryFn: () => trpc.session.current.query(),
  });
}

// in a component:
function SessionPanel() {
  const { data: session, isLoading } = useSession();
  if (isLoading) return <Spinner />;
  if (!session) return <StartButton />;
  return <ActiveSession session={session} />;
}

Don't use useEffect for data

90% of useEffect uses in tutorials are for fetching data. In v2 we use TanStack Query for that — it handles caching, refetching, retry, and stale-while-revalidate out of the box. Reach for useEffect only when you're synchronizing with something outside React (a DOM API, a timer, an event listener).

Hands-on: build this

Build a tiny GitHub repo browser as a Vite + React + TypeScript app:

  1. pnpm create vite@latest gh-browser -- --template react-ts
  2. One screen, two states: input a username, see their top 10 repos.
  3. Use useState for the input, useEffect for the fetch (it's OK here — TanStack Query is in the next chapter).
  4. Render the repo list with proper keys.
  5. Add a loading state, an error state, and an empty state.
  6. Each repo card is its own component with typed props.
  7. No CSS framework yet — write plain CSS modules. Tailwind comes next.

Goal: never reach for the docs to remember what useState returns. After this, re-do it using useReducer instead of multiple useState calls and see how the code shape changes.

How this fits OnlyWorks v2

The v2 renderer is one React tree mounted into <div id="root"> in index.html. Pages are real route components managed by TanStack Router. Components like <Button>, <Modal>, <StatCard> port the v1 owButton/owModal/owStatCard factory API with the same prop names — only the implementation changes from HTML strings to JSX.

Read the "Code That Can Be Ported vs Must Be Rewritten" section of 03-architecture-review.html to see exactly which page modules are getting React equivalents.
