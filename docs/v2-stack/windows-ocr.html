Windows OCR

Windows.Media.Ocr, called via a PowerShell wrapper. Built into Windows 10+, free, ~5x faster than tesseract.js. The Windows equivalent of macOS Vision.

Why we use this

Same logic as macOS Vision: tesseract.js works but is slow and middling quality; the OS-provided OCR is fast, free, and built-in. Windows.Media.Ocr ships in Windows 10+ via WinRT.

Unlike macOS where we compile a Swift binary, on Windows we spawn PowerShell with an inline-loaded script. The script imports the WinRT type, runs OCR, prints JSON. No compilation step, no installer flags — the PowerShell script ships as plain text.

Catch: PowerShell execution policy can block loading WinRT types on locked-down corporate machines. We bypass with -ExecutionPolicy Bypass per-call, but if the user is on a managed device that disables WinRT entirely, we fall back to tesseract.js.

The PowerShell wrapper (skeleton)

# windows-ocr.ps1
param([Parameter(Mandatory)][string]$ImagePath)
$ErrorActionPreference = 'Stop'

try {
  Add-Type -AssemblyName System.Runtime.WindowsRuntime
  # ... load WinRT types ...

  $bitmap = [Windows.Graphics.Imaging.SoftwareBitmap, Windows.Graphics.Imaging, ContentType=WindowsRuntime]
  $engine = [Windows.Media.Ocr.OcrEngine, Windows.Media.Ocr, ContentType=WindowsRuntime]::TryCreateFromUserProfileLanguages()

  # ... decode image, run OCR ...

  $output = @{ extractedText = $text; success = $true }
  $output | ConvertTo-Json -Compress
}
catch {
  @{ error = $_.Exception.Message; success = $false } | ConvertTo-Json -Compress
  exit 1
}

Calling from Node

// src/main/services/ocr/windows.ts
import { exec } from 'node:child_process';
import path from 'node:path';

export async function extractTextWindows(imagePath: string): Promise<string> {
  const scriptPath = app.isPackaged
    ? path.join(process.resourcesPath, 'app.asar.unpacked', 'src', 'utils', 'windows-ocr.ps1')
    : path.join(__dirname, '..', '..', 'utils', 'windows-ocr.ps1');

  const cmd = `powershell -NoProfile -ExecutionPolicy Bypass -File "${scriptPath}" "${imagePath}"`;

  return new Promise((resolve, reject) => {
    exec(cmd, { timeout: 10_000, maxBuffer: 1024 * 1024 }, (err, stdout, stderr) => {
      if (err) return reject(new Error(stderr || err.message));
      try {
        const result = JSON.parse(stdout.trim());
        if (result.error) return reject(new Error(result.error));
        resolve(result.extractedText ?? '');
      } catch (e) {
        reject(new Error(`Parse Windows OCR output: ${e}`));
      }
    });
  });
}

The circuit breaker

let consecutiveFailures = 0;
let useWindowsOcr = true;

async function extract(imagePath: string) {
  if (useWindowsOcr) {
    try {
      const text = await extractTextWindows(imagePath);
      consecutiveFailures = 0;
      return text;
    } catch (err) {
      consecutiveFailures++;
      if (consecutiveFailures >= 3) {
        logger.warn('Windows OCR failed 3 times — switching to Tesseract for this session');
        useWindowsOcr = false;
      }
    }
  }
  return await extractTextTesseract(imagePath);
}

Hands-on: build this

Get Windows OCR working on a Windows machine (or VM):

  1. Copy v1's windows-ocr.ps1. Run it from a PowerShell prompt: .\windows-ocr.ps1 .\test.png. Confirm JSON output.
  2. Wire it into your Electron lab project under an ocr.extract tRPC procedure.
  3. Add asarUnpack for the script. pnpm dist:win. Confirm the packaged app still finds the script.
  4. Implement the circuit breaker: after 3 consecutive failures, switch to a stub fallback (just return "fallback used").
  5. Force a failure: temporarily break the script (rename a WinRT type). Confirm 3 errors → fallback engages → recovery: fix the script, restart, normal operation resumes.

How this fits OnlyWorks v2

v2's OcrConsumer on Windows is: Windows.Media.Ocr (via PowerShell) → tesseract.js fallback. The circuit breaker is mandatory: WinRT failures usually indicate a system-level issue (missing runtime, broken policy, Windows version mismatch) that won't self-heal mid-session. Switching to tesseract.js once is safer than thrashing.

The Windows pre-flight diagnostic (src/services/windows-preflight.js in v1) runs at startup and checks: PowerShell version, VC++ runtime presence, native module load. Failures surface in Settings → Diagnostics so the user can see why capture or OCR isn't working. v2 keeps this exactly.

Tip

If you're on macOS and need to test the Windows path, use Parallels Desktop or VMware Fusion with a Windows 11 VM. The Windows OCR APIs require a real Windows install (not Wine).
