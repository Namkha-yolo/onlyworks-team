Electron

The shell. Chromium + Node.js, glued together. Capture, OCR, tray, deep links, auto-update — all the things a browser can't do.

Why we use this

OnlyWorks needs to: capture the screen, monitor keyboard and mouse, persist tokens encrypted on disk, run native OCR binaries, sit in the system tray, handle onlyworks:// deep links, auto-update itself silently, and survive sleep/wake without losing the user's session. None of those are possible from a browser tab.

Electron gives us a Node.js process (the main process) that can do all of that, plus a Chromium window (the renderer) that shows the UI. They talk via IPC, which we'll wrap with tRPC.

Tauri (Rust + native webview) is more elegant for some of this, but the rough edges that matter to OnlyWorks — desktopCapturer, screen-recording permission flow on macOS, the Mica backdrop on Windows, active-win binding, code-signing, auto-updates — are first-class in Electron and DIY in Tauri. We stay on Electron until the maintenance gap closes.

What you need to know

Architecture

  • Main process — one per app. Has full Node.js APIs, can spawn windows, do IPC, touch the filesystem, etc.
  • Renderer process — one per BrowserWindow. Sandboxed Chromium. Has DOM APIs, can't require('fs').
  • Preload script — runs in the renderer's context with limited Node access. The only safe bridge between the two.
  • contextIsolation: true — mandatory. Exposes APIs to renderer via contextBridge.exposeInMainWorld.
  • nodeIntegration: false — mandatory. Renderer should never have raw Node access.

Core APIs

  • app — lifecycle: whenReady, quit, relaunch, single instance lock
  • BrowserWindow — create & manage windows, web preferences, title bar style
  • ipcMain.handle / ipcRenderer.invoke — async IPC. Use this; we'll wrap it in tRPC later.
  • contextBridge — expose typed APIs to window.electronAPI
  • shell — openExternal, openPath
  • Tray — menu bar / system tray icons
  • Menu — application menus, context menus
  • globalShortcut — system-wide hotkeys (we use one for toggle-session)
  • desktopCapturer — screenshots of any screen / window
  • systemPreferences — macOS TCC (Accessibility, Screen Recording) trust checks
  • powerMonitor — sleep/wake/lock events
  • Notification — native OS notifications
  • protocol — custom URL schemes (onlyworks://)

Security

  • contextIsolation + sandboxing — always on
  • Content Security Policy — restrict what the renderer can load
  • Validate IPC inputs — the renderer is untrusted. Use Zod.
  • Don't require in the renderer — ever

Patterns you'll see in v2

The main process entry

// src/main/main.ts
import { app, BrowserWindow } from 'electron';

app.whenReady().then(async () => {
  const win = new BrowserWindow({
    width: 1200, height: 800,
    webPreferences: {
      contextIsolation: true,
      nodeIntegration: false,
      sandbox: false, // we use preload to bridge, sandbox=true is even safer if achievable
      preload: path.join(__dirname, 'preload.js'),
    },
  });
  await win.loadURL('http://localhost:5173'); // Vite dev server
});

The preload (typed bridge)

// src/main/preload.ts
import { contextBridge, ipcRenderer } from 'electron';

contextBridge.exposeInMainWorld('electronAPI', {
  capture: {
    start: (sessionId: string) => ipcRenderer.invoke('capture:start', sessionId),
    stop:  () => ipcRenderer.invoke('capture:stop'),
  },
  // ...
});

In v2 we replace this hand-rolled API with a tRPC client. See tRPC over IPC.

A handler in main

// src/main/ipc/capture.ts
import { ipcMain } from 'electron';
import { z } from 'zod';

ipcMain.handle('capture:start', async (_event, sessionIdRaw) => {
  const sessionId = z.string().uuid().parse(sessionIdRaw); // validate
  await captureService.start(sessionId);
  return { ok: true };
});

Hands-on: build this

Build a tiny Electron + React + TS + Vite app:

  1. Use electron-vite's template (next chapter has the setup).
  2. One window. The renderer shows a "Hello, OnlyWorks" page.
  3. Add an IPC channel system:get-info that returns { platform, version, uptime } from the main process.
  4. Display that info on the page. Make sure the renderer never require()s anything.
  5. Add a tray icon with a context menu: "Show window" and "Quit".
  6. Add a global shortcut Cmd+Shift+H that toggles window visibility.
  7. Try removing contextIsolation from the BrowserWindow config and watch the security warning. Put it back.

How this fits OnlyWorks v2

The v2 main process inherits most of v1's structure: OnlyWorksApp class, the core/ lifecycle split (window-manager, tray-manager, menu-manager, auto-updater, OAuth flow, token refresh, error handlers, pipeline init), the macOS TCC trust-state snapshot, the Windows background-throttling switches, the sleep/wake token refresh.

What changes:

  • The 169-channel hand-mirrored preload disappears — replaced by a tRPC bridge (next docs).
  • The capture service goes from 1,417 lines to a unified CaptureEngine + several Consumer files (~400 + 4×150 lines).
  • Backend HTTP calls that were proxied through IPC become direct fetch calls from the renderer using a stored JWT.

Read src/main.js in the v1 repo (~400 lines) before you write a single line of v2 main code — that file's lifecycle wiring is what you're porting.
