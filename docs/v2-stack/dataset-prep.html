Dataset preparation

"You are what you eat." For models, what you eat is the dataset. The single highest-leverage area for our training program — better data beats bigger models almost every time.

Why this matters more than you think

You can spend $50K on GPUs and pick the best base model and write the most elegant training loop, and if your dataset is noisy, biased, or mislabeled, the result will be worse than a smaller model trained on a clean dataset. Karpathy: "Garbage in, garbage out, even with bigger models."

For OnlyWorks specifically: we already have the raw data. Every user session produces screenshots in training_frames and activity narratives in generated_reports. We don't need to scrape, license, or buy data. But the raw data needs cleaning, labeling, deduplication, filtering, and structuring into training-ready examples. That's where the next 6 months of model-training intern work lives.

What you need to know

Dataset hygiene

  • Deduplication — near-duplicate detection (perceptual hashes, embedding similarity)
  • Quality filtering — reject low-resolution images, blank screenshots, frames with no OCR text
  • Privacy filtering — OUR critical filter. Bank apps, iMessage, Slack DMs, password fields — never train on these. The v1 sensitive-filter.js is the starting point.
  • Class balance — if 80% of frames are "user is typing in VS Code", the model overfits to it. Subsample.
  • Train / val / test splits — split by user, not by frame. Or you'll leak.
  • Splits over time — reserve the last N weeks of data for eval to catch drift.

Labeling

  • Self-supervised first — use Gemini's existing outputs (from generated_reports) as initial labels. Cheap, gets us started.
  • Human review for eval set — eval data needs human labels; the model is graded against them.
  • Labeling tools — Label Studio, Argilla, or just a custom internal admin UI
  • Inter-annotator agreement — if two humans disagree, the task isn't well-defined

Format

  • Conversation format for instruction tuning — messages with role + content (text or image)
  • Storage — JSON Lines (.jsonl), Parquet, WebDataset
  • Image references — store image paths or URIs; load lazily in the Dataset
  • Hugging Face Datasets format — datasets.Dataset.from_json + .cast_column for image fields

Tooling

  • pandas / Polars — cleaning, joining, slicing
  • DuckDB — SQL queries directly on Parquet files; very fast for analysis
  • Pillow / OpenCV — image loading + transforms
  • fastdup / imagehash — image deduplication
  • Hugging Face datasets — the canonical loader for training

Patterns you'll see in v2

Building the dataset from training_frames

import duckdb

# Pull training-eligible frames from Supabase via DuckDB.
df = duckdb.sql("""
  SELECT
    tf.image_path,
    tf.app_name,
    tf.window_title,
    tf.activity_label,
    gr.report_data->>'executive_summary' AS narrative
  FROM training_frames tf
  JOIN screenshot_sessions s ON s.id = tf.session_id
  JOIN generated_reports gr ON gr.session_id = s.id
  WHERE tf.user_consented = true
    AND tf.captured_at > '2026-01-01'
    AND gr.report_data IS NOT NULL
  LIMIT 100000;
""").to_df()

Privacy filter (port from v1)

SENSITIVE_APPS = {'1Password', 'Bitwarden', 'Slack', 'Messages', 'WhatsApp', 'Banking', ...}
SENSITIVE_KEYWORDS = ['password', 'social security', 'credit card', ...]

def is_safe(row) -> bool:
    if row['app_name'] in SENSITIVE_APPS:
        return False
    title = (row['window_title'] or '').lower()
    if any(kw in title for kw in SENSITIVE_KEYWORDS):
        return False
    return True

df = df[df.apply(is_safe, axis=1)]

Image deduplication

from PIL import Image
import imagehash

def phash(image_path) -> str:
    return str(imagehash.phash(Image.open(image_path)))

df['phash'] = df['image_path'].apply(phash)
df = df.drop_duplicates(subset='phash', keep='first')

Convert to HF dataset

from datasets import Dataset, Image as HFImage

records = []
for _, row in df.iterrows():
    records.append({
        'image': row['image_path'],
        'app': row['app_name'],
        'window': row['window_title'],
        'target_summary': row['narrative'],
    })

dataset = Dataset.from_list(records).cast_column('image', HFImage())
dataset = dataset.train_test_split(test_size=0.05, seed=42)
dataset.save_to_disk('./datasets/ow-activity-v0.1')

Split by user (not by frame)

# Critical: don't split a single session's frames between train and test.
# If you do, the model "cheats" via near-duplicate frames.
users = df['user_id'].unique()
np.random.seed(42)
np.random.shuffle(users)
train_users = users[: int(len(users) * 0.9)]
eval_users = users[int(len(users) * 0.9):]

train_df = df[df['user_id'].isin(train_users)]
eval_df = df[df['user_id'].isin(eval_users)]

Hands-on: build this

Build a tiny synthetic OnlyWorks-style dataset to practice the pipeline:

  1. Take 50 screenshots of yourself working in different apps.
  2. For each: hand-write a 1-sentence summary of what's happening.
  3. Store as a JSONL with {image, app, window, summary}.
  4. Run perceptual-hash deduplication. Remove near-duplicates.
  5. Apply a fake "privacy filter" rule (e.g. drop everything from "Slack").
  6. Split 80/10/10 train/val/test by source-app to avoid leakage.
  7. Load it via datasets.Dataset.from_json + cast the image column.
  8. Use this dataset for a small VLM fine-tune (LoRA, 1 epoch) just to wire the pipeline end-to-end.

The goal isn't a useful model. It's that every step from raw screenshots to trainer.train() works on your machine. That's a fundamental capability.

How this fits OnlyWorks v2

The v2 capture pipeline produces a stream of training data per consented user. The dataset team's job is:

  • Run a privacy filter that's stricter than the v1 one (the bar for "OK to send to backend" is lower than "OK to put in a training set")
  • Deduplicate ruthlessly (a 30-minute session has 600 1-Hz frames but probably 50 unique "moments")
  • Label initial frames with Gemini's existing summaries; build a human-reviewed eval set
  • Version datasets (v0.1, v0.2, ...) so model runs are reproducible
  • Track which user/sessions are in which dataset version — for honoring deletion requests later

This is unglamorous, day-job work. It's also where 70% of the eventual model quality comes from. Spend the time.

Consent + deletion

Training-data inclusion is opt-in per user. If a user revokes consent or asks for deletion, their frames must be removable from the dataset and (eventually) their influence on a model retrained without them. This is non-negotiable for the privacy story. Build the data pipeline with deletion in mind from day one.
