Next.js 14

The marketing site, public OW profiles, hiring search, Talent Community signup. App Router, server components, SSR. Already in production at only-works.com.

Why we use this

The website is at /Users/xerneas/onlyworks_verified. It serves three audiences: prospective users (download the app), employers (search verified candidates, post jobs), and Talent Community members (join via magic link). Each of those is a distinct flow with SEO needs, SSR needs, and auth needs that Next.js handles well out of the box.

App Router (Next 13.4+) is the version we use. Server components by default; client components when interactivity demands. Streaming, suspense, layouts — all native. SEO works because pages render on the server.

The site is deployed to Vercel. Backend calls go through BACKEND_URL — some calls happen in server components (no CORS), some in client code via the Next API routes (which proxy + add auth).

What you need to know

App Router fundamentals

  • File-based routing — app/page.tsx, app/about/page.tsx, app/p/[owId]/page.tsx
  • Layouts — app/layout.tsx wraps everything; nested layouts wrap sub-trees
  • loading.tsx, error.tsx, not-found.tsx — auto-mounted around children
  • Route groups — (marketing)/ doesn't add to the URL path
  • Dynamic segments — [id] and [...slug]
  • Metadata API — generateMetadata, OG images, dynamic titles

Server vs client components

  • Default: server — no useState, no useEffect, no event handlers
  • 'use client' at the top of a file to opt in to client behavior
  • Server can fetch data with await fetch(...) directly — no useEffect
  • Server can read cookies/headers via next/headers
  • Server actions — functions marked 'use server' callable from client forms
  • Streaming with <Suspense> — render parts of the page as data arrives

API routes & middleware

  • app/api/.../route.ts — HTTP handlers
  • middleware.ts — runs on the edge before any route, useful for auth gating
  • NextResponse, NextRequest
  • The proxy pattern — site API routes forward to the Express backend, attaching auth

Supabase auth on Next

  • @supabase/auth-helpers-nextjs + @supabase/ssr
  • Server & client supabase clients with shared cookies
  • The backend-token swap — supabase access_token → backend JWT (see app/api/auth/backend-token/route.ts in onlyworks_verified)

Production concerns

  • Caching — fetch's cache option, revalidate, cache: 'no-store'
  • Image optimization — <Image> component, next.config.js domains
  • Environment variables — NEXT_PUBLIC_ prefix for client-visible vars
  • Deployment — Vercel auto-deploys per branch; preview URLs for PRs

Patterns you'll see in v2 (and v1's website today)

Server component fetching backend data

// app/p/[owId]/page.tsx
import { BACKEND_URL } from '@/lib/config';

async function getProfile(owId: string) {
  const res = await fetch(`${BACKEND_URL}/api/profiles/${owId}`, { cache: 'no-store' });
  if (!res.ok) return null;
  const data = await res.json();
  return data.success ? data.data : null;
}

export default async function ProfilePage({ params }: { params: { owId: string } }) {
  const profile = await getProfile(params.owId);
  if (!profile) notFound();
  return <ProfileCard profile={profile} />;
}

API route as a backend proxy

// app/api/talent-community/join/route.ts
export async function POST(request: NextRequest) {
  const body = await request.json();
  // ...validate with Zod...

  const res = await fetch(`${BACKEND_URL}/api/talent-community/join`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(body),
  });

  const data = await res.json();
  return NextResponse.json(data, { status: res.ok ? 200 : res.status });
}

Supabase auth → backend JWT swap

// app/api/auth/backend-token/route.ts
export async function POST() {
  const supabase = createClient();
  const { data: { session } } = await supabase.auth.getSession();
  if (!session) return NextResponse.json({ error: 'No session' }, { status: 401 });

  const res = await fetch(`${BACKEND_URL}/api/auth/website/website-token`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ supabase_access_token: session.access_token }),
  });
  const data = await res.json();
  return NextResponse.json({ token: data.data.token, user: data.data.user });
}

Dynamic metadata

export async function generateMetadata({ params }: { params: { owId: string } }): Promise<Metadata> {
  const profile = await getProfile(params.owId);
  if (!profile) return { title: 'Profile Not Found | OnlyWorks' };
  return {
    title: `${profile.full_name} | ${profile.ow_id} | OnlyWorks`,
    openGraph: { /* ... */ },
  };
}

Hands-on: build this

Build a tiny App Router site that re-creates the public profile flow:

  1. pnpm create next-app@latest verified-lab --typescript --tailwind --app
  2. Create a route at /p/[owId]/page.tsx — a server component that fetches from a public API (use api.github.com/users/:username as a stand-in).
  3. Add generateMetadata with a dynamic title and OG image.
  4. Add loading.tsx and not-found.tsx at the right level.
  5. Create app/api/echo/route.ts — a POST endpoint that returns the JSON body unchanged.
  6. Build a client component that posts to it (use 'use client') — this teaches the server/client boundary.
  7. Set up middleware that adds a custom header to every response.
  8. Deploy to Vercel via GitHub. Confirm previews work.
