06 · State & data

TanStack Query

Server state, properly. Replaces every "fetch in useEffect, cache in useState, invalidate by hand" pattern in the v1 renderer.

Why we use this

The v1 renderer has a hand-rolled cache layer in src/app/modules/data-manager.js — per-domain TTLs, manual invalidation, an event bus for refetches. It's well-intentioned and broken in subtle ways: when you switch tabs, the cache invalidates at a different time than the UI re-renders, and a stale report flashes for half a second before the loading state.

TanStack Query handles all of this. It manages server state (data you don't own — it lives on a backend) separately from UI state. It caches by query key, deduplicates concurrent fetches, retries on failure, refetches on window focus, and gives you stale-while-revalidate for free. You write useQuery({ queryKey: ['reports'], queryFn: fetchReports }) and it Just Works.

Mutations get a typed useMutation that integrates with the cache: after a successful POST, invalidate the affected query keys and the UI re-fetches automatically.

Patterns you'll see in v2

Provider setup

// src/main.tsx
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';
import { ReactQueryDevtools } from '@tanstack/react-query-devtools';

const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 60 * 1000,        // 1 minute
      refetchOnWindowFocus: false, // we're in Electron, the window is always focused
      retry: 2,
    },
  },
});

<QueryClientProvider client={queryClient}>
  <App />
  <ReactQueryDevtools />
</QueryClientProvider>

Custom hooks per resource

// hooks/useReports.ts
export function useReports() {
  return useQuery({
    queryKey: ['reports'],
    queryFn: () => trpc.reports.list.query(),
  });
}

export function useGenerateReport() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: (input: GenerateReportInput) => trpc.reports.generate.mutate(input),
    onSuccess: () => {
      qc.invalidateQueries({ queryKey: ['reports'] });
    },
  });
}

Stale-while-revalidate without thinking

function ReportList() {
  const { data, isLoading, isFetching } = useReports();

  if (isLoading) return <Spinner />;
  return (
    <div>
      {isFetching && <RefreshIndicator />}
      {data?.map(r => <ReportCard key={r.id} report={r} />)}
    </div>
  );
}

The user sees cached data immediately. A subtle refresh indicator says "loading newer data". When it arrives, the list updates. No flash of blank screen.

Hands-on: build this

Refactor your GitHub repo browser (from the React doc) to use TanStack Query:

  1. Install @tanstack/react-query and the devtools.
  2. Wrap the app in QueryClientProvider.
  3. Replace the useEffect-based fetch with useQuery.
  4. Add a "Refresh" button that calls refetch().
  5. Add a search input that adds a new query key per username so cache stays separated.
  6. Add a mutation: a fake "Star this repo" button that POSTs to a dummy endpoint. Invalidate the repo list on success.
  7. Open the DevTools panel and watch the cache update.
  8. Set staleTime: 60_000 and confirm: re-mounting the component within 60s doesn't refetch.

How this fits OnlyWorks v2

Every IPC call from the renderer goes through TanStack Query. We delete data-manager.js entirely. The per-domain TTLs and event bus are replaced by query keys + invalidateQueries. The "auto-refresh current view after writes" pattern becomes one line in a mutation's onSuccess.

Since v2 uses tRPC over IPC, the queryFn for every hook is one-liner: () => trpc.<router>.<procedure>.query(). End-to-end types: the component code knows the exact shape of data without writing a single interface.
