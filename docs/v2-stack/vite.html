Vite

The dev server and bundler. Replaces "37 script tags loaded in strict order" with hot-module reload, tree-shaking, and a single dist/ folder.

Why we use this

The current desktop loads modules via 37 ordered <script> tags in app.html, with a "load order matters" comment. Add a new module out of order and things break silently. There's no tree-shaking, no source maps, no hot reload — every change is a full window refresh.

Vite gives us all of that, for free, with five lines of config. It uses native ESM in dev (so the dev server is instant, regardless of project size), and uses Rollup for production builds. The Electron renderer can use it via electron-vite; the Next.js site already uses Turbopack/webpack internally, but Vite is the standalone choice for the desktop renderer.

What you need to know

  • Project scaffolding — pnpm create vite@latest, the templates (react-ts, vanilla-ts)
  • Dev server — vite dev, hot module reload, what triggers a full refresh
  • Build — vite build produces a dist/; vite preview serves it
  • Imports — you can import CSS, JSON, SVG, images directly. Vite handles them.
  • Environment variables — import.meta.env, the VITE_ prefix rule, .env files
  • Path aliases — resolve.alias in config, matched with paths in tsconfig.json
  • Plugins — @vitejs/plugin-react, vite-plugin-svgr, vite-plugin-electron
  • Code splitting & dynamic imports — lazy load routes, components
  • Asset handling — public/ vs imports, base URL config

Patterns you'll see in v2

Minimal config for the desktop renderer

// vite.config.ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import path from 'node:path';

export default defineConfig({
  plugins: [react()],
  resolve: {
    alias: {
      '@': path.resolve(__dirname, './src'),
      '@shared': path.resolve(__dirname, '../shared'),
    },
  },
  build: {
    outDir: 'dist/renderer',
    sourcemap: true,
  },
});

Env vars at runtime

// .env.local
VITE_BACKEND_URL=http://localhost:8080

// src/config.ts
export const BACKEND_URL = import.meta.env.VITE_BACKEND_URL;

VITE_-prefixed vars are exposed to client code; non-prefixed ones aren't. Never put secrets in a VITE_ var — they end up in the shipped bundle.

Dynamic imports for route splitting

const Dashboard = lazy(() => import('./pages/Dashboard'));
const Settings = lazy(() => import('./pages/Settings'));

<Suspense fallback={<Spinner />}>
  <Routes>
    <Route path="/" element={<Dashboard />} />
    <Route path="/settings" element={<Settings />} />
  </Routes>
</Suspense>

Hands-on: build this

Set up a Vite + React + TS project from scratch and add three things on top of the template:

  1. pnpm create vite@latest vite-lab -- --template react-ts
  2. Add path aliases (@/…) in both vite.config.ts and tsconfig.json. Use them in at least one import.
  3. Add an env var VITE_GITHUB_TOKEN with a personal access token, read it via import.meta.env. Confirm it's not a secret leak (search for it in the built dist/ bundle — it will be there, that's the point).
  4. Add a second entry point (e.g. src/popup/main.tsx) and an additional rollupOptions.input so the build produces two HTML files.
  5. Add the React Refresh DevTools and verify hot reload preserves state across edits.

How this fits OnlyWorks v2

v2 uses Vite (via electron-vite) for the renderer. The main process is bundled separately with the same toolchain. The website (Next.js) handles its own bundling — you don't directly touch Vite there.

One concrete win from Vite: when you edit a component, you'll see the change in the open Electron window in under 100ms with state preserved. Compare that to the current dev loop (kill, restart, log back in, reopen the session, navigate to the page) which is ~30s per change.
