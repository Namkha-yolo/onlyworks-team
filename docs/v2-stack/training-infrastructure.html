Training infrastructure

Where to actually run the training. Modal, RunPod, Lambda Labs, Vast.ai. How to rent a GPU without burning thousands of dollars by accident.

Why this is its own doc

You will never train on the founders' laptops. GPUs cost real money per hour, get billed continuously while idle, and have a learning curve specific to each provider. The first time an intern leaves a $4/hr GPU running over a weekend, that's $130 of unwanted billing. We avoid this by being deliberate about which provider for which job and by always knowing where the active session is.

This doc covers: the providers we use, when to pick each one, the price/performance landscape, and the discipline patterns that prevent runaway spend.

The providers we use

Modal Labs Recommended default

  • Model: serverless GPU as a function. You write Python, decorate it with @app.function(gpu='A10G'), call it, billed only while running.
  • Best for: short-to-medium jobs (a few hours to overnight). Notebooks, eval runs, ad-hoc fine-tunes.
  • Pricing: per-second billing. ~$1.10/hr for A10G, ~$4/hr for H100.
  • Idle cost: zero — you only pay while the function is running. This is huge.
  • Catch: cold-start can be 30-60s for big container images. Cache aggressively.

RunPod

  • Model: rent a persistent pod (SSH or Jupyter). Cheaper hardware than Modal.
  • Best for: long training runs, interactive debugging.
  • Pricing: ~$0.30/hr for L4, ~$0.60/hr for A40, ~$2/hr for H100.
  • Idle cost: YOU pay while the pod is allocated, even if idle. Always shut down.
  • Catch: storage charged separately; persistent storage between runs costs.

Lambda Labs

  • Model: similar to RunPod, often slightly cheaper.
  • Best for: same use cases as RunPod.
  • Pricing: similar to RunPod, sometimes cheaper for high-tier (H100/H200).

Vast.ai

  • Model: marketplace — rent from individual providers, including consumer GPUs.
  • Best for: cheapest option for experimentation when you don't need an SLA.
  • Catch: variable reliability, providers can disappear, data integrity is on you.

Google Colab / Kaggle

  • Model: free or cheap notebooks with limited GPU time.
  • Best for: tutorials, prototyping, students. Not for production runs.
  • Catch: sessions can be killed; storage doesn't persist; GPU you get is not guaranteed.

What you need to know

  • GPU types — T4, L4, A10G, A40, A100 (40/80GB), H100 (80GB), H200. Memory + compute trade-offs.
  • Quantization & memory — with QLoRA, a 7B model fits on 12 GB; a 70B model fits on 80 GB. Plan accordingly.
  • Storage — checkpoint sizes, where to push them (HF Hub private vs Supabase Storage vs S3)
  • Setup time — first run on a new pod takes 15–30 min for environment setup. Bake it into a Docker image after that.
  • Spot vs on-demand — cheaper but can be preempted; only use with checkpointing.
  • Multi-GPU — rare for LoRA, common for full fine-tunes. Use accelerate launch.
  • Monitoring — provider dashboards, billing alerts, W&B for training progress.

A Modal training function

import modal

app = modal.App('ow-finetune-v0_1')
image = (
    modal.Image.debian_slim()
    .pip_install('torch', 'transformers', 'datasets', 'peft', 'bitsandbytes', 'accelerate', 'wandb')
)

@app.function(
    image=image,
    gpu='A100-80GB',
    timeout=24 * 60 * 60,        # 24h max
    secrets=[modal.Secret.from_name('huggingface'), modal.Secret.from_name('wandb')],
    volumes={'/data': modal.Volume.from_name('ow-datasets')},
)
def train():
    from transformers import Trainer, TrainingArguments
    # ... load model, dataset, run training ...
    trainer.train()
    trainer.push_to_hub('onlyworks/activity-interpreter-v0.1', private=True)

@app.local_entrypoint()
def main():
    train.remote()

Run with modal run train.py. Cost = exactly the time spent training. No idle bill.

The discipline rule: budget per run

ESTIMATED_HOURS = 8
COST_PER_HOUR = 4.00     # H100 on RunPod
TOTAL_BUDGET = ESTIMATED_HOURS * COST_PER_HOUR
print(f"This run is budgeted for ~${TOTAL_BUDGET:.2f}. Press enter to confirm.")
input()

It sounds silly. It saves real money.

Hands-on: build this

Run a training job on rented infrastructure for the first time. Ask before you spin one up; the team will share an account.

  1. Set up a Modal account, install the CLI, run modal token new.
  2. Build a tiny training function that does 100 steps on MNIST + saves a checkpoint.
  3. Run it on Modal with a T4 GPU (cheapest). Confirm it works.
  4. Same task on RunPod: rent an L4, SSH in, run the script. Compare the experience.
  5. Note the costs of each. Shut down the RunPod pod when done. Verify in the billing dashboard.
  6. Add a "if billing > $5, alert me" cap (Modal has built-in spending alerts).
  7. Document for the team: which provider you'd recommend for your task, and why.

House rules for GPU spend

  • Ask before you spin one up — even cheap ones. Especially the first time. The team has shared accounts; we want to know what's running.
  • Set spending alerts — per-provider, before you start. Daily and monthly caps.
  • Shut down when done — literally check the dashboard before you close your laptop. RunPod / Lambda / Vast will keep billing you forever.
  • Use Modal for short jobs — zero idle cost is the killer feature for prototyping.
  • Checkpoint frequently — especially on spot / preemptible instances
  • Push checkpoints off the pod immediately — to HF Hub or S3. Pod storage is ephemeral.
  • Log every run to W&B — if you can't show what a run produced, the spend was wasted (next doc).

How this fits OnlyWorks v2

The model-training stack runs entirely on rented compute. For 2026, the rhythm is roughly:

  • Experiments & eval — Modal, costs <$100/month
  • Monthly LoRA fine-tune of the production model — RunPod H100, ~$50/run (10–15 hours)
  • Quarterly full retraining / base model evaluation — Lambda or RunPod multi-GPU, low hundreds per run

Total infra budget: a few hundred dollars a month, scaling with how aggressive we get on retraining. That's cheaper than 1% of one user-month of Gemini once we hit ~5K users. Hence: the bet pays off.
