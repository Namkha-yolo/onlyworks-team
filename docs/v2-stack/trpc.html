tRPC over IPC

End-to-end typed APIs and IPC channels. Replaces the 169-channel hand-mirrored bridge in v1 with one typed client and one router.

Why we use this

v1 has a 444-line preload.js that hand-mirrors every IPC channel. Every new method requires touching preload AND the matching ipcMain handler AND the call site. The renderer can't autocomplete the API. If you typo a channel name, the call silently returns undefined — or worse, hangs.

tRPC solves this by making the IPC router a TypeScript type. You define procedures in main, the router's type is exported, the renderer imports it as a type-only import, and gets full autocomplete:

// main exports the type
type AppRouter = typeof appRouter;

// renderer imports it
const result = await trpc.session.start.mutate({ name: 'morning focus' });
// result is typed. Typos at the call site fail to compile.

tRPC was built for HTTP, but the transport is pluggable. electron-trpc swaps in IPC. Same API, same types, same client library.

On the backend (Express) we also use tRPC for the same reason — the website and the desktop can both share types with the server.

What you need to know

  • Procedures — queries (read) vs mutations (write)
  • Routers — group procedures, nest routers (session, reports, teams)
  • Input validation with Zod — .input(z.object({ ... }))
  • Context — the per-call object (auth, db, services) injected into every procedure
  • Middleware — requireAuth, requireAdmin, logging
  • Output inference — the procedure's return type flows to the caller automatically
  • Error handling — TRPCError, error codes, formatting for the client
  • Client — createTRPCProxyClient in renderer, the autocomplete experience
  • Integration with TanStack Query — @trpc/react-query gives you useQuery / useMutation hooks per procedure
  • electron-trpc transport — how it replaces the HTTP link

Patterns you'll see in v2

Define the router (main process)

// src/main/trpc/router.ts
import { initTRPC } from '@trpc/server';
import { z } from 'zod';

const t = initTRPC.create();

export const appRouter = t.router({
  session: t.router({
    current: t.procedure.query(() => sessionService.getCurrent()),

    start: t.procedure
      .input(z.object({
        name: z.string().min(1),
        goal: z.string().optional(),
      }))
      .mutation(({ input }) => sessionService.create(input)),

    end: t.procedure.mutation(() => sessionService.end()),
  }),

  reports: t.router({
    list: t.procedure.query(() => reportService.list()),

    get: t.procedure
      .input(z.object({ id: z.string().uuid() }))
      .query(({ input }) => reportService.get(input.id)),
  }),
});

export type AppRouter = typeof appRouter;

Wire the IPC transport

// src/main/main.ts
import { createIPCHandler } from 'electron-trpc/main';
import { appRouter } from './trpc/router';

createIPCHandler({ router: appRouter, windows: [win] });

// src/preload/preload.ts
import { exposeElectronTRPC } from 'electron-trpc/preload';
exposeElectronTRPC();

Use in the renderer

// src/renderer/src/trpc.ts
import { createTRPCProxyClient } from '@trpc/client';
import { ipcLink } from 'electron-trpc/renderer';
import type { AppRouter } from '../../main/trpc/router';

export const trpc = createTRPCProxyClient<AppRouter>({
  links: [ipcLink()],
});

// in a component
const reports = await trpc.reports.list.query();
//    ^ typed as Report[]
const created = await trpc.session.start.mutate({ name: 'morning' });
//    ^ typed as Session
//    autocomplete on { name, goal }

With TanStack Query (the v2 pattern)

// hooks/useReports.ts
export function useReports() {
  return useQuery({
    queryKey: ['reports'],
    queryFn: () => trpc.reports.list.query(),
  });
}

// or use @trpc/react-query which auto-generates hooks:
const { data } = trpcReact.reports.list.useQuery();

Hands-on: build this

Replace your electron-vite lab's IPC with tRPC:

  1. Install @trpc/server, @trpc/client, electron-trpc, zod.
  2. Create src/main/trpc/router.ts. Add three procedures: system.info (query), echo.shout (mutation taking a string, returning the uppercased version), session.tick (subscription emitting a count every second — subscriptions are advanced; you can skip if pressed for time).
  3. Wire createIPCHandler in main and exposeElectronTRPC in preload.
  4. In the renderer, call all three. Confirm autocomplete works end-to-end. Deliberately typo a procedure name and watch it fail to compile.
  5. Add input validation with Zod. Try to pass a wrong type; confirm both compile-time and runtime errors.
  6. Integrate with TanStack Query. Build a button that calls echo.shout via useMutation.

How this fits OnlyWorks v2

tRPC is the centerpiece of the v2 IPC rewrite. The 169-channel preload disappears. src/ipc/ in v1 (109 actual ipcMain.handle callsites across 20 files) collapses into one router file per domain, organized into a tree:

appRouter
  ├── auth          (sign in, sign out, refresh)
  ├── session       (start, end, current, history)
  ├── capture       (status, start, stop, takeScreenshot)
  ├── pipeline      (activity stream control, training data control)
  ├── reports       (list, get, generate, share)
  ├── teams         (CRUD, invite, members)
  ├── profile       (read, update, OW Profile data)
  ├── settings      (read, update)
  └── system        (platform info, permissions, paths)

The backend tRPC router (Express transport) shares types with the desktop via a shared types package, so a renderer call to trpc.reports.list and the backend's /api/reports return the exact same TypeScript type.

Roughly half of v1's IPC handlers are pure HTTP pass-throughs. Those get deleted: the renderer calls the backend directly via fetch with the stored JWT. Only the genuinely-native handlers (capture, OCR, permissions, deep-link OAuth, auto-update, tray, settings persistence, clipboard) stay in the IPC router. Expected count: ~20–30 procedures, down from 109.
