React Hook Form + Zod

Forms with schema validation, no boilerplate. Replaces the v1 pattern of document.getElementById('email-input').value + ad-hoc string checks.

Why we use these together

Forms are where untyped runtime data enters your app. v1 handles this with getElementById reads and regex checks scattered across page modules (AuthService.validateEmail() being one example of a real validation function, surrounded by many that aren't).

In v2:

  • Zod defines the shape and constraints of the form data. One schema, one source of truth.
  • React Hook Form manages the form state without re-rendering the world on every keystroke (uncontrolled inputs + refs).
  • @hookform/resolvers/zod bridges them: the Zod schema becomes the form validator.
  • shadcn/ui Form components wire it all into accessible, styled inputs.

End result: a form has a single schema, types flow everywhere automatically, validation messages come from one place, and the submit handler receives parsed-and-validated data.

What you need to know

Zod

  • Primitive schemas — z.string(), z.number(), z.boolean(), z.date()
  • Object schemas — z.object({ email: z.string().email() })
  • Refinements — .min(8), .regex(/.../), .refine(...)
  • Composition — .extend, .pick, .omit, .merge, .partial
  • Unions / discriminated unions — z.union, z.discriminatedUnion
  • Parsing — .parse() (throws), .safeParse() (returns Result), .parseAsync()
  • Type inference — type FormData = z.infer<typeof schema>
  • Transforming — .transform() for output shaping
  • Custom error messages — z.string({ required_error: '...' }).min(1, '...')

React Hook Form

  • useForm — the hook, with defaultValues, resolver, mode
  • register vs Controller — uncontrolled vs controlled inputs
  • handleSubmit — wraps your submit; runs validation first
  • formState — errors, isSubmitting, isDirty, isValid
  • watch / setValue / reset
  • Validation modes — onSubmit, onBlur, onChange, all
  • Field arrays — useFieldArray for dynamic lists

Patterns you'll see in v2

One schema, many uses

// schemas/profile.ts
import { z } from 'zod';

export const profileSchema = z.object({
  username: z.string()
    .min(3, 'Username must be at least 3 characters')
    .regex(/^[a-z0-9_]+$/, 'Letters, numbers, underscore only'),
  fieldOfWork: z.enum(['engineering', 'design', 'data', 'product', 'other']),
  experienceLevel: z.enum(['entry', 'junior', 'mid', 'senior', 'lead', 'executive']),
  bio: z.string().max(500).optional(),
});

export type ProfileInput = z.infer<typeof profileSchema>;

That schema validates the form, validates the API request body, generates the TypeScript type, and ends up in the database column constraints.

The form (with shadcn/ui)

const form = useForm<ProfileInput>({
  resolver: zodResolver(profileSchema),
  defaultValues: { username: '', fieldOfWork: 'engineering', /* ... */ },
});

const onSubmit = form.handleSubmit(async (values) => {
  await trpc.profile.update.mutate(values);
});

<Form {...form}>
  <form onSubmit={onSubmit} className="space-y-4">
    <FormField
      control={form.control}
      name="username"
      render={({ field }) => (
        <FormItem>
          <FormLabel>Username</FormLabel>
          <FormControl><Input {...field} /></FormControl>
          <FormMessage />
        </FormItem>
      )}
    />
    {/* ... more fields ... */}
    <Button type="submit" disabled={form.formState.isSubmitting}>
      Save
    </Button>
  </form>
</Form>

Parsing API responses

// at the boundary, parse — don't cast
const response = await fetch('/api/profile');
const json = await response.json();
const profile = profileSchema.parse(json); // throws on invalid shape
// profile is typed as ProfileInput now

Hands-on: build this

Build a settings page with three forms:

  1. Profile form: username (3+ chars, lowercase letters/numbers/_ only), display name, bio (max 500).
  2. Email change form: current email (read-only), new email (must be valid), confirm new email (must match new email — use .refine on the object schema).
  3. Notification preferences: a dynamic list of channels using useFieldArray — add/remove channels at runtime.
  4. Each form should show inline errors below the field, disable submit while submitting, and show a toast on success.
  5. Define every schema in src/schemas/ and import the inferred type wherever the form data shape is needed.

How this fits OnlyWorks v2

Every form in the v2 desktop — sign-in, sign-up, password reset, profile setup, settings, session-start dialog, report-generation modal, custom-report-name — uses this stack. The same Zod schemas live in a shared package and validate on both sides of the wire: in the desktop form and in the Express request body parser.

For input that comes via IPC instead of a form (e.g. a tRPC procedure on the main process), the same Zod schema validates the input. One source of truth.
