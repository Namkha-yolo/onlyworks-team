14 · Native plumbing

Native modules

uIOhook, active-win, sharp, tesseract.js. The Node packages that wrap C/C++/Rust binaries. Where most Electron release-day surprises live.

Why this needs its own doc

OnlyWorks depends on a handful of npm packages that wrap native binaries:

  • uiohook-napi — system-wide mouse + keyboard hooks (libuiohook → Cocoa / Win32 / X11)
  • active-win — identify the focused window's app + title (Cocoa / Win32 / X11)
  • sharp — libvips image processing (resize / re-encode screenshots before sending)
  • tesseract.js-core — WASM OCR fallback when platform OCR isn't available

These have non-obvious failure modes: native modules need to be rebuilt for Electron's Node ABI, they break when packaged inside app.asar, they need asarUnpack entries, and they can abort the whole process at the C level when permissions are missing (uIOhook on macOS without Accessibility is the canonical example).

You don't need to write a native module. You do need to know how to consume them, ship them, and debug them when they fail in a packaged build.

What you need to know

  • Native vs pure-JS modules — node-pre-gyp, napi-rs, node-gyp; what each does
  • electron-rebuild — rebuilds C++ bindings against Electron's Node ABI. Triggered by electron-builder install-app-deps postinstall.
  • asarUnpack — native files must live outside the app.asar archive at runtime; asarUnpack globs decide what gets extracted to app.asar.unpacked/
  • Lazy require in services that touch native modules — so a missing dep doesn't crash startup
  • Try/catch around start() calls — some native libs abort at C level if you call them without permissions. You can't catch a libuv assert. Gate the call.
  • Pre-flight permission checks — on macOS, systemPreferences.isTrustedAccessibilityClient(false) before uIOhook.start(); on Windows, check VC++ Runtime presence
  • User-friendly fallbacks — a missing native module shouldn't break the whole app. Show a notification, degrade gracefully.
  • How to debug a packaged build — you can open -a "OnlyWorks Desktop" --args --inspect to attach a debugger to a signed app

Patterns you'll see in v2

Lazy-require with graceful degradation

// src/main/services/capture-service.ts
let uIOhook: typeof import('uiohook-napi').uIOhook | null = null;
try {
  ({ uIOhook } = require('uiohook-napi'));
} catch (err) {
  logger.error('Failed to load uiohook-napi:', err.message);
  // Continue without input listeners. Notify user.
}

Permission-gated startup (macOS)

// Before calling uIOhook.start() on macOS, check Accessibility.
// If the user hasn't granted it yet, libuiohook will abort at the
// C level (libuv assertion). Try/catch can't catch it.
if (process.platform === 'darwin') {
  const trusted = systemPreferences.isTrustedAccessibilityClient(false);
  if (!trusted) {
    logger.warn('Accessibility not granted — input listeners deferred');
    emit('inputListenersDeferred');
    return;
  }
}

try {
  uIOhook.start();
} catch (err) {
  logger.error('uIOhook.start failed:', err);
}

asarUnpack in package.json

"asarUnpack": [
  "node_modules/uiohook-napi/**",
  "node_modules/sharp/**",
  "node_modules/@img/**",       // sharp's prebuilt libvips
  "node_modules/active-win/**",
  "node_modules/tesseract.js-core/**",
  "src/utils/*.ps1"             // Windows OCR + active-win PowerShell scripts
]

Active-win fallback chain

// Try the native package first, fall back to platform-specific helpers.
async function getActiveWindowInfo() {
  if (process.platform === 'win32') {
    try {
      const { activeWindow } = require('active-win');
      const win = await activeWindow();
      if (win) return { app: win.owner?.name, title: win.title };
    } catch (err) {
      logger.warn('active-win failed, falling back to PowerShell');
    }
    // Fallback: spawn PowerShell with Win32 API script
    return await getActiveWindowViaPowerShell();
  }
  // macOS: AppleScript via System Events
  // Linux: TBD
}

Hands-on: build this

Add a native module to your Electron lab app:

  1. Install sharp as a dependency.
  2. Add an IPC procedure image.resize that takes a Buffer + max width and returns a resized JPEG buffer.
  3. Test it from the renderer with a file picker + image preview.
  4. Run pnpm dist. Open the packaged app. Confirm it still works (this is where unconfigured asarUnpack bites you).
  5. Now remove sharp from asarUnpack, rebuild, and watch the failure mode (the resize call will throw "Could not find module").
  6. Bonus: add active-win and surface "current focused window" in the UI.

How this fits OnlyWorks v2

v2 keeps the same native module list as v1 (uIOhook, active-win, sharp, tesseract.js-core). The wrappers move into Consumer files on the unified CaptureEngine, but the underlying packages don't change — they have years of platform bug fixes embedded.

The Windows pre-flight diagnostics in v1 (src/services/windows-preflight.js) check for missing VC++ Runtime, PowerShell execution policy issues, and native module load failures at startup. That code ports straight to v2 with minimal changes — read it before you debug your first packaged-build issue.

War wound

The macOS TCC trust-state snapshot in v1's main.js exists because if the user grants Accessibility while the app is running, the running process doesn't pick it up — and the next uIOhook.start() aborts at the C level. v1 snapshots trust state at launch, and onboarding-complete triggers a app.relaunch() if state changed. v2 must preserve this exact behavior.
