Vision-Language Models

Models that take an image + text and produce text. The architecture family we're starting from. Long-term, our proprietary "work activity interpreter" is a fine-tuned VLM.

Why this is the right family

The desktop captures screenshots + OCR text + app/window metadata. The output we want is structured narrative ("user spent 35 minutes editing a design in Figma, then 15 minutes in Discord discussing the change with their team"). Gemini Vision does this today — it takes the image + a system prompt and outputs the narrative.

A Vision-Language Model is the same shape: image input → text output, conditioned on a text prompt. CLIP started the family (text-image embeddings). LLaVA, Qwen-VL, InternVL, MiniCPM-V are modern open-weight versions that can accept multiple images and follow instructions like a chat model.

For OnlyWorks: pick a strong open VLM, fine-tune it on our task, deploy it, and slowly graduate users from Gemini to our model as quality matches. The economics — "$0 per inference call instead of $0.001" — pay back the training cost within a few thousand users.

What you need to know

The architecture in one paragraph

A modern VLM is: a vision encoder (often a CLIP or SigLIP image transformer) that turns an image into a sequence of visual tokens, + a connector (a small MLP or learned mapping) that projects those tokens into the language model's embedding space, + a language model (a regular causal LLM like Llama, Qwen, Mistral) that consumes the visual tokens as if they were text tokens and generates output. Training mixes pure-text data with image-instruction-response triples.

Models worth knowing

  • CLIP (OpenAI, 2021) — the foundational text-image embedding model. Not generative, but every downstream VLM uses something like its image encoder.
  • BLIP / BLIP-2 (Salesforce) — early instruction-following VLMs.
  • LLaVA — the open-source breakthrough; first VLM trained from a small dataset of GPT-4-generated instructions. Iterations: LLaVA-1.5, LLaVA-1.6, LLaVA-OneVision.
  • Qwen-VL / Qwen2-VL / Qwen2.5-VL (Alibaba) — current state-of-the-art open weights. Multiple sizes (2B, 7B, 72B). Strong on UI / screenshot understanding.
  • InternVL — another strong family, mostly competitive with Qwen-VL.
  • MiniCPM-V / Phi-3.5-vision — smaller (~3B) VLMs that fit on a 12GB GPU. Useful for prototyping.
  • Idefics — Hugging Face's open VLM line.
  • Closed-weight reference — Gemini Flash/Pro, Claude Sonnet (vision), GPT-4o. We compare our model's output to these as the moving "good enough" bar.

Patterns you'll see in our pipeline

Running inference with Qwen2.5-VL

from transformers import AutoProcessor, AutoModelForImageTextToText
from PIL import Image

model_id = 'Qwen/Qwen2.5-VL-7B-Instruct'
processor = AutoProcessor.from_pretrained(model_id)
model = AutoModelForImageTextToText.from_pretrained(
    model_id, torch_dtype='auto', device_map='auto',
)

image = Image.open('screenshot.png')

messages = [
    {
        'role': 'user',
        'content': [
            {'type': 'image', 'image': image},
            {'type': 'text', 'text': 'Describe what the user is working on. Be specific.'},
        ],
    },
]

inputs = processor.apply_chat_template(messages, add_generation_prompt=True, tokenize=True, return_tensors='pt').to(model.device)
outputs = model.generate(**inputs, max_new_tokens=300)
print(processor.decode(outputs[0][inputs['input_ids'].shape[1]:], skip_special_tokens=True))

Hands-on: build this

Build a VLM playground for OnlyWorks-style screenshots:

  1. Rent a single small GPU (e.g. RunPod A10 24GB, ~$0.30/hr) — ask first. Or use a Mac M-series with MLX-Qwen-VL for free local inference.
  2. Load a small VLM: Qwen/Qwen2.5-VL-3B-Instruct or llava-hf/llava-1.6-mistral-7b-hf.
  3. Take 10 screenshots of yourself working in different apps (VS Code, Figma, browser, Discord).
  4. For each: ask the model "Describe what the user is working on. List specific apps and any visible text."
  5. Compare the output to what Gemini Flash gives you on the same image.
  6. Try multi-image: pass 3 consecutive screenshots and ask for an "activity summary."
  7. Document the failure modes: where does the open model lag behind Gemini? Hallucinated text? Wrong app names? Generic summaries? That gap is what fine-tuning targets.

How this fits OnlyWorks v2

The v2 plan:

  1. Stage 0 (now) — Gemini Flash + Pro do all the interpretation. Pipeline 2 in the desktop sends OCR text + occasional images to /api/pipeline/activity/stream; backend calls Gemini; results land in activity_log.
  2. Stage 1 (3–6 months) — collect a labeled dataset. Pipeline 1 (the training-data consumer) is already capturing high-resolution screenshots into training_frames. Add user-consented labels — eventually self-supervised, initially the report text that Gemini produced is the supervision target.
  3. Stage 2 (6–12 months) — pick a base VLM (probably Qwen-VL-7B), fine-tune with LoRA on our dataset. Evaluate against Gemini Flash on a held-out test set.
  4. Stage 3 (12–18 months) — if eval passes, A/B test on a fraction of traffic. If it matches or beats Gemini, graduate the model to default. Continue to retrain quarterly.

This is the bet that's driving the whole training-data side of the v1 → v2 architecture. Read the "Capture Pipeline: Unified Single-Trigger, Multi-Consumer" section of 03-architecture-review.html for the long-form rationale.
