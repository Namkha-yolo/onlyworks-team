TypeScript

Non-negotiable for v2. Most of the bugs the current desktop has come from missing or wrong runtime checks that types would have caught at compile time.

Why we use this

Open src/app/modules/page-dashboard.js in the current desktop repo. Count the defensive ?. chains and if (result && result.data && result.data.user) blocks. Now count the times sanitizeHTML() is called next to a template literal because the author wasn't sure whether the value would be a string or an object.

That's the gap TypeScript fills. With a typed IPC bridge (tRPC) and typed React props, roughly 80% of the runtime defensiveness in the current codebase becomes a compile-time guarantee.

On a team of 3+ interns shipping in parallel, TS is also the cheapest form of documentation: function signatures double as inline contracts, and refactoring is safe because the compiler flags every caller.

What you need to know

  • Primitive types: string, number, boolean, null, undefined, literal types
  • Object & array types, optional / readonly properties, index signatures
  • Function types: parameter types, return types, optional & default params, rest params
  • Type aliases vs interfaces — interfaces for objects/extends, types for unions/utilities
  • Union (A | B) and intersection (A & B) types
  • Narrowing: typeof, instanceof, in, discriminated unions, type guards (x is Foo)
  • Generics — how to read them, when to write them. Don't write fancy generics in your first month; do read them in library code.
  • Utility types: Partial, Pick, Omit, Record, ReturnType, Parameters, Awaited
  • Modules: import type vs import, default vs named exports
  • unknown vs any — any turns off the type checker, unknown keeps it on. Default to unknown at boundaries.
  • tsconfig basics: strict, target, module, moduleResolution, jsx, paths
Rule

Set "strict": true in every tsconfig you touch. Half the value of TS comes from strictness; don't turn it off because something errored.

Patterns you'll see in v2

Discriminated unions for results

type Result<T, E = string> =
  | { ok: true; data: T }
  | { ok: false; error: E };

async function getUser(id: string): Promise<Result<User>> {
  const res = await fetch(`/api/users/${id}`);
  if (!res.ok) return { ok: false, error: `HTTP ${res.status}` };
  return { ok: true, data: await res.json() };
}

const r = await getUser('123');
if (r.ok) {
  console.log(r.data.email); // typed as User
} else {
  console.error(r.error); // typed as string
}

Branded types for IDs

type SessionId = string & { readonly __brand: 'SessionId' };
type UserId = string & { readonly __brand: 'UserId' };

function getSession(id: SessionId) { /* ... */ }

const userId = '123' as UserId;
// getSession(userId); // ERROR: not a SessionId

Zod for runtime parsing of API responses

At the boundary — backend responses, IPC payloads — types alone aren't enough. We use Zod to parse and validate at runtime so the types are actually true. See the forms doc.

Hands-on: build this

Convert your prerequisites-page CLI tool to TypeScript:

  1. Create a fresh repo, run pnpm init, install typescript and @types/node.
  2. Set up a tsconfig.json with strict: true, target: "ES2022", module: "ESNext", moduleResolution: "Bundler".
  3. Define a Repo interface for the fields you read off the GitHub API.
  4. Write a typed fetchUserRepos(username: string): Promise<Repo[]> that uses Zod to parse the API response (don't just cast it — parse it).
  5. Print the top 5 repos by stars, with a nicely-typed sort.
  6. Use tsx or node --import=tsx to run it. Add a "start" script.
  7. Open a PR adding strict type-checking to CI (tsc --noEmit).

How this fits OnlyWorks v2

Every file you write in v2 is .ts or .tsx. The tRPC router uses types to give the renderer end-to-end autocompletion of every IPC call. The React components use typed props. Backend handlers parse request bodies with Zod and return typed responses.

If a function in the v2 codebase takes any, that's a code smell — surface it in PR review. The whole point of the rewrite is that the renderer doesn't have to defensively check whether result.data.user.email exists every time.
