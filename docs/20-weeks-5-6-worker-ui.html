Weeks 5-6 — Worker Desktop UI

The pipeline works; now it needs a face. By the end of Week 6 the worker desktop app has all five screens — Status, Portfolio, Privacy, Settings, and the First-run onboarding flow — functional and built against the design system. The bar is a fresh user reaching their first capture in under four minutes, and every screen looking like it belongs to the same product.

This phase is mostly Group A. Groups B and C support it with the native plumbing and auth integration the screens need.

0 · How to use this document

Same conventions as the foundation plan and doc 19: Mission, Required reading, week-by-week tasks, Hand-offs, Definition of done, Pitfalls, Self-check. All shared operating norms still apply.

The non-negotiable for this phase: implement against visuals/00-design-system.html; do not invent tokens. The desktop surface is the GitHub-dark palette (--bg-canvas #0d1117), accent purple #7c3aed, Space Grotesk body, JetBrains Mono for labels and IDs. Sharp corners (rounded-none) everywhere except pill CTAs, avatars, and status dots.

1 · Where we are right now

  • Three routes exist (/ Status, /portfolio, /settings); Privacy and First-run still need routes.
  • The five shadcn primitives are in the repo (button, card, dialog, input, tabs). Restyle at the call site with Tailwind classes; do not edit the primitive source files — that keeps A's and C's screens consistent and avoids merge churn.
  • From W3-4: the capture subscription drives live Status; the onboarding shell and consent UI exist; auth works; the recent-captures reel component is scaffolded.
  • Design tokens may already be partly in tailwind.config.js from W2 — this phase is where they get used in anger.

2 · The phase at a glance

  • Group A — Jacqueline. Build Portfolio, Privacy, finalize Settings, finish First-run onboarding. Implement the design tokens. This is the bulk of the phase.
  • Group B — Lihan. Native support: tray, global shortcut to toggle a session, permissions-flow polish, window management — the OS-level affordances the screens trigger.
  • Group C — Yingzhu. Auth fully integrated into the app shell: sign-in, sign-out, session recovery on launch.

3 · Cross-group dependency & hand-off map

  • C → A: the signed-in state and the auth views that mount inside A's app shell and onboarding. A pairs with C so C's views reuse A's shadcn primitives.
  • B → A: the tray and global shortcut start/stop a session; A's Status and the menubar must agree on session state (read it from the same Query source).
  • Namkha → A: the Privacy screen's recent-captures reel and the sensitive-app indicator read from the engine / a thumbnail consumer.

4 · Group A — Frontend (Jacqueline)

Mission. Ship the five screens. Make them feel like one product, get a new user to first capture fast, and give users real control over their data.

Required reading. visuals/wireframes.html (all five screens), visuals/00-design-system.html (tokens, button/modal system, spacing), docs/v2-stack/shadcn-ui.html and docs/v2-stack/tailwind.html.

Week 5.

  • Implement the design tokens from the design system as CSS variables in the Tailwind theme; build a small set of OnlyWorks-specific components (StatCard, SessionPanel, ReportCard) in the same shadcn style.
  • Build the Portfolio screen: the verified-skills view backed by GET /api/me/portfolio via TanStack Query, with empty / loading / error states. Use the six report-section semantic colors from the design system.
  • Finalize the Settings screen, including the version string ("Version 4.0.0"), capture preferences, and the sign-out control.

Week 6.

  • Build the Privacy screen: the recent-captures reel (now real), the sensitive-app indicator, and the delete-my-captures control wired to DELETE /api/me/captures (the GDPR path) with a confirm Dialog.
  • Finish the First-run onboarding 6-step flow with real content and the consent step (four off-by-default toggles, bound to C's consent logic). Make the welcome → permissions → sign-in → consent → first-capture → done path clean.
  • Measure time-to-first-capture on a fresh install and cut whatever pushes it over four minutes.

Definition of done (Friday W6).

  • ☐ All five screens render and navigate; each uses real data or a real empty state.
  • ☐ Portfolio shows verified skills from the backend with proper loading/error states.
  • ☐ Privacy can delete captures (GDPR) behind a confirm dialog.
  • ☐ First-run reaches first capture in under four minutes on a clean machine.
  • ☐ Every screen matches the design system; no invented tokens; sharp corners.

Pitfalls.

  • Do not edit the shadcn primitive files; restyle at the call site.
  • Do not hardcode colors — use the token variables, or dark mode and rebrands break.
  • The five-screen scope is a hard cap. New ideas go to a backlog, not into v2.0.

Self-check. Could a stranger install the app and reach their first capture without you in the room? If not, the onboarding is not done.

5 · Group B — Desktop/IPC (Lihan)

Mission. The OS-level affordances that make the app feel native: tray, global shortcut, permissions polish, window behaviour — without breaking capture.

Week 5-6 tasks.

  • Port/build the tray manager (menubar/system-tray with session state + quick start/stop) and the global shortcut that toggles a capture session (port v1's CmdOrCtrl+Shift+S behaviour). Tray and Status must read the same session source of truth.
  • Polish the permissions flow end to end with Group A: the request screen, the grant detection, and the macOS relaunch survive a real first run.
  • Window management: correct behaviour on close/minimize (capture keeps running when the window is hidden), and a clean cold start.

Definition of done. Tray reflects live session state and can start/stop a session; the global shortcut toggles capture; closing the window does not stop capture; permissions flow works on a clean macOS install.

Pitfalls. Tray and renderer drifting out of sync on session state; a global shortcut colliding with a system shortcut; the window-hidden case silently pausing capture.

Self-check. When the window is closed but a session is "running," what is actually still running, and how does the tray know?

6 · Group C — Auth & Secure Storage (Yingzhu)

Mission. Auth that lives comfortably inside the app shell: sign-in, sign-out, and reliable session recovery — reusing A's components.

Week 5-6 tasks.

  • Integrate the email-code sign-in views into A's app shell and the onboarding sign-in step; reuse A's shadcn primitives so they look native.
  • Sign-out wipes tokens and email and returns to the signed-out state cleanly.
  • Session recovery on launch: a valid stored token resumes the session; an expired one routes to sign-in. auth.status stays a Query poll (refetchInterval ~10s).
  • Harden the consent persistence from W4: toggles survive restart, revocation takes effect, and changes are server-logged.

Definition of done. Sign-in/out work inside the shell; a returning user with a valid token lands signed-in; an expired token routes to sign-in without a crash; consent state persists across restarts.

Pitfalls. A half-wiped sign-out leaving stale tokens; treating an expired token as a hard error instead of a re-auth; consent toggles not persisting.

Self-check. On a cold launch, how does the app decide whether to show the sign-in screen or the Status screen?

7 · Namkha's track — the CaptureEngine

This phase is mostly UI, so Namkha's engine work is supporting: a thumbnail / metrics path for the Privacy screen's recent-captures reel, and hardening from W3-4 dogfood. The engine remains Namkha's; A reads from its published contract and any thumbnail consumer, not from internals.

8 · End-of-phase demo (Thursday, Week 6, 19:00)

  • Group A: a fresh install walked through onboarding to first capture, then a tour of all five screens with real data.
  • Group B: start/stop a session from the tray and the global shortcut; close the window and show capture still running.
  • Group C: sign out, restart, sign back in; show session recovery.

9 · Risks watch list

  • Scope creep on screens. Five screens, no more. A sixth idea is a v2.1 backlog item.
  • Design drift. Without strict token discipline, the screens diverge. Review against the design system in PR.
  • Time-to-first-capture. If onboarding has too many steps or a slow permission dance, the four-minute target slips — measure early.
  • Capture regressions from UI work. Window/tray changes can accidentally pause or kill capture; keep an eye on the W3-4 done-when boxes.

10 · When something is wrong

Surface blockers in standup and #swe-team immediately. Fix red CI first. Write decisions in the PR. If a screen needs a design-system token that does not exist, that is a conversation with Namkha (who owns the design system), not a new invented token.
