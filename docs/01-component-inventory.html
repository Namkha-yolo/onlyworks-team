OnlyWorks Desktop — Component Inventory (v1)

Source: code audit at 2026-05-28. This is the "what exists today" map, organized by feature area. Use this when deciding what to port, rebuild, or drop in v2.

Feature Area: Authentication

Login methods (3):

  • Google OAuth via system browser (shell.openExternal) → 3-minute poll loop hitting GET /api/auth/oauth/poll?state=... every 2s
  • LinkedIn OAuth (same pattern as Google)
  • Email + password (signup + signin, with 6-digit verification codes for new signups, 10-min expiry)
  • Custom protocol handler onlyworks:// for deep-link OAuth callback on macOS
  • Embedded OAuth via BrowserView (legacy, still wired up at src/components/embedded-oauth.js but not actively used)

Auth files:

  • src/auth/auth.html + auth.js — sign-in/sign-up page, RFC-5322-ish email validation, password rules (8+ chars, upper/lower/number), verification code entry
  • src/auth/loading.html/.js/.css — interstitial loading
  • src/ipc/auth.handlers.js — 8 IPC handlers (google, linkedin, email, signup, verify-email, resend-verification, signout, complete)
  • src/core/auth-utils.js — handleAuthError(), authenticatedFetch() with auto-token-refresh + 401 retry, setupAuth() startup routing
  • src/core/oauth-flow.js — completeEmbeddedOAuth(), syncOnboardingToBackend(), exchangeCodeForTokens()
  • src/services/backend-auth.js — encrypted token storage via electron-store (machine-specific encryption key from hostname/user/platform), proactive token refresh
  • src/core/token-refresh.js — 50-line background timer
  • src/services/supabase.js — Supabase client (regular + admin) with file-backed session persistence via safeStorage

State owned: access/refresh tokens (encrypted), current user, token expiry. Sleep/wake handling via powerMonitor.on('resume') re-validates tokens.

Feature Area: Session Capture

Two-pipeline architecture:

Pipeline 2 — Activity Analysis (core):

  • src/services/capture-service.js (52KB, ~1417 lines) — CaptureService extends EventEmitter: 3s interval screenshots + on user actions (click, Enter, copy). Manages: uiohook-napi keyboard/mouse listeners, clipboard polling (1.5s), idle/inactivity detection (10-min warning, 15-min auto-stop), SHA-256 duplicate detection, per-app time tracking, window-title change detection, capture-health failure reporting
  • src/services/ocr-service.js — Platform-native priority: macOS Vision (compiled Swift binary src/native/ocr_helper first, Swift CLT interpreter second) → Windows.Media.Ocr via PowerShell wrapper (3-failure circuit breaker → Tesseract fallback) → Tesseract.js (Linux + global fallback)
  • src/services/activity-buffer-service.js — Rolling 20-frame buffer (~60s); flushes batches every 30s to POST /api/pipeline/activity/stream
  • src/services/process-detection-service.js — Detects 30+ competing surveillance tools (Cluely, Teramind, TimeDoctor, Hubstaff, ActivTrak, RescueTime, etc.) by process name

Pipeline 1 — Training Data Collection (core):

  • src/services/training-data-service.js — Captures full-res screenshots on user actions. Auto-labels frames by app category (APP_CATEGORIES: VS Code=high, Discord=medium, YouTube=low). Uploads to POST /api/pipeline/training/frames

Session orchestration:

  • src/services/session-service.js — Promise-based concurrency lock; orphan-session reconciliation; coordinates createSession/endSession with /api/sessions/*; paired bidirectionally with capture-service
  • src/services/cleanup-service.js — Strips raw OCR text from activity_log rows older than 7 days (retention)
  • src/core/pipeline-init.js — Wires sendCallback/uploadCallback between buffer services and backend HTTP layer

Native helpers:

  • src/native/ocr_helper.swift + compiled ocr_helper binary — bundled in extraResources so users don't need Xcode CLT
  • src/utils/windows-ocr.ps1 — PowerShell wrapper around Windows.Media.Ocr
  • src/utils/get-active-window.ps1 — PowerShell+Win32 fallback when active-win fails
  • src/utils/linux-ocr.sh — Linux helper

Global shortcut: CmdOrCtrl+Shift+S toggles session start/stop.

Feature Area: Hiring

Desktop: zero first-class hiring UI today. The OW Profile page is the only desktop touchpoint (it's the public verified profile shared via /ow/{shareToken}). All hiring infrastructure is web/backend-only.

IPC (profile:*, 9 handlers): profile:get, checkUsername, submit, update, getOwId, getStats, getOWProfile, shareOWProfile, copyProfileUrl. Plus resume:* (4: pickFile/upload/get/delete) and settings:* (3).

Feature Area: Onboarding

Pages (src/onboarding/):

  • welcome.html/.js/.css — First-run page: macOS-only permission cards (Screen Recording + Accessibility, clickable to open System Settings panes via x-apple.systempreferences: URLs), permission polling, Terms + Privacy buttons, terms-acceptance checkbox, "Get Started" CTA (disabled until permissions + terms satisfied)
  • profile-setup.html/.js/.css — Required: full name, display name (regex-validated, backend username availability check), field of work (custom dropdown, 11 options), experience level (5 tiers), optional job title

Mid-session permission handling: macOS TCC snapshot at launch (_launchTrustState in OnlyWorksApp). If user grants permissions during onboarding, onboarding:complete detects transition and relaunches the app (uIOhook would otherwise abort at C-level on a poisoned process).

IPC (onboarding:* + tutorial:*, 6 handlers): checkComplete, complete, syncToBackend, tutorial:isPending, markCompleted, setPending.

Tutorial: src/app/modules/tutorial.js (26KB) — 8-step coach-mark walkthrough across Dashboard → Activity → OW Profile with spotlight cutouts, demo content injection, "Replay Tutorial" in Settings.

Infrastructure

IPC bridge (src/preload.js, 444 lines)

contextBridge.exposeInMainWorld('electronAPI', {...}) with namespaces: auth, profile, resume, session, capture, pipeline.{activity,training,ocr}, sessions, reports, teams, goals, processMonitor, platformTracking, floatingPanel, onboarding, tutorial, system, permissions, legal, settings, debug. Plus event listeners. Bridges console.log to main with JWT/Bearer/password regex sanitization.

Window/tray/menu (src/core/)

  • window-manager.js — createWindow() (1200×800 default, 650×400 min, native traffic-light positioning on macOS, Mica backdrop on Win11, DevTools dev-only)
  • tray-manager.js — System tray w/ Start/Stop Session, Show App, Sign Out, Quit; rebuilds menu state when session toggles
  • menu-manager.js — Application menu (Edit/View/Window). Reload/forceReload/toggleDevTools are dev-only; macOS gets standard app menu
  • lifecycle.js — setupWhenReady, setupWindowAllClosed (quit on non-darwin), setupBeforeQuit (graceful session end with event.preventDefault() until report generates)
  • error-handlers.js — Global uncaughtException/unhandledRejection handlers

Auto-updater

  • src/services/auto-updater.js (20KB) — electron-updater wrapper, GitHub releases (Namkha-yolo/ONLYWORKS_DIST), tenant-policy hooks, event queue buffering for startup-phase renderer
  • src/core/auto-updater-flow.js — Startup orchestration: 5s update-check timeout → 90s download timeout → "Restart Now/Later" prompt
  • src/startup/startup.html/.js/.css — Update-check splash

UI primitives (src/app/modules/components.js)

App entry (src/main.js)

OnlyWorksApp lifecycle:

  1. Startup — Windows-specific Chromium flags before app.whenReady(); silences console.log in packaged builds; captures macOS TCC trust snapshot
  2. Init — createApplicationMenu → createWindow (loads startup/startup.html) → builds shared context for IPC handlers → registerAllHandlers → createSystemTray → instantiates CaptureService (defers initialize until after onboarding) → initializeAutoUpdaterWithStartup → setImmediate(initializePipelineServices) → windows-preflight.runAll() on win32 → registerGlobalShortcuts(CmdOrCtrl+Shift+S) → startTokenRefreshTimer → powerMonitor.on('resume') for post-sleep token refresh
  3. Custom protocol — app.setAsDefaultProtocolClient('onlyworks'); app.on('open-url') handles OAuth deep-link (exchanges short code for tokens via /api/auth/exchange-code)
  4. Activate — macOS dock click recreates window, skips startup, jumps to targetPage
  5. Shutdown — setupBeforeQuit ends active session gracefully (event.preventDefault() until report generates), cleans up captureService, autoUpdater, token timer, unregisters global shortcuts

Routing (src/app/modules/router.js)

showSection(name) switch with cases: dashboard, activity (and legacy alias 'sessions'), settings, manager, ow-profile, goals (placeholder "under development"), profile (inline HTML). Cleans up event listeners and timers via window.sectionCleanup before each switch; emits section-changed events through window.DataManager.

State management (in-flight migration)

  • src/app/modules/store.js — AppStore observable state (intended replacement for window.* globals)
  • src/app/modules/data-manager.js — TTL cache for reports (2min), sessions (1min), owProfile (5min); pub/sub; withTimeout() (30s default)

Diagnostics (Windows)

src/services/windows-preflight.js — 5 startup checks: VC++ Redistributable (vcruntime140.dll), uiohook-napi loadability, sharp loadability, PowerShell execution policy, Windows Defender exclusion suggestion. Results persisted to user-settings.

User settings (src/services/user-settings.js)

electron-store namespaces: onboarding, profile, widget, tutorial. Tracks completion flags, pending-sync state, terms/privacy versions accepted.

Config (src/config/)

  • config-manager.js — Loads .env in dev, hardcoded production URLs in packaged builds
  • production.js — Hardcoded constants. Backend URL hardcoded in 3 places (backend-auth.js, backend-api.js, todo-service.js): https://onlyworks-backend-server.onrender.com

Notable Complexity / Pain Points

  1. Two parallel session pages. page-sessions.js exists with its own UI but router maps 'sessions' → showActivity(). Likely dead but unremoved.

  2. Two parallel session API surfaces. electronAPI.session.* (singular, 8 channels) and electronAPI.sessions.* (plural, 5 channels) overlap. Legacy top-level routes like get-sessions, delete-session, setSessionGoal, startNewSessionWithGoal exposed in preload but barely used.

  3. page-ow-profile.js is 47KB / ~900 lines. Resume parsing + OW Profile rendering + share + upload + delete + legacy data-shape handling all in one file.

  4. page-activity.js is 41KB. Timeline + custom-report overlay + day-report generation + session rename + scroll gradients — should be 4-5 files.

  5. capture-service.js is 52KB / ~1417 lines. Single class handling screenshots, uiohook, clipboard, idle, dedup, per-app time, window-title, capture-health. Two-pipeline architecture is the wrong abstraction (organized by output type, not trigger).

  6. Dead/skeleton code (40KB+):

    • src/services/integration-service.js (19KB) — Monday/Jira/Asana OAuth, never imported
    • src/services/todo-service.js (20KB) — never imported
    • src/services/Untitled-1 — empty file
    • src/app/app.js.bak — leftover backup

  7. Legacy preload channels with no handler (will hang silently if called): take-screenshot, get-screenshot-count, set-team-context, set-analysis-mode, generate-intelligent-report, get-stored-report, get-reports-list, generate-range-report, export-screenshots, get-platform-usage, get-ai-tool-usage, floating-panel-*, widget:test-trigger-alert, setWidgetTheme.

  8. macOS TCC permission relaunch. Onboarding-complete handler detects mid-session permission grants and calls app.relaunch() to avoid uIOhook C-level abort. Hard-won fix — must survive v2.

  9. Subscription system is stubbed. src/app/modules/subscription.js always returns { isPaid: false, isOnTrial: false, statusText: 'Free' }. No paid-tier logic exists in desktop.

  10. Goals UI partially landed. Full UI in page-goals-teams.js, 7 IPC handlers, but router renders placeholder "under development".

  11. Tutorial.js (26KB) manages DOM-level coach marks across navigation — spotlight cutouts, demo content injection, element-wait polling.

  12. No proper testing. vitest configured; few tests beyond auth.js validators.

  13. Legacy field-name normalization. normalizeReport() in reports.handlers.js shuffles fields between top-level and report_data.* for backend evolution. enhanceReportWithOnlyWorks() fills fallback strings.

Surprises and Loose Ends

  • Floating panel API in preload (floatingPanel.{toggle,getCaptureState,toggleRecording,openDashboard,openSettings,resize}) — no handlers. Abandoned.
  • Widget theme — setWidgetTheme, widget namespace in user-settings defaults — references a widget that doesn't exist.
  • Analytics namespace — getDailyAnalytics, getWeeklyAnalytics, getProductivityInsights, getSessionAnalytics, getIntelligentAnalysis exposed in preload; only get-analytics has a handler.
  • Platform tracking namespace — getPlatformUsage, getPlatformBreakdown, getAIToolUsage, getPlatformTimeline exposed; no handlers.
  • Export feature stubbed — exportScreenshots, getExportHistory, openExportFile exposed; "No local downloads" per project rules.
  • Two intelligent-analysis APIs dead — set-analysis-mode, get-intelligent-analysis, generate-intelligent-report, set-team-context — preload only.
  • Two empty-state systems — owEmptyState() (canonical) AND window.createEmptyState used in page-activity:209 and page-sessions:243. Different DOM. Should unify.
  • Manager Reports duplication — page-level Reports section AND per-member report-generation modal have duplicated logic.
  • Sleep/wake capture loop — powerMonitor.on('resume') refreshes tokens but doesn't pause/resume capture; 3s timer keeps running through sleep, could spam screenshots on wake.
  • Backend URL hardcoded in 3 files. Should be centralized.
  • No state management library — window.* globals everywhere; AppStore mid-migration.
  • 37 scripts loaded in order from app.html (actual: ~30). Fragile dependency — exports.js must load last, _namespace.js first.
  • CSS file: single ~5,700 lines at src/app/styles/styles.css.
  • Sidebar nav: 4 items + Settings + Profile (Dashboard/Activity/Team/OW Profile + Settings + Profile in bottom). Reports is NOT a top-level nav item — only accessible via Activity → "Generate Custom Report".
