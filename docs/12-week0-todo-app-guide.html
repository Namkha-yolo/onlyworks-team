Week 0 — Todo App Build Guide

This is your Week 0 template-app exercise. By the end of it you will have built an Electron + React + TypeScript + tRPC todo app from scratch, in roughly 4-6 working days. Then you will spend the rest of Week 0 reading planning docs and v1 source.

The exercise exists for one reason: to make sure you can ship in our stack before you touch real OnlyWorks code. It is intentionally tiny. Don't try to make it impressive. Make it work, then make sure you understand every choice.

You don't need to read the whole guide before starting. Read sections 1–3 (Prereqs + Stack), then work top-to-bottom through Section 4. The "Common pitfalls" section is for when something breaks.

1 · Prerequisites

Install these before you do anything else.

Recommended VS Code extensions:

  • Prettier (esbenp.prettier-vscode)
  • Tailwind CSS IntelliSense (bradlc.vscode-tailwindcss)

If any of the above doesn't work on your machine, stop and ping Namkha. Don't spend 4 hours debugging Node install paths.

2 · The stack (and why each piece)

You'll wire all of these into one app. Skim the official "Getting Started" page for each before you start typing.

Layer Library Official docs Why we use it
Desktop runtime Electron 33 electronjs.org/docs Cross-platform desktop with Chromium renderer + Node main process
Dev tooling electron-vite electron-vite.org Fast HMR for both main + renderer. No webpack pain.
UI framework React 18 react.dev Industry standard, what v2 uses
Types TypeScript 5.5 (strict) typescriptlang.org/docs No any. Strict mode catches half your bugs at compile time.
Build tool Vite vitejs.dev/guide Drives the renderer side of electron-vite
Styling Tailwind v3 tailwindcss.com/docs Utility-first; matches v2 scaffold
Components shadcn/ui ui.shadcn.com Copy-paste components, not a library — you own the source
Typed IPC tRPC v11 trpc.io/docs End-to-end type safety between main + renderer
Server state TanStack Query v5 tanstack.com/query Caching, refetching, mutations. Wraps tRPC.
UI state Zustand zustand.docs.pmnd.rs Tiny global stores. For state that isn't from the server.
Animation framer-motion motion.dev List-insert animations
Testing Vitest vitest.dev Vite-native test runner

The four-word sentence that explains everything: type safety from DB to pixel. If you change a procedure's return type in main, the renderer's TypeScript compile fails. You catch the bug before you run it.

A note on Tailwind versions: Tailwind v4 exists and works. We use v3.4.10 because shadcn/ui's setup story is most stable on v3 and that's what the v2 scaffold pins. Don't fight the version.

3 · What you're building

A todo app with:

  • Add / list / toggle-complete / delete todos
  • Filter by All / Active / Completed
  • A list-insert animation when you add an item
  • Persistence to disk in the main process (a JSON file is fine — no DB)
  • One unit test

That's it. No login. No accounts. No backend server. The "server" is your Electron main process.

A reasonable visual reference: todomvc.com. Don't copy the styling — use Tailwind + shadcn defaults.

4 · Step-by-step build

Step 1 · Create the repo + scaffold

mkdir my-todo && cd my-todo
git init
gh repo create my-todo --private --source=. --remote=origin

Scaffold with electron-vite:

pnpm create @quick-start/electron@latest my-todo-app -- --template react-ts
cd my-todo-app
pnpm install
pnpm dev

You should see a window pop up that says "Electron + Vite + React". Close it.

Read the scaffold:

  • electron/main/index.ts — the main process entry point
  • electron/preload/index.ts — the bridge between main and renderer
  • src/App.tsx — the React renderer
  • electron.vite.config.ts — build config

Commit the scaffold before you change anything:

git add . && git commit -m "chore: initial electron-vite scaffold"

Step 2 · Pin versions

Open package.json. Remove every ^ and ~ from version numbers. Pin everything. Example:

{
  "dependencies": {
    "react": "18.3.1",
    "react-dom": "18.3.1"
  }
}

Run pnpm install again. Commit:

git add package.json pnpm-lock.yaml && git commit -m "chore: pin all dependencies to exact versions"

Why pinning: see Section 7. Be ready to explain this in your review.

Step 3 · Add Tailwind v3 + shadcn/ui

Install Tailwind:

pnpm add -D tailwindcss@3.4.10 postcss@8.4.41 autoprefixer@10.4.20
pnpm dlx tailwindcss init -p

Edit tailwind.config.js:

/** @type {import('tailwindcss').Config} */
export default {
  content: ['./index.html', './src/**/*.{ts,tsx}'],
  theme: { extend: {} },
  plugins: [],
}

Replace the contents of src/assets/index.css (or wherever the global CSS lives) with:

@tailwind base;
@tailwind components;
@tailwind utilities;

Make sure it's imported by your renderer entry (src/main.tsx):

import './assets/index.css'

Run pnpm dev. If the existing styles look different (no rounded buttons, etc.), Tailwind is in. Good.

Now install shadcn/ui. Follow the official Vite guide step-by-step: ui.shadcn.com/docs/installation/vite. The key commands:

pnpm dlx shadcn@latest init
pnpm dlx shadcn@latest add button input checkbox

You'll be prompted about TS config paths and style choice. Pick TypeScript / Default style / Neutral / yes CSS variables.

This will add files into src/components/ui/. Read one (button.tsx) — notice it's your source code, not an import from a package. That's the point.

Commit:

git add . && git commit -m "feat: tailwind v3 + shadcn/ui setup"

Common breakage here: shadcn's CLI may write a tailwind.config.ts next to your tailwind.config.js. Delete the .js one and keep the .ts. Also: it adds CSS variables to your global CSS — that's correct, don't remove them.

Step 4 · Add tRPC v11 to the main process

pnpm add @trpc/server@11.0.0-rc.477 zod@3.23.8

Create electron/main/trpc/router.ts:

import { initTRPC } from '@trpc/server'
import { z } from 'zod'
import { todoStore } from '../store'

const t = initTRPC.create({ isServer: true })

export const router = t.router({
  todos: t.router({
    list: t.procedure.query(() => todoStore.list()),

    add: t.procedure
      .input(z.object({ text: z.string().min(1).max(280) }))
      .mutation(({ input }) => todoStore.add(input.text)),

    toggle: t.procedure
      .input(z.object({ id: z.string() }))
      .mutation(({ input }) => todoStore.toggle(input.id)),

    delete: t.procedure
      .input(z.object({ id: z.string() }))
      .mutation(({ input }) => todoStore.delete(input.id)),
  }),
})

export type AppRouter = typeof router

Create electron/main/store.ts — the in-memory + JSON-persisted store:

import { app } from 'electron'
import { existsSync, readFileSync, writeFileSync } from 'fs'
import { join } from 'path'
import { randomUUID } from 'crypto'

export type Todo = { id: string; text: string; done: boolean; createdAt: number }

const file = join(app.getPath('userData'), 'todos.json')
let todos: Todo[] = existsSync(file) ? JSON.parse(readFileSync(file, 'utf8')) : []

const persist = () => writeFileSync(file, JSON.stringify(todos, null, 2))

export const todoStore = {
  list: () => todos,
  add: (text: string) => {
    const t: Todo = { id: randomUUID(), text, done: false, createdAt: Date.now() }
    todos = [t, ...todos]
    persist()
    return t
  },
  toggle: (id: string) => {
    todos = todos.map(t => t.id === id ? { ...t, done: !t.done } : t)
    persist()
  },
  delete: (id: string) => {
    todos = todos.filter(t => t.id !== id)
    persist()
  },
}

Wire the router to IPC. tRPC over IPC isn't built-in to v11, so you need a tiny adapter. The community pattern is to use ipcMain.handle for each procedure call. Easiest path: install electron-trpc:

pnpm add electron-trpc@0.7.1

Then in electron/main/index.ts, before app.whenReady():

import { createIPCHandler } from 'electron-trpc/main'
import { router } from './trpc/router'

app.whenReady().then(() => {
  createIPCHandler({ router, windows: [mainWindow] })
})

Reference: github.com/jsonnull/electron-trpc.

In electron/preload/index.ts, expose the IPC bridge:

import { exposeElectronTRPC } from 'electron-trpc/main'
process.once('loaded', () => { exposeElectronTRPC() })

Commit:

git add . && git commit -m "feat: tRPC router + persistent store in main process"

Step 5 · Wire the typed tRPC client into the renderer

pnpm add @trpc/client@11.0.0-rc.477

Create src/lib/trpc.ts:

import { createTRPCReact } from '@trpc/react-query'
import { ipcLink } from 'electron-trpc/renderer'
import { httpBatchLink } from '@trpc/client'
import type { AppRouter } from '../../electron/main/trpc/router'

export const trpc = createTRPCReact<AppRouter>()

export const trpcClient = trpc.createClient({
  links: [ipcLink()],
})

Note that AppRouter is imported as a type only — there's no runtime code crossing the process boundary. This is the magic of tRPC. Hover over trpc.todos.add.useMutation in your renderer and TypeScript will know the exact input shape.

Step 6 · Set up TanStack Query

pnpm add @tanstack/react-query@5.51.23 @trpc/react-query@11.0.0-rc.477
pnpm add -D @tanstack/react-query-devtools@5.51.23

In src/main.tsx, wrap your app:

import { QueryClient, QueryClientProvider } from '@tanstack/react-query'
import { ReactQueryDevtools } from '@tanstack/react-query-devtools'
import { trpc, trpcClient } from './lib/trpc'

const queryClient = new QueryClient()

ReactDOM.createRoot(document.getElementById('root')!).render(
  <trpc.Provider client={trpcClient} queryClient={queryClient}>
    <QueryClientProvider client={queryClient}>
      <App />
      <ReactQueryDevtools />
    </QueryClientProvider>
  </trpc.Provider>
)

When you next run pnpm dev you'll see a flower-shaped icon in the corner of the window — that's the Query Devtools. Open it. You will use this every day.

Step 7 · Add Zustand for filter state

pnpm add zustand@4.5.4

Create src/stores/filter.ts:

import { create } from 'zustand'

export type Filter = 'all' | 'active' | 'completed'

type FilterStore = {
  filter: Filter
  setFilter: (f: Filter) => void
}

export const useFilterStore = create<FilterStore>(set => ({
  filter: 'all',
  setFilter: filter => set({ filter }),
}))

Why Zustand for this and TanStack Query for todos? TanStack Query is for server state — anything that lives outside your renderer (in main, on a backend, in a DB). The filter is purely a UI choice; it doesn't exist on the server. Zustand owns it.

If you only remember one rule: "is this data the source of truth, or a cached copy?" If cached → TanStack Query. If source of truth → Zustand (or useState if it's local).

Step 8 · Build the UI

Replace src/App.tsx with something like this. Read it, don't paste blindly:

import { useState } from 'react'
import { trpc } from './lib/trpc'
import { useFilterStore } from './stores/filter'
import { Button } from './components/ui/button'
import { Input } from './components/ui/input'
import { Checkbox } from './components/ui/checkbox'
import { AnimatePresence, motion } from 'framer-motion'

export default function App() {
  const [text, setText] = useState('')
  const { filter, setFilter } = useFilterStore()
  const utils = trpc.useUtils()

  const todos = trpc.todos.list.useQuery()
  const add = trpc.todos.add.useMutation({
    onSuccess: () => utils.todos.list.invalidate(),
  })
  const toggle = trpc.todos.toggle.useMutation({
    onSuccess: () => utils.todos.list.invalidate(),
  })
  const remove = trpc.todos.delete.useMutation({
    onSuccess: () => utils.todos.list.invalidate(),
  })

  const visible = (todos.data ?? []).filter(t =>
    filter === 'all' ? true : filter === 'active' ? !t.done : t.done
  )

  return (
    <div className="mx-auto max-w-xl p-8">
      <h1 className="mb-6 text-2xl font-semibold">Todos</h1>

      <form
        onSubmit={e => {
          e.preventDefault()
          if (text.trim()) {
            add.mutate({ text: text.trim() })
            setText('')
          }
        }}
        className="mb-4 flex gap-2"
      >
        <Input value={text} onChange={e => setText(e.target.value)} placeholder="Add a todo" />
        <Button type="submit" disabled={add.isPending}>Add</Button>
      </form>

      <div className="mb-4 flex gap-2">
        {(['all', 'active', 'completed'] as const).map(f => (
          <Button
            key={f}
            variant={filter === f ? 'default' : 'outline'}
            size="sm"
            onClick={() => setFilter(f)}
          >
            {f}
          </Button>
        ))}
      </div>

      <ul className="space-y-2">
        <AnimatePresence>
          {visible.map(t => (
            <motion.li
              key={t.id}
              layout
              initial={{ opacity: 0, y: -8 }}
              animate={{ opacity: 1, y: 0 }}
              exit={{ opacity: 0, x: -16 }}
              className="flex items-center gap-3 rounded border p-2"
            >
              <Checkbox checked={t.done} onCheckedChange={() => toggle.mutate({ id: t.id })} />
              <span className={t.done ? 'flex-1 text-muted-foreground line-through' : 'flex-1'}>
                {t.text}
              </span>
              <Button variant="ghost" size="sm" onClick={() => remove.mutate({ id: t.id })}>
                ✕
              </Button>
            </motion.li>
          ))}
        </AnimatePresence>
      </ul>
    </div>
  )
}

Install framer-motion:

pnpm add framer-motion@11.5.4

Run pnpm dev. Add a todo. Watch the animation. Toggle one. Reload the window — your todos persist because the JSON file in app.getPath('userData') survives.

Commit:

git add . && git commit -m "feat: working todo UI with tRPC + tanstack query + zustand + framer-motion"

6 · Common pitfalls

"Cannot find module '../../electron/main/trpc/router'" in the renderer

Your tsconfig paths aren't picking up the relative reference. Solutions in order of preference:

  1. Use a path alias like @shared/router and define a shared types directory.
  2. Move AppRouter to a shared/ folder.
  3. Use the relative import as-is — it works as long as you import type it.

Tailwind classes do nothing

  • Did you import the CSS file in main.tsx?
  • Does tailwind.config.js's content glob match where your components actually live?
  • Did postcss.config.js get generated? Without it, Tailwind silently no-ops.

shadcn components look unstyled

  • Check that the global CSS variables block is present in your global CSS.
  • Check that tailwind.config.ts has the shadcn-extended theme (the init script writes this).

tRPC calls fail with "no handler" or hang forever

  • electron-trpc requires both createIPCHandler in main and exposeElectronTRPC in preload. Both, not one.
  • The preload script must run before app.whenReady. If you've moved preload setup, this can silently break.

Build fails with native module errors on Windows

Install Visual Studio Build Tools with C++ workload. Restart your terminal. Rerun pnpm install.

Build fails with Apple notarization errors on macOS

You don't need notarization for this exercise. Run pnpm dev not pnpm build. If you want to build, comment out signing config in electron-builder.yml.

"Why is pnpm dev so slow on first run?"

It's not slow — Electron has to download (~70 MB), Chromium has to bootstrap, Vite has to do its first build, and the tRPC IPC handshake has to complete. Subsequent runs use HMR and feel instant.

"Should I use HashRouter / BrowserRouter for routing?"

Not for this exercise — one page is fine. In v2 we use TanStack Router. Don't pre-emptively add routing here.

7 · The "why" questions you must be able to answer

These come up in your review. Prepare answers now.

Q: Why tRPC over IPC instead of plain ipcRenderer.invoke? A: tRPC gives you end-to-end TypeScript types between two processes. If you change a procedure's return type in main, the renderer's useQuery call fails to typecheck — you find the bug at compile time, not at runtime. Plain ipcRenderer.invoke is any on both ends; you'd manually mirror types and hope they don't drift.

Q: Why TanStack Query if tRPC already calls main? A: TanStack Query handles caching, deduplication, optimistic updates, refetching, retry-on-error, and SSR — all things you'd otherwise hand-roll. The @trpc/react-query adapter wires tRPC procedures into Query primitives so you get both type safety and state management.

Q: Why Zustand if React has useState / useContext? A: Context re-renders every consumer on every change. Zustand uses selectors so only components that read a specific slice re-render. For app-wide UI state (theme, modals, filters), it's cheaper than Context and simpler than Redux.

Q: Why pin every dependency? A: We were burned in v1 by an electron-builder minor release that broke macOS signing. With ^ versions, two engineers can install the same package.json and get different node_modules, which breaks "works on my machine" debugging. Pinning costs us nothing (we can always upgrade explicitly) and removes a class of bug.

Q: Why Electron and not Tauri? A: Tauri is excellent but its native API surface is narrower than Electron's, and our team's expertise — and v1's existing code — is Electron. We'd save bundle size but lose months of platform-specific knowledge. For v2 we keep Electron and let Tauri prove itself further before reconsidering.

Q: Why electron-vite specifically? A: It gives us Vite's HMR on both processes (main + renderer) without writing custom build orchestration. The alternative is Webpack via electron-forge, which is slower to iterate on and more config-heavy.

Q: Why border-radius: 0 in the OnlyWorks design system? A: Sharp corners are a brand choice — they read as serious, editorial, and unlike the rounded-everything default of most SaaS. Avatar circles and pills are the documented exception. (You're not testing this in the todo app, but you'll be asked about it.)

8 · When you're stuck

In this order:

  1. Read the error message. Sounds obvious. Most people don't.
  2. Search this guide. Cmd-F / Ctrl-F. Section 6 covers most issues.
  3. Search the official docs of the failing layer. If it's a Tailwind error, search the Tailwind docs — not Stack Overflow.
  4. Search GitHub Issues for the failing package. Filter by issues from the last 6 months.
  5. Ask in Slack. Don't sit silent for more than 30 minutes if you're truly blocked. Include:
    • What you ran
    • What you expected
    • What you got (error verbatim — paste the whole thing, not your interpretation)
    • What you already tried
  6. Ask Namkha. Only after 1–5.

Anti-pattern: going off-script and using a different stack (Vue, plain Electron forge, Next.js) because tRPC is annoying. The annoyance is the lesson. Don't route around it.
