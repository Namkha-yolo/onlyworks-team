OnlyWorks — Intent and History

Source: git log, recovered CLAUDE/ docs (deleted in commit 6134ed0, recoverable from 6134ed0^), CHANGELOG.md, README.md, and the v2 roadmap. Captures the story behind the product so the v2 rebuild doesn't discard hard-won lessons.

Origin Story

OnlyWorks began September 14, 2025 with a single commit ("Initial commit: Only Works Desktop App") describing itself as a "Desktop productivity tracker for Only Works". Within nine days the founders had screenshots → Gemini AI → reports working end-to-end ("THIS IS THE WORKING VERSION WITH GEMINI", Sep 23).

The earliest pitch in intern_tasks/INTERN_ONBOARDING.md frames it bluntly:

"AI-powered work verification platform. Think of it as 'proof of work' for knowledge workers." "You run the desktop app while you work. It captures what you're doing (screenshots, active apps, OCR text). When you're done, it generates an AI report summarizing your accomplishments. Over time, these reports build into a verified portfolio — your 'OW Profile' — that you can share with employers or clients."

Three audiences named from day one: individuals ("prove what you worked on"), managers ("see what your team accomplished without micromanaging"), employers ("hire candidates with verified work history — coming soon"). The order matters: individual self-portfolio came first, hiring was the future bet.

Evolution Timeline

Sep–Oct 2025 — MVP & "is this a Gemini wrapper?" Working screenshot capture, Google OAuth, Gemini + Qwen experiments (local model abandoned — Gemini won). Reports + sharable links + dashboard. Teams schema appears Oct 13 — collaboration was always in scope.

Nov–Dec 2025 — "Mentor" era and the help widget detour A "supportive mentor" prompt drove reports. A help widget with gamification ("satisfying micro-interactions", Dec 15) and an "AI tutor / assistant service / struggle detection" system was built by a contributor (Brodey). Cost optimization docs and "fatal flaws analysis" appear late December — the team was already worried about Gemini token spend.

Jan 2, 2026 — The forensic auditor pivot Commit 4534a66 ("Implement forensic work auditor system") replaces the "supportive mentor" prompt with a "forensic auditor" prompt, adds dual Manager/Worker views, and introduces "detection patterns for performative vs genuine work." This is the philosophical shift — from coach to verifier. Two days later: "two-tier report system" (2fa44a8).

Feb 22, 2026 — v3.0.0/3.0.1: the big cleanup "Modular architecture + 28 bug fixes for Windows/Mac release." Then v3.0.1 explicitly removes the AI tutor: "Remove AI tutor system (assistant service, struggle detection, help widget)" and introduces "Vision-first reports" with adaptive 10–60s capture and image resizing for bandwidth. Schema consolidation drops session_summaries, deprecates batch_reports. The product had bloated; v3 stripped it down to verification.

Mar 3–7, 2026 — Killing the paywall Two commits within four days: 1672140 "product updated with OW-ACCESS-CODE removed", then d8d9af1 "Remove access code paywall system entirely" — deletes a 505-line access-gate.js. The early-access gating model sunsetted.

Mar 10, 2026 — OW Profile as a resume 6de282a "Redesign OW Profile page as resume-style layout", followed by c62130e "verified skills resume, distinct from reports". The product reframes from "reports about your day" to "a resume that builds itself."

Apr 8, 2026 — The hiring pivot Migration 014 creates 5 tables: hiring_managers, job_posts, job_matches, interest_expressions, shortlists. Two-tier model launches: anonymous public search + paid job posting. The website /hiring page is "overhauled from waitlist to search + registration." This is the explicit commercial bet — OnlyWorks becomes a marketplace, with the desktop app as the data-collection engine for candidate verification.

Apr 14, 2026 — The "OW Profile Rescue" A multi-bug postmortem in CHANGELOG.md: users reported profiles showing "2 skills when they had 16 reports analyzed." Four cascading bugs:

  1. Broken cache invalidation
  2. Hardcoded ['technical','soft','domain'] category whitelist dropping 71% of skills
  3. Doubled OW-OW- ID prefix
  4. OAuth signup failure

New migration 016 creates a durable verified_skills table because "skills no longer live only in ow_profile_cache.ow_profile_data (lossy cache) or generated_reports.skills_used (per-report JSONB) — they now have a permanent home that survives cache corruption." Real user impact noted: "namkha's profile: 2 skills → 21 skills; brodeywang2004's profile: now 39 skills."

Apr 27, 2026 — Doc nuke All CLAUDE/ markdown docs and intern_tasks/ deleted (commits 6134ed0, 17d9476) — possibly to clean the repo before bringing in interns/PRs, but it destroyed the institutional memory.

Apr 27 onward — Windows polish Last five commits are all Windows-specific: native OCR via Windows.Media.Ocr, active-win binary, pre-flight diagnostic service, Mica backdrop on Win11, global shortcut to toggle sessions. The Windows port is being hardened.

Current State

The architecture diagram (CLAUDE/architecture.mmd, May 14) shows the current shape: desktop app + Next.js website + Express backend + Supabase, plus external Resend (email), Gemini (AI), Stripe (payments), Pinecone ("profile vector index, skill embeddings"), and GitHub Releases.

Background workers run a "stale-session finalizer every 10 min" and a "talent-community nurture cycle every 1 hr · day-3 / 14 / 30" — the most recent net-new system is a Talent Community with email nurture sequences and /talent, /talent/verify, /talent/welcome website routes plus talent_* tables (migration 018). The search service has been split: _searchVerified (real users) + _searchCommunity (with a "+50 boost"). This appears to be a pre-seeded candidate pool — possibly scraped or imported profiles being nurtured into becoming real users.

Recent commits (last 7 days) focus on Windows reliability (pre-flight diagnostics, OCR, throttling, signed builds) and small UX fixes (Mac traffic-light safe-area, removing unimplemented language selector stub). No new product features — the team is in stabilization mode.

Stated v2 Direction (from v2-roadmap.md)

P0 (cost/quality):

  • Invalid session handling for <5min sessions (skip AI processing — explicit cost concern)
  • Split models (gemini-2.0-flash for pipeline, gemini-2.0-pro for final reports)
  • Remove "Desktop" from UI strings
  • Fuzzy hiring search via synonyms

P1 (revenue):

  • Stripe integration for hiring-manager paid tier (hiring_managers.subscription_status and stripe_customer_id columns exist but /api/hiring/subscribe endpoint never built)
  • Job Post Creation UI (backend done, frontend missing)
  • Resume on onboarding

P2/P3 (long-term):

  • Resume Verification Pipeline (the core loop): "upload resume → extract skills → generate assignments → user completes in OW → AI verifies skills" — assignments table and skill_verifications table planned but unbuilt
  • Notification center
  • LinkedIn connector (PDF approach favored — "fragile, may violate ToS" for scraping)
  • Conceptual skill detection ("Figma → UI Design" mapping)
  • Saved searches, JD-paste-to-match

The roadmap explicitly defers fine-tuning ("only after approaches 1-3 are exhausted") and semantic embeddings.

Open Questions / Tensions

  1. Self-portfolio vs hiring marketplace. The features doc says "Core Product: AI-powered work verification platform" — singular. But the architecture has two viable product modes glued together: a personal proof-of-work tool (OW Profile, reports, sessions) and a B2B hiring marketplace (search, job posts, Express Interest, paid HM tier). Revenue lives in hiring; daily usage lives in self-portfolio.

  2. Monitoring vs self-reflection. April 6's commit Remove spyware terminology is the most telling moment: "Recording → Session Active, REC → SESSION, Stop Recording → End Session, screenshots → visual snapshots, Screen Recording → Screen Access, Employee Monitoring → Workplace Use, monitoring → data collection." The mandatory "What's your focus today?" intention modal was killed too. The team is actively running away from surveillance framing — but every underlying mechanism (continuous screenshots, OCR, keystroke counts, click coordinates, training-data pipeline) IS surveillance. There's an unresolved gap between what the product does and how it's allowed to talk about itself.

  3. Manager view vs worker view. January's "dual-report generation (Manager + Worker views)" persists in the manager page, but the recent emphasis is on the worker building their own portfolio.

  4. Pre-seeded vs real talent pool. The Talent Community + _searchCommunity · +50 boost strongly implies salting search results with scraped/imported non-user profiles. This is an unresolved go-to-market question being papered over with infrastructure.

  5. Reports — for whom? Custom reports capped at 5/month, daily reports auto-generate, share via link or email only ("No local downloads — reports are shared via link or email only; the download handlers were removed on 2026-04-02"). Removing downloads suggests reports are increasingly shareable artifacts rather than personal records.

Customer Signal

Direct evidence is thin but specific:

  • The "OW Profile Rescue" postmortem (Apr 14) is the strongest signal. "Users reported OW profiles showing empty or severely under-populated ('2 skills when they had 16 reports analyzed')." Four users were stuck in stale-cache state confirmed by SQL. Named users: namkha (founder, 2→21 skills), brodeywang2004 (contributor, 39 skills), aw896@cornell.edu (24 skills). 14 users total had populated skill portfolios.
  • 97 reports across 10 users in the backfill (scripts/backfill-verified-skills.js --apply → "97 reports scanned, 111 skills upserted, 90 rows created, 10 users captured"). After AI re-extraction: 141 verified-skill rows across 14 users.
  • Two stuck users: nishkarsh1512@gmail.com, claireqcheng@gmail.com with "onboarding_completed=true, profile_complete=false, username=null... Zero reports each, no active complaint."
  • Issue #12 (GitHub): Google signup failing with "Database error saving new user" — a real user-blocking bug.
  • Brodey appears as a contributor and the source of the gamification/help-widget detour in Dec 2025.

Total active-user base: ~10–15 people, mostly the founders, interns, and friends. No NPS, no churn metric, no support-ticket system referenced anywhere.

Lessons Worth Preserving

  1. The "forensic auditor" vs "supportive mentor" lesson. A swing from coach to verifier produced higher-signal reports. Don't let the AI hallucinate praise.

  2. Privacy filtering is load-bearing. sensitive-filter.js (two-layer: window-title and free-text) anonymizes Discord/iMessage/banking apps before data hits Gemini. The only way the product can ethically exist; don't regress.

  3. Caches lie. Build durable storage for high-value derived data. The April 14 postmortem learned this the hard way: ow_profile_cache.ow_profile_data was the only home for verified skills, and one invalidation bug nuked it. The fix — first-class verified_skills table with reader-side fallback — is the pattern.

  4. Hardcoded enums kill features silently. The ['technical', 'soft', 'domain'] whitelist dropped 71% of skills emitted by the AI. Schema/UI must accept what the AI actually produces.

  5. OAuth in embedded BrowserWindow doesn't work. Both Google and LinkedIn block it. System browser + polling is the only durable pattern (lessons from v3.7.2 → v3.7.7).

  6. Sessions need three-layer recovery (April 24, v3.11.0): renderer rehydration on every load + desktop orphan reconciliation + backend stale-session finalizer every 10 minutes. "Catches sessions abandoned by crashed / force-quit / dead-battery clients even if the user never relaunches the desktop app."

  7. Spyware language poisons the product. The terminology overhaul (April 6) was treated as a real feature shipped across 15 files.

  8. Cost is a top-tier feature. Multiple commits about Gemini token spend, image resizing (1280x720 JPEG q70), adaptive 10–60s capture intervals, sessions <5min skipping AI, and 5/month custom-report cap. The team has been forced to learn this.

  9. The IPC bridge is the schema. "~169 channels" between renderer and main. Any rebuild that doesn't model this surface area carefully will regress dozens of small UX behaviors.

  10. Build one canonical thing. The April 11 design audit found 8 parallel button systems and 3 modal systems with duplicate .modal-overlay selectors. The fix was painful (~683 lines of CSS deleted). Don't ship a rebuild that opens this door again.
