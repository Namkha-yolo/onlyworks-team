Weeks 9-10 — Dogfood & Hardening

Feature work is mostly done; now we make it trustworthy. This phase the whole team runs the app on their own machines every day, burns down the bugs that surfaces, closes the platform gaps (Windows CI and signing), hits the performance targets, and — critically — builds the telemetry that makes the launch metrics measurable at all. You cannot pass a metric you cannot see.

0 · How to use this document

Same conventions as docs 15 and 19-21. This phase is less about new surface and more about burn-down, gaps, and instrumentation, so the structure leans on per-area missions and checklists rather than greenfield tasks.

1 · Where we are right now

  • The product is feature-complete on the happy path: capture → OCR → reports, five screens, auth, web marketplace, Stripe.
  • CI is macOS-only (.github/workflows/build.yml, macos-latest). There is no Windows lane, yet the phase definition-of-done requires CI green and signed + notarized on both platforms.
  • There is no telemetry, support channel, or ticket system. The W12 metrics (weekly actives, opt-in rate, frames-with-consent, "zero bug reports") cannot be measured today. That is a build task this phase, not an afterthought.
  • Two dependency pins still carry carets (electron-log, electron-updater) — clean these up while hardening.

2 · The phase at a glance

  • Everyone: dogfood daily, file bugs, burn down P0/P1.
  • Group B: the Windows CI lane + signing, notarization parity.
  • Namkha: sensitive-filter hardening, the retention engine, and the telemetry/instrumentation.
  • Groups A & C: stabilize their surfaces and add the per-surface instrumentation (funnel events, auth-failure logging).

3 · Group A — Frontend (Jacqueline)

Mission. Stabilize the screens under real use and instrument the funnel.

  • Burn down UI bugs found in dogfood; tighten empty/error/loading states under real (messy) data.
  • Instrument the first-run funnel: emit events for each onboarding step so the launch can measure drop-off and time-to-first-capture.
  • Verify the app stays responsive and within the memory/CPU budget while capture runs (coordinate with B/Namkha on the perf measurement).

Done-when. No open P0/P1 on the screens; funnel events flow to the telemetry sink; the UI holds up over a multi-hour session.

4 · Group B — Desktop/IPC & CI (Lihan)

Mission. Close the platform gap. Windows must build, sign, and ship — and the performance budget must be met.

  • Add a Windows CI lane to .github/workflows/, build-only (no signing) first, so Windows breakage is caught on every PR.
  • Then Windows code-signing: [CONFIRM with Namkha: the Windows EV signing cert is a separate procurement — is it ordered?]. Verify the unusual v1 Windows signing config (verifyUpdateCodeSignature: true with signAndEditExecutable: false) actually signs on CI.
  • Confirm macOS notarization parity (already on main for macOS) and that auto-update artifacts resolve on both platforms.
  • Performance: measure renderer memory and CPU after an 8-hour capture session; target <300MB RAM and <5% CPU sustained. Fix the OCR/capture loop if it exceeds budget. Verify sleep/wake correctness and the OCR circuit breaker under real use.
  • Pin the two caret deps to exact versions.

Done-when. CI is green on macOS and Windows; both platforms produce signed (macOS notarized) installers; the renderer meets the 8-hour perf budget; deps are exact-pinned.

Pitfalls. Native rebuilds differing across platforms; the Windows signing config silently not signing; perf regressions from the capture loop hiding under "works on my Mac."

5 · Group C — Auth & Secure Storage (Yingzhu)

Mission. Make auth and account-deletion bulletproof under real use.

  • Burn down auth bugs from dogfood: token refresh edge cases, sleep/wake, sign-out completeness, the 429 and token-shape traps.
  • Verify GDPR account deletion end to end (the desktop delete path + DELETE /api/me/captures + server-side purge within the retention window).
  • Add auth-failure logging/telemetry (without logging tokens) so launch can see sign-in success rate.

Done-when. No open auth P0/P1; account deletion verifiably removes data; sign-in success/failure is observable in telemetry; no token ever appears in a log.

6 · Namkha — filters, retention, telemetry

  • Sensitive-content filter hardened at the engine level (it was a no-op hook in W3-4): block the sensitive-app blocklist (password managers, banking, healthcare, messaging) before any frame is captured/sent.
  • Retention engine cron (RFC-002): enforce the retention windows (OCR text + activity 90 days; reports 24 months; deleted-account residuals 30 days).
  • Telemetry / instrumentation: a sink for weekly-active workers, training opt-in rate, and frames-collected-with-consent — none of which is measurable today. Also stand up a bug-reporting / feedback channel so "zero capture bug reports in the final four weeks" can even be tracked.
  • AI cost tracking starts this phase (watch Gemini spend daily).

7 · Security & compliance gates

From doc 14, these must be resolved before launch — surface any that are not owned:

  • RFC-001 (guardrails / Presidio) and RFC-002 (retention engine) must be locked, not pending. [CONFIRM with Namkha: both RFCs accepted.]
  • Training-data T&C lawyer review — the consent addendum users actually agree to. [CONFIRM with Namkha: lawyer + T&C done.]
  • DPIA (GDPR Art. 35) before the first EU user.
  • Reconcile the consent default: training is opt-in only; fix any wireframe or screen that defaults it on.

8 · End-of-phase demo (Thursday, Week 10, 19:00)

  • Group B: green CI on both platforms; a signed Windows installer and a notarized macOS build; the 8-hour perf numbers.
  • Namkha: the sensitive-filter blocking a banking app; the telemetry dashboard showing live dogfood metrics.
  • Groups A/C: the bug burn-down chart at zero P0/P1; account deletion demonstrated end to end.

9 · Risks watch list

  • Windows discovered broken late. No Windows CI until now means native rebuilds, OCR, and signing problems all surface at once — start the Windows lane on day one of W9.
  • Unmeasurable metrics. If telemetry and a feedback channel are not built, the W12 go/no-go has no data. This is the highest-leverage hardening item.
  • Security gates unowned. Two pending RFCs, a lawyer review, and a DPIA all block a clean launch and none can be skipped.
  • Performance. An 8-hour capture session is where memory leaks and CPU creep show up; budget failures here become support tickets at launch.

10 · When something is wrong

Dogfood bugs go straight into the tracker with a P-level. Sev-1 (app unusable) is all-hands; Sev-2 gets fixed within a working day by the zone owner with a short written postmortem. Surface unowned security/compliance gates to Namkha immediately — they cannot be hand-waved at launch.
